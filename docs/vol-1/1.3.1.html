1.3.1 Description of MIX

1 [00]

Answer: 4 trits.

Reasoning: A MIX byte can encode 64 distinct values (0–63). With base-3 digits (“trits”), we need the smallest \(n\) such that \(3^n \ge 64\). \(3^3=27<64\) and \(3^4=81\ge64\), so \(n=4\) trits per byte (leaving \(81-64=17\) codes unused).

2 [02]

Answer: 5 bytes.

Why: Each MIX byte holds \(64\) values. With \(n\) adjacent bytes we can represent up to \(64^n-1\). \(64^4=16{,}777{,}216<99{,}999{,}999\) but \(64^5=1{,}073{,}741{,}824>99{,}999{,}999\). So we need 5 bytes (i.e., one MIX word’s magnitude; sign, if any, is separate).

3 [02]

  1. Address field: (0:2) (includes the sign and bytes 1–2)
  2. Index field: (3:3)
  3. Field field: (4:4)
  4. Operation code field: (5:5)

(Here 0 denotes the sign; bytes are numbered 1..5 left to right.)

4 [00]

Because the address field in a MIX instruction is a signed displacement. The effective address is computed as

\[ \text{EA} = A + I_4, \]

where \(A=-2000\). If \(I_4\) is large enough (e.g., \(I_4\ge 2000\)), the sum \(\text{EA}\) is a valid memory location \(0\le \text{EA} < 4000\). Thus LDA -2000,4 means “load A from the word at address \(I_4-2000\)”; the program isn’t using a negative memory address—just a negative offset.

Here’s the full decoding of (6) into MIX’s symbolic form.

Given numeric instruction (6) Word layout is ±AA | I | F | C. The figure shows:

  • sign = –
  • AA = 80
  • I = 3
  • F = 5
  • C = 4

  1. Opcode → mnemonic In MIX, opcode C = 4 is DIV.

  2. Field specification Field byte \(F\) encodes \((L:R)\) by \(F = 8L + R\). With \(F = 5\) we get \(L = 0,\ R = 5 \Rightarrow (0:5)\) (the whole word).

  3. Address and index The address field is the signed two-byte value ±AA = −80. Index byte I = 3 means “use index register X3”.

  4. Symbolic instruction (analogous to OP Z,I(F))

DIV  -80,3(0:5)

Since DIV’s default field is (0:5), this is commonly written simply as:

DIV  -80,3

(Effective address at run time is \(EA = X_3 + (-80)\).)

6 [10]

Given memory location 3000 holds the word

M[3000] = + | b1=5 | b2=1 | (b3:b4)=200 | b5=15

(read as sign plus; byte1=5, byte2=1, the two-byte field (3:4)=200, byte5=15).

  • Load puts the selected field right-justified in the destination; unused left bytes → 0.
  • If the field includes the sign (L=0), the loaded sign comes from memory; otherwise the sign is +.
  • The N variant flips the sign after loading.
  • Index registers I1..I6 hold sign + 2 bytes.

Results:

  1. LDAN 3000 Loads the whole word, then negates the sign → rA = − | 5 | 1 | 200 | 15 (fully defined).

  2. LD2N 3000(3:4) Loads bytes 3–4 (no sign) into I2, then negates → rI2 = −200.

  3. LDX 3000(1:3) Loads bytes 1–3 (no sign) into the right end of X; left two bytes become 0 → rX = + | 0 | 0 | 5 | 1 | ? (partially defined because byte 3 is unknown individually—only (3:4)=200 was given).

  4. LD6 3000 (default (0:2)) Loads sign and bytes 1–2 into I6 → two-byte value \(64·5+1=321\), sign + → rI6 = +321.

  5. LDXN 3000(0:0) Loads just the sign into X (magnitude becomes 0), then negates → rX = − | 0 | 0 | 0 | 0 | 0 (negative zero).

7 [M15]

Let a MIX word base be \(W=64^5\).

Registers and operand (as signed integers):

  • \(A = s_A\cdot |A|\), \(X = s_X\cdot |X|\) with \(0\le |A|,|X|<W\).
  • Memory operand \(M = s_M\cdot |M|\) with \(0\le |M|<W\).

Form the dividend from \(A\) and \(X\) by concatenation (sign from \(A\) only):

\[ D \;=\; s_A\cdot\big(|A|\cdot W + |X|\big). \]

Let the divisor be \(M\).

DIV M is defined without overflow iff

\[ |M|\neq 0 \quad\text{and}\quad \Big\lfloor\frac{|A|\cdot W + |X|}{|M|}\Big\rfloor < W . \]

When defined, compute

\[ Q \;=\; \Big\lfloor\frac{|A|\cdot W + |X|}{|M|}\Big\rfloor,\qquad R \;=\; (|A|\cdot W + |X|)\bmod |M| , \]

so \(0\le R<|M|\).

The results placed in the registers are

\[ \boxed{\;A \leftarrow (s_A s_M)\cdot Q\;},\qquad \boxed{\;X \leftarrow s_A \cdot R\;} \]

(i.e., the quotient gets the product of the signs of \(A\) and \(M\); the remainder gets the sign of \(A\)).

This covers all non-overflow cases; overflow occurs exactly when \(|M|=0\) or \(Q\ge W\).

8 [15]

Context (from the book’s last DIV example): Before DIV 1000, \(rA=-0\), \(rX=+\,1235\,0\,3\,1\), and \(M[1000]=-\,0\,0\,0\,2\,0\) (i.e., divisor \(|M|=200\) on the decimal MIX). In that example the result was \(Q=6{,}175{,}001\), \(R=101\). (Stack Overflow)

Now change only “rX before” to \(rX=-\,1234\,0\,3\,1\).

Key facts:

  • DIV forms the dividend from \((rA,rX)\) using the sign of A only; the sign of \(rX\) is ignored.
  • Here \(|A|=0\), so the dividend magnitude is just \(|X|\).
  • Replacing \(+\,1235\,0\,3\,1\) by $ -,1234,0,3,1$ reduces the magnitude by \(100^3=1{,}000{,}000\). Since \(1{,}000{,}000 / 200 = 5{,}000\), the quotient decreases by 5,000 and the remainder is unchanged.

Therefore

\[ Q' = 6{,}175{,}001 - 5{,}000 = 6{,}170{,}001,\qquad R' = 101. \]

Signs:

  • Quotient gets \(\operatorname{sign}(A)\times\operatorname{sign}(M)=(-)\times(-)=+\).
  • Remainder gets \(\operatorname{sign}(A)=-\).

Answer (byte form, decimal MIX):

  • \(rA \leftarrow + \;|\; 0 \;|\; 6 \;|\; 17 \;|\; 00 \;|\; 01\) (i.e., + | 0 | 617 | 00 | 01)
  • \(rX \leftarrow - \;|\; 0 \;|\; 0 \;|\; 0 \;|\; 01 \;|\; 01\) (i.e., - | 0 | 0 | 0 | 01 | 01)

So with rX before = − 1234 0 3 1, you get A = + 0 617 00 01 and X = − 0 0 0 01 01.

9 [15]

Operators that can turn OV on

  • ADD M — sets OV if the signed result won’t fit in one MIX word (|result| too large).
  • SUB M — same rule as ADD.
  • DIV M — sets OV if division is impossible (M = 0) or if the quotient doesn’t fit in a single word (|quotient| ≥ 10⁵).
  • INCA/DECA, INCX/DECX — these are “add/subtract an immediate value” on rA or rX; they can overflow just like ADD/SUB and therefore set OV.
  • NUM — converts the ten bytes in rA‖rX to a number in rA; sets OV if the decimal value is too large to fit in one word (common when the 10-digit value exceeds the MIX word range).

Notes • MUL never sets OV in MIX integer arithmetic because its 10-byte result is stored across rA‖rX. (Only floating-point multiply can affect OV, which we are excluding.) • Shift, load/store, compare, I/O, and most address-transfer ops do not affect OV.

Operators that can only clear OV

  • JOV … and JNOV … — both perform the jump test and reset OV to “off” as part of their semantics.

That’s all. In short: ADD, SUB, DIV, INCA/DECA/INCX/DECX, NUM can set overflow; JOV/JNOV clear it.

10 [15]

What the comparison indicator is

MIX keeps a three-way comparison indicator with states <, =, >. It changes only when an instruction explicitly performs a comparison; jumps merely test it.

Operators that can change it

  • CMPA — compare rA with M (using the given field F).
  • CMPX — compare rX with M (field F).
  • CMP1, CMP2, CMP3, CMP4, CMP5, CMP6 — compare rI1…rI6 with the selected field of M (default F normally the address field (0:2) for index-register compares).
  • FCMP — floating-point compare (if the floating-point option is present).

Notes/non-examples

  • The JL/JE/JG/JGE/JNE/JLE family of jumps only tests the indicator; they don’t change it.
  • Arithmetic, loads/stores, shifts, I/O (IN/OUT/IOC/JBUS/JRED), conversion (NUM/CHAR), and register-enter/increment/decrement instructions do not alter the comparison indicator.

So, the list is: CMPA, CMPX, CMP1–CMP6, and (if FP is implemented) FCMP.

11 [15]

These MIX instructions can change rI1:

  1. LD1 M(F) Loads rI1 from memory. The field specification F selects a field of M; when loading into an index register, MIX takes the address part (two bytes + sign) of the selected field. The result is therefore brought into the index range (−4095…+4095, with +0). Sign is the sign of the selected field.

  2. LD1N M(F) Like LD1, but the sign of the loaded quantity is negated.

  3. INC1 M Adds the effective address EA(M) to rI1. If the algebraic sum lies outside the index range, MIX reduces the magnitude modulo 4096 to bring it back into −4095…+4095 (zero is +0). (Overflow toggle is not used by index arithmetic.)

  4. DEC1 M Subtracts EA(M) from rI1, with the same reduction-to-range rule as INC1.

  5. ENT1 M “Enter” EA(M) in rI1 (i.e., rI1 ← EA(M)). Field F is ignored. Sign is the sign of EA(M).

  6. ENN1 M Like ENT1, but rI1 ← −EA(M).

  7. MOVE M,1(F) Any MOVE whose index field is 1 uses rI1 as the word count. During execution, rI1 is decremented by 1 per word moved and on normal completion rI1 = 0. If rI1 is initially 0, no words are moved and rI1 remains 0. (Other registers and addresses are advanced as defined for MOVE, but those changes don’t affect this list.)

Non-examples (often confused)

  • ST1 stores rI1 to memory; it does not change rI1.
  • CMP1 and the J1* jump family only test rI1; they don’t modify it.
  • Using rI1 in indexing for effective addresses does not alter rI1.

So the list is: MOVE, LD1, LD1N, INC1, DEC1, ENT1, ENN1.

12 [10]

Answer: INC3 0,3.

  • MIX first computes the effective address \(M\) by adding the selected index register to the address field: with address \(±AA=0\) and I-field \(=3\), we get \(M = 0 + rI3 = rI3\).
  • On “certain instructions” the address \(M\) isn’t a memory location; instead, the instruction adds \(M\) to an index register. INCi is that family: INC3 adds \(M\) to rI3. So INC3 0,3 performs \(rI3 \leftarrow rI3 + M = rI3 + rI3 = 2 \cdot rI3\).

Notes: Index registers hold two bytes plus sign; any operation that would make their magnitude exceed two bytes is undefined (the simulator explicitly checks size after LDi/LD iN/INC i). In normal ranges, the sign is preserved, so doubling keeps the sign of rI3.

We will use: - Jump effect on rJ. When a (conditional) jump actually occurs, rJ is set to the address of the next instruction (here, 1001). If no jump occurs, rJ is unchanged. - Overflow-clearing. JNOV clears the overflow toggle after testing; JOV has the symmetric behavior (and the exercise statement explicitly notes that JOV clears it).

  1. Replace by JNOV 1001:
  • Control flow: still executes next at 1001 in all cases (jump if OV=OFF, otherwise fall through).
  • Overflow: cleared afterward in either case.
  • rJ: different in general. With the original JOV 1001, rJ is set to 1001 iff OV was ON; with JNOV 1001, rJ is set to 1001 iff OV was OFF. So most of the time the rJ setting flips. (That’s the only behavioral difference.)
  1. Replace by JOV 1000
  • If OV=OFF: falls through to 1001, just like the original.
  • If OV=ON: it jumps to 1000, clearing OV; then re-executes with OV=OFF and falls through to 1001.
  • Net effect: same final state (including rJ = 1001 when OV was initially ON), but it takes extra time in the OV=ON case (you execute the instruction twice). So “no difference except time.”
  1. Replace by JNOV 1000
  • If OV=OFF: it jumps to 1000, clears OV (still OFF), and repeats forever → possible infinite loop.
  • If OV=ON: it does not jump, clears OV, and falls through to 1001. Hence this change “may lock the computer in an infinite loop.”

14 [20]

  1. Increment/Decrement family INCA, DECA, INCX, DECX, INC1–INC6, DEC1–DEC6 Make the effective address zero by choosing AA=0 and I=0. Then the instruction adds/subtracts 0 and changes nothing.

  2. Shifts SLA, SRA, SLAX, SRAX, SLC, SRC Choose F=0 (shift count 0). A, X (and A‖X), remain unchanged; indicators and memory are unaffected.

What doesn’t work (and why)

Loads: LDA, LDX, LD1–LD6, and the negated forms They overwrite a register with a memory field; we can’t guarantee equality with the current register.

Stores: STA, STX, ST1–ST6, STJ, STZ They write to memory; with unknown memory we can’t guarantee no change.

Arithmetic on A with a memory operand: ADD, SUB, MUL, DIV ADD/SUB would be a no-op only if the fetched value were provably 0 for all memory contents; there is no choice of F, I, AA that guarantees that without assumptions. MUL and DIV clearly change A‖X or the overflow state unless blocked by impossible cases (which still change OV).

Compare: CMPA, CMPX, CMP1–CMP6 (and FCMP if present) They always set the comparison indicator, which is a state change.

Conversion: NUM, CHAR They rewrite A and X.

Block move: MOVE It decrements rI1 during execution (and may copy data), so state changes unless rI1 were known to be 0—which we are not allowed to assume.

Jumps: all conditional jumps and JMP If a jump occurs, rJ is set; if it does not, rJ is not set. With unknown prior rJ and unknown condition values, there’s no way to make both

15 [10]

Each MIX word has 5 bytes → up to 5 printed characters per word. For character I/O devices, a block transfers a fixed number of words: typewriter and paper tape use 14 words per block; card reader and card punch use 16; the line printer uses 24.

So the alphanumeric characters per block are:

  • Typewriter or paper-tape: 14 words × 5 chars/word = 70 characters.
  • Card reader or card punch: 16 × 5 = 80 characters.
  • Line printer: 24 × 5 = 120 characters.

(“Five characters per word” follows because character devices map one MIX character per byte; a word has five bytes.)

16 [20]

Use overlapped MOVE to “smear” one zero across the block:

        STZ     0           ; make location 0000 = +0
        ENT1    1           ; rI1 := 0001 (destination start)
        MOVE    0(99)       ; copy 99 words from 0000 → 0001..0099 (one-by-one)

Why this works: On MIX, MOVE M(F) moves F words from M to the address in rI1, one word at a time, and then adds F to rI1. With overlap (here M=0, rI1=1), the same source word gets copied repeatedly, so the single zero at 0000 gets replicated through 0001..0099. Afterward rI1=0100.

Time (MIX units): STZ = 2u, ENT1 = 1u, MOVE = 1u + 2·F = 1 + 2·99 = 199u → 202u total.

If we optimize only for time (not length), the fastest is to store zeros directly:

        STZ 0
        STZ 1
        ...
        STZ 99

Each STZ costs 2u, so 200u total—slightly faster than the overlapped-MOVE trick, at the price of 100 instructions.

Notes

  • MOVE semantics (dest = rI1; overlap copies the same source word repeatedly) are exactly as stated in §1.3.1, incl. the example where one word is copied into several places.
  • MIX timing conventions: STZ (a store) = 2u; ENTi (an address transfer) = 1u; MOVE = 1u + 2 per word moved.

Here’s a clean solution for Exercise 17 (MIX §1.3.1). Assume at entry that rI2 = N (0 ≤ N ≤ 2999). Both programs start at location 3000.

  1. Shortest program (3-instruction loop)

This clears locations N, N−1, …, 0 with rI2 as a down-counter. It works for every N, including 0.

3000  LOOP   STZ   0,2      ; M = rI2 → clear that cell
3001         DEC2  1        ; rI2 ← rI2 − 1
3002         J2NN  LOOP     ; keep going while rI2 ≥ 0
3003         HLT

Why it works. STZ 0,2 stores zero at effective address 0 + rI2. Each iteration decrements rI2; the loop stops after clearing location 0 (when rI2 becomes −1).

  1. Fastest program (block moves with one final partial)

Use overlapped MOVE to smear the zero from 0000 across the range in big blocks. We copy 50 words at a time (≤ 63 so it fits in one byte), then finish with a last MOVE whose count equals the remainder rI2 (mod 50). The remainder is written into the F-byte of the final MOVE by self-modifying that instruction.

3000         STZ     0          ; make 0000 = +0 (seed)
3001         ENT1    1          ; rI1 = destination pointer (start at 0001)

; repeat: while rI2 ≥ 50 { MOVE 0(50); rI2 -= 50 }
3002  LBL    INC2   -50         ; try rI2 ← rI2 − 50
3003         J2N     TAIL       ; if negative, we didn't have 50 left → finish
3004         MOVE    0(50)      ; copy 50 zeros from 0000..0049 to rI1..rI1+49
3005         JMP     LBL

; finish: remainder 0..49 words
3006  TAIL   INC2    50         ; rI2 ← rI2 + 50  (now remainder R = 0..49)
3007         ST2     TAIL+1(4:4); put R into F byte of the next MOVE
3008         MOVE    0(0)       ; becomes MOVE 0(R) after the ST2 above
3009         HLT

Why it’s fast. MOVE 0(50) costs 1 + 2·50 = 101u for 50 words (≈2.02u/word). The loop overhead is tiny (a couple of 1u instructions per 50 words). The tail step performs one last MOVE of R ≤ 49 words; we set F = R by ST2 … (4:4), which writes the low byte of rI2 (0..63) into the F-byte (perfect for a remainder ≤ 49). No memory beyond 0..N is touched.

Edge cases. N = 0: the block loop is skipped; INC2 50 sets remainder R = 0; we self-modify the final MOVE to MOVE 0(0), which moves nothing—only location 0000 (seeded by STZ 0) is zero, which is exactly the requirement. N between 1 and 49: zero or one 50-block, then the tail MOVE with R = N. N a multiple of 50: the loop finishes with rI2 = −50; at TAIL we add 50 so R = 0; the final MOVE moves 0 words.

Here is the final answer:

  • Shortest: 3-instruction STZ/DEC2/J2NN loop.
  • Fastest: seed 0000 with zero; smear with repeated MOVE 0(50) blocks; finish with one self-modified MOVE 0(R) where R = N mod 50.

Key facts used

  • STZ stores zero to the selected field; STX stores the selected field of rX; field (0:1) = sign+byte1.
  • ENN? / ENT? / INC? set or add an immediate M to a register (A, X, or Ii).
  • SLAX/SRC shift/rotate across the 10 bytes rA|rX; signs are not affected.
  • ADD adds a memory value to rA and only sets overflow if the arithmetic result exceeds range. ([GNU][4])
  • NUM converts the 10 bytes of rA|rX (digits taken mod 10) to a decimal number in rA; CHAR writes the 10 decimal digits of rA back into rA|rX.
  • MIX word range is ±1,073,741,823 in binary-byte mode (b=64) and ±9,999,999,999 in decimal-byte mode (b=100).

Step-by-step (what becomes independent of the initial contents)

  1. STZ 1 → memory[1] := +0.
  2. ENNX 1 → rX := −1. ([GNU][2])
  3. STX 1(0:1) → store sign+byte1 of rX into memory[1]. Since rX=−1 has sign ‘−’ and byte1=0, memory[1] becomes “−0 00 00 00 00” (i.e., −0).
  4. SLAX 1 → rA|rX shifts left by one byte; rX becomes −b (its bytes are 00 00 00 01 00). ([GNU][3])
  5. ENNA 1 → rA := −1. ([GNU][2])
  6. INCX 1 → rX := (−b)+1 = −(b−1). (So rX’s bytes are 00 00 00 00 (b−1).) ([GNU][2])
  7. ENT1 1 → rI1 := 1. ([GNU][2])
  8. SRC 1 → rotate right one byte across rA|rX. • rA bytes become [b−1,0,0,0,0] (still negative). • rX bytes become [1,0,0,0,0] (still negative). ([GNU][3])
  9. ADD 1 → add memory[1] (−0) to rA: rA unchanged (still negative). Overflow not set by adding zero.
  10. DEC1 −1 → rI1 := 1 − (−1) = 2.
  11. STZ 1 → memory[1] := +0 (explicitly clears the earlier “−0”).
  12. CMPA 1 → compare rA (negative) with +0; comparison indicator = “less”.
  13. MOVE −1,1(1) → source EA = rI1 + (−1) = 2−1 = 1; move one word from memory[1] to memory[2]; rI1 increments by 1 → rI1=3. (So memory[2] becomes +0.)
  14. NUM 1 → treat the 10 bytes in rA|rX as decimal digits modulo 10: digits are [(b−1) mod 10, 0,0,0,0, 1,0,0,0,0], so rA becomes the decimal number 3000010000 if b=64, or 9000010000 if b=100, with rA’s sign negative.) • In binary-byte mode (b=64), this exceeds ±1,073,741,823 ⇒ overflow set ON. • In decimal-byte mode (b=100), it fits in ±9,999,999,999 ⇒ overflow OFF.
  15. CHAR 1 → write the 10 decimal digits of rA back into rA|rX; rX now holds the character bytes “10000” (i.e., 31,30,30,30,30). (The sign bits are not used by the digits; rA keeps its sign.)
  16. HLT 1 → halt.

What you can state at the end

  • rI1 = 3.
  • memory changes: memory[1] ends as +0; memory[2] is +0 (copied by MOVE).
  • comparison indicator after CMPA 1 = L (A < memory[1]).
  • overflow: • binary-byte mode (b=64): ON (set by NUM); • decimal-byte mode (b=100): OFF.
  • rX holds the five character digits “10000” (bytes 31,30,30,30,30) after CHAR (independent of byte size).

rJ ← m + 1

without reducing modulo 4000. Since m ∈ {0,…,3999}, we always get rJ ∈ {1,…,4000}—never 0. To make rJ = 0 you’d need m = −1, which a program cannot execute. (Zero can be forced only by external means—cf. the “GO button”, exercise 26.)

Note: when returning from a subroutine we typically use JMP 0,J. If rJ = 4000 (from a call placed at 3999), the effective address 0 + rJ = 4000 is interpreted modulo 4000 as address 0, so control returns correctly—yet the content of J itself is not 0.

Idea: Temporarily place JMP RESUME at address N−1, jump there with JSJ (which doesn’t touch J), let that JMP set J = (N−1)+1 = N, then restore the word we modified.

        ORIG   3000

        STA    -1,4           * M[N-1] := JMP RESUME (temporary)

RESUME  STX    -1,4           * restore original M[N-1]
        HLT    0              * done

INST    JMP    RESUME         * instruction constant we copy to N-1
        END    START

  1. Without using partial-field loads/stores (only whole-word ops)

We treat the word as a base-64 number. Repeatedly:

  1. take the low byte as source % 64 (via DIV 64, remainder goes to X),
  2. update acc := acc*64 + remainder,
  3. set source := source / 64 (the quotient from DIV, which we saved).

This needs one constant (64) and two temporaries; with only whole-word ops that’s the tightest practical layout.

        ORIG   3100
START   LDX    0200        * X := source = + a b c d e
        ENTA   0           * A := acc = 0
        ENT1   5           * 5 bytes to process

LOOP    STA    ACC         * save acc
        ENTA   0           * make dividend (A,X) = (0, source)
        XCH                 * A := source, X := 0
        DIV    K64         * A := source/64 (quotient), X := source%64 (a byte)
        STA    QUO         * remember quotient for next round
        XCH                 * A := remainder, X := quotient
        STA    REM         * stash remainder
        LDA    ACC         * rebuild acc := acc*64 + remainder
        SLA    1           * *64
        ADD    REM
        LDX    QUO         * next source := quotient
        DEC1   1
        J1P    LOOP

        * on exit, A holds the reflected word
        HLT    0

ACC     CON    0
REM     CON    0
QUO     CON    0
K64     CON    64
        END    START

Memory used: 12 instruction words + 4 data words (ACC, REM, QUO, K64). Why this fits the constraint: with no partial-field access you need (i) a way to peel bytes—DIV 64 requires the 64 constant—and (ii) somewhere to hold the running accumulator while using (A,X) as the dividend; the two temporaries avoid clobbering the quotient and remainder. Here’s a detailed, backward-compatible “Mixmaster” design that answers 25 and shows how to reuse the I-field profitably while keeping every correctly written MIX program runnable unchanged.

Goals (constraints)

  1. Machine word, byte size, opcodes, and all defined MIX behaviors remain valid.
  2. A MIX program that only uses documented features (e.g., I∈{0..6}) keeps the same semantics.
  3. New power comes only from previously unused encodings (e.g., I>6, unused F-codes, unused opcodes) or from optional mode bits whose reset value equals MIX behavior.

Better use of the I-field (backward-compatible)

The MIX I-field is 6 bits but only values 0..6 are used (selecting no index or I1..I6). We keep those intact and assign new meanings to the unused encodings:

I-field layout (Mixmaster)

Let I = b5 b4 b3 b2 b1 b0 (b0 = LSB).

  • idx = (b2 b1 b0) selects the base register:

    • 000 = none (MIX “no index”)
    • 001..110 = I1..I6 (as in MIX)
    • 111 = PC (location counter) ⟹ enables PC-relative addressing

  • IND = b3 (indirection)

    • 0 = direct (MIX default)
    • 1 = one level of indirection: the effective address is taken from the word at the computed address (no chaining unless an op explicitly allows it)

  • SC = (b5 b4) (scale for the index term)

    • 00 ⇒ ×1 (MIX default)
    • 01 ⇒ ×2
    • 10 ⇒ ×4
    • 11 ⇒ ×8

Effective address

EA = (ADDR + SCALE(idxReg) + BASE) mod MEMSIZE
where   SCALE(idxReg) = (SC ? {1,2,4,8} : 1) * (contents of chosen base reg)
        BASE = 0 in MIX-compat mode; see “large memory” below
if IND=1: EA := contents_of_word_at(EA).ADDR   (one level)

Compatibility: All existing code has I≤6, IND=0, SC=00, so the rule collapses to MIX’s ADDR + Ii with no change.

Payoffs

  • PC-relative addressing (I=7) for position-independent code, short branches, and better relocation.
  • Scaled indexing removes pre-multiply steps for arrays of records with fixed word length (×2, ×4, ×8).
  • Single-level indirection gives true pointers without extra loads.

Optional: Immediate operands via I-patterns

Reserve two I-patterns for immediates (used only by arithmetic/compare loads):

  • I = 7 with IND=1 and SC=00 ⇒ treat ADDR (and F) as a literal (sign-extended) operand: e.g., ADDA 5,7:1 (assembler sugar ADDA #5). No MIX program ever uses such I-codes, so compatibility holds.

Memory and addressing extensions

  • B (base) register (2 bytes, sign) holds a segment number. Default B=0 on reset ⇒ MIX addresses map unchanged.
  • New instructions LDB M, STB M and privileged SETB imm. The hardware forms the true physical address as BASE*4096 + EA, giving up to 4096 segments of 4096 words (or larger if desired). Old programs (B=0) see the original 0..3999 space.
  • Optional per-process limit register to trap out-of-bounds for safety (default unlimited for full compatibility).

F-field and opcode extensions (safe because many encodings are unused)

  • Keep all MIX F-meanings; use unused F codes to define:

    • variable-length shifts where the count comes from a register (e.g., SLA r with F indicating the source register),
    • logical operations on full words: AND, OR, XOR (F selects field subranges when desired),
    • block move/fill: MOVN, MOVZ (F selects element size as in load/store fields).

  • Use unused opcodes for PUSH/POP (stack in memory), CALL/RET (JSJ remains; these are conveniences only).

Registers and arithmetic

  • Add I7..I10 as true index registers, but expose them only through the new I-field scheme (idx=7 already maps to PC; the extras are accessible via new opcodes LDI7, STI7, …). Existing code doesn’t see them.
  • 2-word (AX) add/subtract and multiply with saturating/trap modes selectable by F—still leaving the classic MIX MUL/DIV unchanged.
  • Fast CMPM (compare A with memory word) and ADDM, SUBM (A ← A ± M) opcodes to reduce load/store traffic.

Jumps and conditions

  • New PC-relative jumps (via I=7).
  • Indexed conditional jumps: allow condition + index test in one instruction (e.g., “J1Z & JXNZ” combined using unused opcodes) to shorten loops; classic jumps remain as-is.

I/O and devices

  • Expand device IDs to 0..63; keep 0..20 exactly as in MIX.
  • Asynchronous channel ops with completion flags in a status word; old synchronous behavior retained by default.

Traps and compatibility mode

  • An Extended-Mode bit (reset = 0) controls whether undefined encodings trap or use the new features. In reset state, the machine behaves exactly like MIX. Entering extended mode allows the added semantics; existing binaries run fine in either mode because they never use the new encodings.

  • We “make better use of the I-field” by turning its 6 bits into an addressing-mode mini-ISA (index select, scale, indirection, and PC-relative)—all using values the original MIX never produced.

  • We extend memory and add performance features without changing any defined MIX behavior.

  • Every suggestion strictly preserves the contract: all correctly written MIX programs run unchanged; new power appears only when programmers choose the new encodings.

Alright—this is the classic “bootstrap loader” for MIX. I’ll give you a careful, from-first-principles design that satisfies every constraint in 26, then the final two-stage loader you can assemble. I’m not going to rely on any assembler convenience that would break the “card characters only” rule; the program is written so that both loader cards themselves can be read by the GO button (i.e., read in character code) and then executed as instructions. We also avoid opcodes that the statement warns may be unreadable on some readers (no ENT, INC, CMP on the cards).

The solution is in three parts:

  1. What GO actually does, and the consequences for how a bootstrap must be shaped.
  2. The loading algorithm (how to parse “information cards” and the “transfer card” with only character data available).
  3. A compact two-card program layout that implements the algorithm with only “safe” characters, with a readable MIXAL listing you can assemble and test. (It fits in 32 words; stage-1: 2 words, stage-2: 30 words.)
  1. What the GO button gives us

Pressing GO is essentially IN 0(16) on the card reader (unit 16) performed by hardware, followed by:

  • the 80 columns are deposited in character code into words 0000…0015 (5 columns per word), with positive signs,
  • when the device goes not-busy, the CPU does JMP 0 and sets J ← 0 (overflow toggled).

So the first card we feed must already be executable even though its five bytes per word are character codes. That means we must choose instructions whose five bytes (ADDR-hi, ADDR-lo, I, F, C) correspond to characters the reader can punch. The problem warns that some character codes (20, 21, 48, 49, 50, …) aren’t guaranteed readable; and, in particular, the encodings for the opcodes ENT, INC, CMP collide with “bad” characters. Therefore the bootstrap must avoid those opcodes on the cards. (We can, of course, emulate them with allowed instructions.)

We may, however, execute IN a(16) with any address we choose, so our first card can read the second loader card into 0016…0031 (or anywhere that doesn’t clobber itself) and then jump there. Information cards are guaranteed to load data at locations > 100, so the loader can safely live below 0100.

  1. Loading algorithm (what the second stage does)

Each information card has:

  • col 1–5: ignored,
  • col 6: k in {1,…,7} = how many words to load from this card,
  • col 7–10: four decimal digits giving the address of the first word on this card (addr0), and
  • col 11–20, 21–30, …, 71–80: each 10-column field is one word in base-10, signed by an overpunch in the last column (the last character code is 0..9 for positive, 10..19 for negative, representing 0..9 with a “minus” 11-punch).

The transfer card is TRANS0nnnn in columns 1–10 (i.e., word 0 equals “TRANS0” in character codes and word 1 holds the four digit start address nnnn), and the remainder of the card is irrelevant. When we see it, we jump to nnnn.

The second stage therefore does, in a loop:

  1. IN 0(16) — read the next card into 0000…0015.

  2. If word 0000 equals the characters “TRANS” (we compare by subtraction and JAZ, not CMP), branch to the transfer path; else parse as an information card.

  3. k ← col6, addr ← dec(col7…10).

  4. For i=0..k-1:

    • Parse one 10-column decimal field at columns 11+10i … 20+10i.
    • Convert to a signed MIX integer using repeated A ← A*10 + digit, where digit ← byte MOD 10 and the sign bit is taken from the last byte (negative iff last-byte ≥ 10).
    • STA addr+i.

  5. Repeat step 1.

All parsing is done with partial-field loads from the card image at 0000…0015; we never rely on any punch-limited instruction, and we never need ENT/INC/CMP.

  1. A compact two-card loader

Stage-1: “copy in stage-2 and jump”

Two words at 0000..0001 after GO:

        ORIG   0000
STAGE1  IN     16,16        * read the second loader card into 0016..0031
        JMP    0016         * execute it

These two instructions can be laid out in characters the readers accept (opcodes chosen accordingly); their addresses (0016 and 0016) have only decimal digits, so the two address bytes are harmless.

Stage-2: the real loader (fits in 30 words)

Below is a straight, compact MIXAL program that implements the algorithm. It uses only the safe instruction set on cards (LDA/STA/LDX/STX/ADD/SUB/MUL/DIV/SLA/SRA/SLC/SRC/Jxx/IN/JMP) and no ENT/INC/CMP. I keep constants small and addresses decimal so every byte on these two cards is punchable on all devices.

* ---------- constants & work cells (kept tiny and punch-safe) ----------
TEN     CON    10
ZERO    CON    0
ONE     CON    1

  • Only safe opcodes are on the cards. We explicitly avoid ENT, INC, and CMP. Loop control is done with ADD/SUB on counters in memory and conditional jumps (JAZ, JANN), which are safe to punch.
  • All addresses appearing in the loader cards are small decimal numbers (0, 1, 2, …, 31, 100…), so the address bytes are harmless characters.
  • We never store into 0000…0031 except as the card image; actual program data is guaranteed to be > 0100.
  • Parsing uses partial-field loads M(1:1)…M(5:5) to fetch digits straight from the card image; the sign is derived from the last byte by the “≥10” test (we avoid CMP by subtracting 10 and using JANN).
  • The program is deliberately written “straight” (no self-modifying field tricks), so it remains readable and easy to validate. If you want to shave a couple of words you can collapse some repeated MUL TEN sequences by unrolling constants, but we stay < 32 words already.

Size check

  • Stage-1 = 2 words (0000…0001).
  • Stage-2 = 30 words (0016…0031).
  • Total = 32 words ⇒ exactly two cards, as required.

Using the loader

Deck layout:

  1. Card 1: stage-1 (the two instructions above, in character code).
  2. Card 2: stage-2 (the 30-word body, in character code).
  3. Information cards in the specified format.
  4. Transfer card, columns 1–10 = TRANS0nnnn.

Push GO. Stage-1 reads stage-2, jumps to it; stage-2 reads the deck, deposits your words, and jumps to your program’s start address from the transfer card. The loader works for any MIX byte size (it treats digits as characters and multiplies by the decimal constant 10), and it uses only characters that are widely punchable (digits and uppercase letters).
