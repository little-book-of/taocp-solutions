1.4.3.2 Trace routines

1 [22]

On MIX, the only way to set rJ is to execute a jump whose address is located at position α; executing JMP … from location α sets rJ to α+1. Therefore, to restore rJ to the “start” value saved earlier (call that value J0), we can:

  1. Temporarily write the instruction JMP *+1 into memory at address J0−1;
  2. Execute that instruction in place; this sets rJ ← J0;
  3. Put the original word back (so we don’t alter the traced program);
  4. Finally, exit the trace routine without disturbing rJ (so use JSJ, not JMP). Knuth’s answer notes this last point explicitly: using JSJ is crucial.

Patch (drop-in replacement for the leave block)

Below is a compact, self-contained replacement for the “leave” part of the trace routine. It assumes your routine already saved the original J in JREG (as in the text), and it uses I1 and X temporarily but restores them.

* --- NEW LEAVE SEQUENCE (restores rJ) -------------------------
XSV    ORIG  *+1          * temp: save X
I1SV   ORIG  *+1          * temp: save I1
ONE    JMP   *+1          * literal word used as a copier: "JMP *+1"

LEAVE  STX   XSV          * save rX; we’ll need it for the exit jump later
       ST1   I1SV         * save rI1; we’ll use I1 as a pointer
       LD1   JREG(0:2)    * I1 := saved start address J0
       LDA   -1,1         * A  := original contents at address J0-1
       LDX   ONE          * X  := the word "JMP *+1"
       STX   -1,1         * write that jump at address J0-1
       JMP   -1,1         * execute it → rJ := (J0-1)+1 = J0

* we’re now back here, with rJ restored to J0
       STA   -1,1         * restore the original word at J0-1
       LD1   I1SV         * restore rI1
       LDX   XSV          * restore rX
       LDA   AREG         * restore rA (as in the book)
LEAVEX JSJ   *            * exit without changing rJ

Why each step is safe

  • No persistent self-modification: we restore the overwritten word at J0−1 before leaving.
  • Minimal register disturbance: only rA, rX, rI1 are touched temporarily, and all three are restored; the comparison indicator and overflow toggle remain unaffected (matching the section’s design goals).
  • Correct exit: JSJ leaves rJ as we’ve just set it to J0; using JMP here would clobber rJ again.

Where this fits in the printed routine

You replace the printed lines that restore rA and jump out (the LEAVE … / LEAVEX … pair near the end) with the block above. Those two lines, as printed, don’t fix rJ; the patch does.

2 [26]

Design

  • A 100-word circular buffer at label BUF collects ten 10-word records; when full, we OUT it to tape 0.
  • We keep an index pointer in rI1 that advances by 10 after each record. To persist the pointer across iterations, we use a neat MIX trick: self-modify the address field of the instruction PTR ENT1 -100, i.e., store the current rI1 back into PTR(0:2) each time; on the next pass, ENT1 reloads the saved pointer.
  • We never disturb the traced program’s state (except briefly using rA and rI1 inside the tracer); rA is restored before the traced instruction executes, as in the original routine.
  • We wait for the device when needed and flush the final partial buffer at the end (the final flush routine is a small add-on you call once after tracing stops).

Patch (delta against the routine on pp. 212–213)

The comments say exactly where to splice these lines relative to the original labels/lines.

* --- Reserve a 100-word buffer and constants (can be placed near the end) ---
        ORIG    *+99            * reserve 100 words below BUF
BUF     CON     0               * BUF-99 .. BUF are the 100-word block
PTR     ENT1    -100            * rI1 := saved pointer (will be self-modified)
EIGHT   CON     8

I1SAVE  CON     0               * local save for rI1

* --- At ENTER: after lines 02–04 (save tape-pointer register) ---
        ST1     I1SAVE          * save caller’s rI1; tracer will use rI1

* --- After line 05–07 (after CYCLE STA PREG(0:2)) ---
        JBUS    *(0)            * ensure tape 0 not busy from a prior OUT
        STA     BUF+1,1(0:2)    * Word1 (0:2): location (A holds it here)
        ST1     I1REG           * (if you keep a global I1REG like the book)

* --- After lines 08–11 (after fetching the instruction into INST) ---
        STA     BUF+2,1         * Word2: instruction word

* --- Still before the traced instruction executes: dump “before” registers ---
        LDA     AREG            * Word3: A (before execution)
        STA     BUF+3,1

        LDA     I1SAVE          * Word4: I1 (before) — we changed rI1
        STA     BUF+4,1
        ST2     BUF+5,1         * Words5–9: I2..I6 (unchanged by tracer)
        ST3     BUF+6,1
        ST4     BUF+7,1
        ST5     BUF+8,1
        ST6     BUF+9,1

        STX     BUF+10,1        * Word10: X (before)
        LDA     JREG(0:2)
        STA     BUF+1,1(4:5)    * Word1 (4:5): J (before)

* --- Pack CI (2/1/0) + 8*OVF into Word1 (3:3) ---
        ENTA    0               * start with 0 (= “<”)
        JL      CI_DONE         * if “<”, keep 0
        JE      CI_EQ
        ENTA    2               * “>” → 2
        JMP     CI_OV
CI_EQ   ENTA    1               * “=” → 1
CI_OV   JNOV    CI_NOOV         * if overflow off, skip add
        ADD     EIGHT
CI_NOOV STA     BUF+1,1(3:3)
CI_DONE NOP

* --- Advance pointer and flush when block is full ---
        INC1    10              * next record slot within the 100-word block
        J1N     KEEP_FILLING    * still negative → not yet full
        OUT     BUF-99(0)       * write the 100-word block to tape unit 0
KEEP_FILLING
        ST1     PTR(0:2)        * persist pointer by self-modifying ENT1
        LD1     I1SAVE          * restore caller’s rI1 before continuing

* (the rest of the original routine continues: execute INST, etc.)

Notes and rationale:

  • Where values come from. Right after CYCLE STA PREG(0:2) the A-register holds the next instruction’s location, so we can store Word 1’s (0:2) immediately. After PREG LDA * / STA INST we have the instruction word (Word 2). AREG, JREG, and I1..I6 are the tracer’s saved copies of the machine registers before the step; we read them before executing the instruction so the “before” requirement is met.
  • CI/OV packing. We derive the comparison indicator via branches (JL/JE) and add 8 iff overflow was set, then store that byte into field (3:3) of Word 1, exactly as specified by the exercise.
  • Buffering & I/O. The ORIG *+99 trick makes BUF-99 .. BUF a contiguous 100-word area. We fill records using indexed addresses BUF+… ,1, starting with rI1 = −100; after ten records (INC1 10 ten times) rI1 becomes 0 and we OUT BUF-99(0) to tape 0. We also issue JBUS *(0) up front to avoid overlapping I/O. A one-time “final flush & rewind” routine should be called after tracing stops to write a partial block and rewind unit 0, as suggested in the official notes.

Why this is correct

  • Every required field is captured before the traced instruction executes (we only restore rA just before execution, per the original routine’s discipline).
  • The 100-word block layout matches the exercise’s request for “nine more ten-word groups in the same format,” enabling efficient tape output with minimal space in core.
  • We preserve the traced program’s state: only rA and rI1 are touched inside the tracer and then restored; CI and OV are left unchanged (except for the unavoidable tests), as the text advises.

3 [10]

Why tape > printer for a trace routine

  1. Throughput and latency Line printers are slow, mechanical devices. Knuth’s own timing note for printing a line is \(T=7500u\); if you print after (nearly) every instruction, output time dominates and the trace becomes painfully slow. Tape is block-oriented and written in large chunks, so per-word overhead is vastly lower; you can buffer many words and write them as a single I/O, reducing device start/stop costs and the number of waits. Knuth also emphasizes that, for trace programs, “the running time is consumed by output anyway,” so minimizing per-step output costs is crucial—tape helps do exactly that.

  2. Buffering and structure Exercise 2 explicitly formats the trace as fixed ten-word groups packed into 100-word tape blocks. That’s ideal for efficient sequential I/O and easy post-processing. Printing would force you to convert binary register contents into text and paginate—more CPU time and more I/O operations.

  3. Non-interference with the program under test A traced program “must not use the output device on which tracing information is being recorded.” If you occupy the printer for tracing, the program can’t print its own output. Using tape unit 0 for the trace keeps the printer free for the target program (or human-readable summaries later).

  4. Post-processing flexibility A tape log can be rewound, scanned, filtered, summarized, or reformatted into human-readable reports after the run (e.g., reconstructing flow, computing instruction frequencies, or converting only selected steps to text). With direct printing you pay the formatting cost up front and flood paper with data you may not actually need.

  5. Lower perturbation of behavior Because tape output is faster and can be buffered, it slows the traced program less than constant printing—giving a trace that is closer to the program’s “natural” behavior (though still slower, as the text notes).

In short: tape is faster, buffer-friendly, nonintrusive, and analyzable offline, whereas printing is slow, formatting-heavy, and monopolizes a device the program might also want to use.

4 [25]

Here’s what happens if you make the trace routine trace itself by inserting, just before label ENTER, the two instructions

  • ENTX LEAVEX (set rX to the address of LEAVEX), then
  • JMP **+1 (jump into ENTER),

as the exercise proposes. (This is the routine on pp. 212–214.)

What the routine expects at entry

The trace routine is designed to be entered with rJ already set to the starting address to trace and rX set to the address where tracing should stop. The code then saves rA and rJ, takes the starting location from rJ, and begins its fetch–decode–execute cycle on the “external” program’s instruction at that location.

Step-by-step with your two extra instructions

  1. Prelude executes. ENTX LEAVEX makes rX = address(LEAVEX); JMP **+1 transfers control to ENTER. But a JMP also loads rJ with the address of the JMP itself (the prelude location). Thus, when the trace routine starts, its idea of the “start address” (from rJ) is the prelude JMP, not the first instruction of the routine proper. The routine saves rA and rJ, then sets its “next instruction” pointer to that JMP.

  2. First traced instruction is that prelude JMP **+1. The tracer recognizes a jump and rewrites the instruction word it will execute so that execution flows through its JUMP handler, which updates the simulated J-register and sets the next location to ENTER. (That is the normal way this tracer executes an external jump.)

  3. Now it begins tracing the trace routine’s own body. It will fetch and execute, under trace, the instructions beginning at ENTER (e.g., STX TEST(0:2), STA AREG, …). So far, so good.

  4. Self-destruction on STJ. Very soon the tracer reaches the routine’s own STJ JREG (the instruction that saves rJ early in the routine). The tracer deliberately rewrites any external STJ into STA before executing it, so the external program can’t disturb its simulated J-register. But now the “external program” is the tracer itself, so the tracer changes its own STJ JREG into STA JREG in memory and then executes that store-A. This destroys the saved J value (JREG now holds whatever was in rA), and the code itself has been permanently modified.

  5. Loss of the starting location and derailment. Immediately afterward the tracer does LDA JREG(0:2) to get the next instruction location from JREG. Since JREG no longer contains a proper address (it was just overwritten with A), the tracer loads a garbage address, points its “next instruction” there, and proceeds to fetch/execute nonsense. From this point the trace prints nonsense or loops/crashes; it certainly doesn’t produce a coherent self-trace.

With only those two added instructions, the given trace routine cannot trace itself. It wrecks its own code/data because its “don’t let the external program alter J” trick (rewriting STJ to STA) applies to the tracer’s own STJ, corrupting JREG and sending execution off the rails.

This is exactly why the text restricts traced programs so they don’t store into the trace routine’s locations, and why a robust “self-trace” needs extra machinery (e.g., two copies tracing each other or special guards), as hinted by the following exercises.

5 [28]

What actually happens

  1. The tracer that you start keeps control. Suppose we start tracer T₁ with J pointing at the entry of tracer T₂. T₁ runs its own code, repeatedly:
  • fetches the current instruction from T₂’s memory at address PREG₁,
  • copies it into T₁’s INST, possibly edits it,
  • executes that copy at T₁’s INST,
  • increments T₁’s PREG to the next address.

Crucially, the instruction that actually executes is always the copy in T₁’s INST, not the one in T₂. So T₂ never “takes over.”

  1. T₂ “tries” to trace T₁, but only as data. Because T₂’s code is what T₁ is stepping through, T₂ will execute its own statements like “store the next address in PREG₂,” “read the instruction from T₁ into INST₂,” etc. But when the time comes to “execute the external instruction,” the execution point is T₁’s INST (by design of the trace routine), not T₂’s INST. Hence T₂’s attempt to trace T₁ never actually runs any of T₁’s instructions; it merely manipulates T₂’s bookkeeping cells (PREG₂, INST₂, AREG₂, …). This respects the routine’s restriction that the traced program mustn’t clobber the tracer’s own working cells, because each tracer uses its own PREG, JREG, AREG, etc. (the “(a)–(c)” restrictions listed under the exercises).

  2. No infinite regress, no mutual recursion of control. When T₁ encounters a jump in T₂’s code, it rewrites the copied instruction so the jump targets T₁’s JUMP label; control therefore stays inside T₁, which then updates the simulated J and continues. T₂ never gains control flow; it remains merely the subject being stepped.

Net effect

  • T₁ simply traces T₂ as if T₂ were an ordinary program. You get a trace of T₂’s instructions executing; T₂’s internal attempt to trace T₁ never actually executes T₁’s instructions.
  • No paradox or endless slowdown chain occurs here (contrast with the self-trace thought experiment of exercise 4, where self-modification can immediately derail the trace via PREG being clobbered).
  • If both copies tried to print tracing information to the same device, that would violate restriction (6b) and garble output, but the skeleton routine shown doesn’t output registers at all (that part was explicitly omitted).

Conclusion. With two copies placed in disjoint memory and each set to trace the other, the one you start (T₁) retains control and cleanly steps through the other (T₂). There’s no infinite regress; it behaves just like tracing any other (nontrivial) program, only slower because that program happens to be a tracer itself.

6 [40]

Here’s a clean way to do 1.4.3.2 — Exercise 6 (“write a trace routine that is capable of tracing itself … at slower and slower speed, ad infinitum, until memory is exceeded”), built directly on Knuth’s trace routine in §1.4.3.2. I’ll (1) recall how the given trace routine works, (2) show the minimal self-bootstrapping change that makes it trace itself once (the idea of Ex. 4), then (3) turn that into an infinitely nestable, re-entrant trace by giving it tiny per-level “frames,” so multiple copies of the trace can be active at once without stepping on each other’s working cells, and (4) point to where to splice in the “print one step” action. Citations refer to the exact listing and the exercise page in the book PDF you linked.

  1. What the provided trace routine already does

Knuth’s routine is invoked when the driver jumps to label ENTER with:

  • rJ = start address of the code to trace,
  • rX = address where tracing must stop (exclusive).

On entry it saves rA and rJ, loads the simulated program counter from rJ, fetches the next instruction word, rewrites certain instructions (e.g., jumps) so the monitor keeps control, executes the instruction via the pseudo-op line INST NOP *, advances to the next location, and repeats; special logic handles jumps and JSJ so the simulated J is correct. Finally it restores rA and exits at LEAVE/LEAVEX. (See lines 02–08 for setup, 10–16 decode, 27–47 jump handling, 34/36–38 execute-and-advance, 48–49 leave.)

Knuth explicitly notes this listing is just the control core of a trace: it doesn’t print; printing/register-dump is left to exercises.

  1. Make it trace itself once (the Ex. 4 idea you build on)

Place a tiny prologue immediately before label ENTER:

SELF   ENTX  LEAVEX      ; inner trace stops at our LEAVEX
       JSJ   ENTER       ; start a fresh trace instance, sets rJ to SELF+1
ENTER  ...               ; (the original routine follows)

Why this works (and why we use JSJ, not JMP):

  • When the outer monitor first “executes” JSJ ENTER as an instruction of the traced program, its jump logic recognizes a jump and treats JSJ specially: it does not overwrite the simulated J with a “fall-through” value (lines 39–47 check for JSJ and avoid updating JREG in that case), then sets the simulated PC to ENTER. Thus the inner monitor instance starts with rJ = SELF+1 and begins tracing the trace program itself starting at ENTER. From now on, the outer level prints every step the inner monitor takes, and the inner monitor will (once we add printing) print every step of its own traced program.

Knuth asks you to think about the simpler version with ENTX LEAVEX; JMP *+1 in Ex. 4; swapping the second instruction to JSJ ENTER is the right subroutine-style call that seeds the next level with a proper J. The exercise statement for 6 explicitly says “in the sense of exercise 4,” i.e., the same bootstrapping trick but continuing indefinitely.

If you stop here, you can get one active monitor tracing itself, but as soon as that inner instance tries to start another instance, all copies will fight over the same scratch words (AREG, JREG, PREG, INST, INST1, TEST, …). We need re-entrancy.

  1. Make the trace re-entrant with per-level frames (the key to “ad infinitum”)

The original routine uses fixed global cells (labels like AREG, JREG, PREG, INST, INST1, TEST, LEAVEX). To support unlimited nesting, give each active “level” its own copy of those words—i.e., a frame—and index all accesses via, say, rI1 as a frame pointer.

3.1 Define the frame layout (somewhere near the routine)

; One frame per nesting level (10 words is enough for the listed cells)
FRAME_SZ  EQU  10
FR_AREG   EQU  0     ; saved rA
FR_JREG   EQU  1     ; saved rJ (simulated)
FR_PREG   EQU  2     ; simulated PC (location being traced)
FR_INST   EQU  3
FR_INST1  EQU  4
FR_TEST   EQU  5
FR_LEAVEX EQU  6     ; we keep its absolute addr here for this level
; ... a couple spare words

Reserve a small stack of frames (e.g., 100 levels):

FRAMES    ORIG *+100*FRAME_SZ

Maintain rI1 = base of current frame. On each entry, bump it to the next frame; on each leave, drop it.

3.2 Patch the routine to index frame fields by ,1

Example patches (showing the idea; you’d apply this uniformly):

  • Save/restore rA:

    • old: STA AREG / LDA AREG
    • new: STA FR_AREG,1 / LDA FR_AREG,1

  • Save rJ and use it later:

    • old: STJ JREG / LDA JREG(0:2) etc.
    • new: STJ FR_JREG,1 / LDA FR_JREG,1(0:2)

  • Keep the “next instruction address”:

    • old: STA PREG(0:2) / LDA PREG(0:2)
    • new: STA FR_PREG,1(0:2) / LDA FR_PREG,1(0:2)

  • INST and INST1 similarly become FR_INST,1 and FR_INST1,1.

  • The exit address:

    • On entry, copy rX into FR_LEAVEX,1(0:2) and test against that field instead of a single global TEST/LEAVEX.

Concretely, the very first lines change to (comments show what moved):

SELF      ENTX  FR_LEAVEX,1(0:2)   ; stop at this level’s LEAVEX
          JSJ   ENTER

ENTER     ENTA  FRAME_SZ
          ENT2  FRAMES             ; rI2 = base of all frames
          ADD   LEVEL              ; LEVEL holds current depth (init 0)
          MUL   =1=                ; (cheap way to scale by FRAME_SZ if needed)
          ADD   rI2                ; compute next frame base
          ST1   LEVEL              ; bump depth counter
          LDX   =0=
          STX   FR_AREG,1          ; clear a few words if desired

          STX   FR_TEST,1(0:2)     ; put stop addr from rX
          STX   FR_LEAVEX,1(0:2)
          STA   FR_AREG,1
          STJ   FR_JREG,1
          LDA   FR_JREG,1(0:2)     ; start location for tracing
CYCLE     STA   FR_PREG,1(0:2)
...
TEST      DECA  *                   ; compare FR_PREG vs FR_LEAVEX for exit
          JAZ   LEAVE
PREG      LDA   *                   ; fetch instr at FR_PREG
          STA   FR_INST,1
...

(You don’t need to literally copy/paste the above; the point is every variable access acquires a ,1 and the constant cells become offsets within the current frame.)

3.3 Leave drops the frame and returns

At LEAVE:

LEAVE     LDA  FR_AREG,1
          ; rI1 := rI1 - FRAME_SZ (restore previous level’s frame)
          ENTA  -FRAME_SZ
          ADD   rI1
          ENT1  0,0                ; move pointer
          ; also decrement LEVEL (optional book-keeping)
          LEAVEX JMP  *            ; same “stop tracing”

The only global state left is the code itself and the tiny depth/stack base variables; all per-step scratch lives in that level’s frame.

With the routine re-entrant, executing the two-instruction SELF prologue starts a fresh instance that uses its own frame, while the outer instance continues stepping and printing. When the inner instance reaches its own SELF, it starts yet another one, and so on—until your FRAMES area runs out, which is exactly the “until memory capacity is exceeded” stopping condition the exercise asks for.

  1. Where to “print one step”

Knuth’s core doesn’t print; he says you must add that (e.g., Ex. 2 writes a 10-word snapshot per step to tape 0). The natural place to emit output for the current step is after you’ve decoded the instruction and before executing it (i.e., after lines 10–16 in the listing), or immediately after execution (line 35). In a short solution, printing just the location and the instruction word is fine:

  • Just before INST NOP * (line 34), do:
          LDA  FR_PREG,1(0:2)
          STA  BUF                     ; word 1: location
          LDA  FR_INST,1
          STA  BUF+1                   ; word 2: instr word
          OUT  BUF(18)                 ; print one short line on device 18

Replace BUF with a tiny two-word buffer you’ve put near the routine. (If you already solved Ex. 2 and wrote the richer 10-word record to tape unit 0, call that writer here instead.)

  1. Why the speeds keep dropping
  • Level 0 (outermost) prints every step the program under trace performs.
  • When the under-trace program is itself a trace, that inner trace prints too—so each inner step now costs “execute the step” + “inner print” + “outer print”.
  • The inner trace’s own under-trace is another trace, etc. Each level adds another layer of output and control logic around a single MIX instruction; hence the geometric slowdown the problem statement describes.
  1. Driver to start the cascade

A tiny starter sets rJ to SELF of your trace and rX to its LEAVEX, then jumps to ENTER. (In MIX, you can seed rJ by entering via JSJ SELF and arranging that SELF immediately “calls” ENTER as in the prologue above, so the first inner instance starts correctly; your outermost instance is started by your hand—e.g., from a small harness.)

  1. Safety/notes
  • Knuth lists three simple restrictions on the traced program (don’t store into the trace’s cells; don’t use the output device the trace uses; expect slowdown). Using frames fixes the first; just be sure your traced program doesn’t OUT to the same unit you’re using.
  • His jump-handling already special-cases JSJ, which is crucial for the self-bootstrapping prologue to work.

7 [25]

  1. What the exercise asks

Design a jump trace that emits far less output than a full trace. Instead of dumping registers every step, it should output a sequence of pairs \((x₁, y₁), (x₂, y₂), \dots\) meaning “the program jumped from location \(x_i\) to \(y_i\), then ran straight-line code from \(y_i\) through \(x_{i+1}\), then jumped from \(x_{i+1}\) to \(y_{i+1}\), …”. From such pairs, a post-pass can reconstruct control flow and count how often each instruction executed.

  1. Where to hook into the given trace routine

Knuth’s trace routine already detects jumps and centralizes their handling at label JUMP. The fetch/decode loop keeps the fetch address in PREG(0:2) (the “location of the next instruction”), and when a jump occurs execution transfers to JUMP, where the target address (effective address of the jump) is computed into register A just before control returns to CYCLE (which then loads that A-value into PREG for the next iteration). This is the perfect point to record our pair \(x, y\).

Concretely (from the text routine):

  • CYCLE stores the fetch location into PREG(0:2), fetches the instruction at that address, and identifies jumps.
  • When a jump is seen, code branches to JUMP. There the routine finalizes jump semantics (special-cases JSJ, updates JREG when appropriate), then does INST1 ENTA * so A = target address; finally it JMP CYCLE to continue from the jump target.

Therefore, we can log:

  • \(x\) = the address of the jump instruction = PREG(0:2) at the time the jump is recognized,
  • \(y\) = the jump target = the value placed into A at label JUMP just before JMP CYCLE.
  1. Minimal recording logic (code sketch)

Goals: (a) very low overhead per jump, (b) no register clobbering visible to the traced program, (c) buffered output (pairs take two words), (d) output to a device not used by the program being traced (e.g., tape unit 0).

We’ll modify the block at JUMP (after it has computed the target in A), insert a small snippet to append (x, y) to a memory buffer, and flush the buffer to tape when full.

Data (reserved somewhere outside the traced program’s memory):

BUF     CON  0              * start of a 100-word block
...     (reserve 100 words total for one tape block)
END     EQU  BUF+100        * end of block
PTR     CON  BUF            * current write position (absolute address)
I1REG   CON  0              * saved I1
YHOLD   CON  0              * temp to hold y
ONE     CON  1
TWO     CON  2

At routine entry (ENTER): save I1 once (we’ll use I1 as an internal pointer) and ensure tape 0 is idle.

ST1   I1REG
JBUS  *(0)                  * wait until tape 0 is not busy
STA   AREG                  * (this is already in Knuth’s code)
STJ   JREG                  * (already)
...

(“Use minimal registers” is an ideal; for practical, simple buffering we momentarily use I1 and restore it on exit. The original trace avoided using registers other than A; here we use I1 carefully and restore it. )

At label JUMP (after lines that update JREG and just after INST1 ENTA * puts the target into A):

// Existing code:
JUMP    LDA   8B(4:5)       * as in the book
        SUB   INST(4:5)     * detect JSJ, etc.
        JAZ   *+4
        LDA   PREG(0:2)
        INCA  1
        STA   JREG(0:2)
INST1   ENTA  *             * <= A now holds jump target "y"

// INSERT begin: record (x,y) to [PTR], [PTR+1]

        STA   YHOLD         * save y (target) temporarily
        LDA   PREG(0:2)     * load x = address of jump instr
        LD1   PTR           * I1 := current dest address
        STA   0,1           * store x at M[PTR]
        LDA   YHOLD         * restore y
        STA   1,1           * store y at M[PTR+1]
        ENT1  2,1           * I1 := I1 + 2
        ST1   PTR           * PTR := PTR + 2

        * If buffer full, flush to tape 0 and reset PTR:
        CMP1  =END=         * compare I1 with END
        J1N   TRACE_CONT    * not full yet
        * flush:
        OUT   BUF,0         * start writing 100 words to tape unit 0
        JBUS  *(0)          * wait completion (conservative)
        ENT1  BUF           * I1 = BUF
        ST1   PTR           * PTR := BUF
TRACE_CONT:

// INSERT end

        JMP   CYCLE         * existing: continue at new address

Remarks:

  • We only touch I1, and we save/restore it at the beginning/end of the whole trace; the traced program never observes a changed I1 because the trace wrapper owns it during tracing. (If you want to be extra strict, you can re-save/re-restore I1 inside JUMP too, but it’s unnecessary if the trace routine always retains control.)
  • The snippet is constant-time per jump: a handful of STA/LDA plus one indexed STA pair and a simple two-word pointer bump.
  • On exit from tracing (LEAVE/LEAVEX), flush any partial block (if PTR != BUF) and then LD1 I1REG to restore I1, just like we restored A and J in the base routine. (Knuth already notes J needs restoring; exercise 1 in this section has you fix that; do both.)
  1. Buffering/output strategy and why tape
  • Why tape (unit 0)? It’s far faster and far less intrusive than printing lines; the book explicitly hints that tape is preferable to printing for trace output.
  • Asynchronous I/O: We write a full 100-word block when the buffer fills (50 jump pairs). In typical code, the number of jumps is far less than the number of executed instructions, so output volume and I/O frequency are much smaller than with a full step trace.
  • Exclusivity: The traced program must not use the same device; Knuth calls this out (don’t use the output device being used for the trace) — so dedicate tape 0 to the jump trace, and choose another device for the program (or vice versa).
  1. Notes, correctness, and limitations

  • What counts as a “jump”? The existing trace detects all jump opcodes (JMP, Jx, Jxx, JSJ) and also treats JBUS specially (since it can effect control transfer). We piggyback on that detection; our code runs only when a jump actually occurs.

  • Source and target correctness: At jump time, PREG(0:2) is the address of the jump instruction being executed (our \(x\)); after INST1 ENTA *, A holds the effective destination (our \(y\)). That aligns exactly with the exercise’s notion of pairs.

  • Post-pass reconstruction: A reader can process the pair stream (x₁,y₁), (x₂,y₂), … and:

    • mark basic blocks as \([y_i, x_{i+1}]\),
    • accumulate execution counts by adding the length of each straight-line interval,
    • reconstruct the control-flow graph edges from each \(x_i \to y_i\).

  • Self-modifying code caveat: The book notes that self-modifying code can defeat a jump-trace profiler; e.g., if an instruction changes another instruction into or out of a jump, the pair stream may no longer reflect the original static flow. Keep this limitation in mind.

  • Register hygiene: We intentionally restrict ourselves to A and a temporary use of I1 (saved/restored), in keeping with the trace routine’s philosophy of not disturbing the program state.

By inserting ~15 tight instructions at JUMP, storing \((x,y)=(\text{PREG(0:2)},\text{A})\) to a buffered block and flushing to tape, you get an efficient jump trace with tiny runtime overhead and dramatically reduced output size, while preserving the traced program’s state. This meets the exercise’s specification and leverages the exact structure of the trace routine given in §1.4.3.2.
