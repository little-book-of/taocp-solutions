1.4.2 Coroutines

1 [10]

A coroutine is symmetric: two (or more) routines repeatedly transfer control to each other, each time resuming exactly where it last left off (state must persist across transfers). This is unlike a subroutine, where control flows down and then returns once.

  1. One transfer ŌćÆ not really a coroutine. If A calls B only once (and B never resumes A mid-run), the interaction is observationally identical to an ordinary subroutine call. So the ŌĆ£smallestŌĆØ control exchange doesnŌĆÖt demonstrate anything essentially coroutine-specific. To show coroutine behavior, you need multiple alternations (AŌåö’ĖÄBŌåö’ĖÄAŌåö’ĖÄBŌĆ”), which already lengthens the example.

  2. State must be preserved between activations. Real coroutine examples must keep local state alive across yields. On a bare machine (like MIX in the text) that means explicit linkage and register saves/restoresŌĆöextra instructions that swell even toy programs. KnuthŌĆÖs own coroutine linkage shows saving/restoring registers and jump handshakes, which are inherently not ŌĆ£tiny.ŌĆØ

  3. Nontrivial data flow is needed to be meaningful. To motivate why coroutines help, you typically show a producer/consumer, scanner/parser, or ŌĆ£turn an n-pass pipeline into a one-pass processŌĆØ example. But as soon as you pass realistic data (buffers, termination conditions, etc.), the code grows beyond a few lines.   4. Toy examples are either contrived or misleading. Minimal illustration (conceptual)

  • Subroutine pattern: main ŌåÆ sub() ŌåÆ return (one descent, one return).
  • Coroutine pattern: A ŌåÆ B ŌåÆ A ŌåÆ B ŌåÆ ŌĆ” (multiple resumptions), plus explicit context preservation. Without at least two transfers, AŌåö’ĖÄB collapses to subroutine semantics and doesnŌĆÖt illuminate coroutines.

To genuinely demonstrate coroutines you need (i) more than one control transfer, (ii) preserved execution context, and (iii) meaningful data exchange. Each of these adds lines and machinery, so examples that are both short and truly illustrative are rare; once theyŌĆÖre short enough, they stop being distinctly coroutine examples and look like plain subroutines instead.

Whenever OUT executes JMP IN, rA in OUT must be exactly what it was when control returns to the next OUT instruction; likewise for IN when it executes JMP OUT. So each coroutine must (1) save its own rA at exit, and (2) restore its own rA at re-entry, without constraining the other coroutineŌĆÖs use of rA while it runs.

Knuth already shows the one-sided variant (ŌĆ£if OUT would destroy A, save/restore A around the transferŌĆØ), right above Fig. 22; we just symmetrize that pattern. We splice this linkage where the book places the standard two-coroutine linkage (lines 57ŌĆō65), replacing those four short lines with the version below.

MIX code (two-sided A preservation)

* --- Linkage data (two save slots, one per coroutine) ---
HOLDIN  CON  0        * last-saved rA for IN
HOLDOUT CON  0        * last-saved rA for OUT

* --- Initialization and linkage ---
START   ENT6 0        * as in text: init rI6 for NEXTCHAR
        ENTX 0        * as in text: init rX for NEXTCHAR
        JMP  OUT1     * start with OUT (exercise 2 discusses starting with IN)

        STA  HOLDIN   * save caller's A (leaving IN -> coming here)
        LDA  HOLDOUT  * restore OUT's own A for re-entry
OUTX    JMP  OUT1     * enter OUT proper

  • When OUT ŌåÆ IN (JMP IN inside OUT): control lands at label IN. We immediately

    1. save the callerŌĆÖs A (STA HOLDOUT), then
    2. restore INŌĆÖs previously saved A (LDA HOLDIN), and jump to IN1. While IN runs, it may freely clobber rA. When IN later yields back to OUT (JMP OUT), label OUT restores OUTŌĆÖs A from HOLDOUT before resuming at OUT1. Thus rA in OUT is unchanged across the yield/return pair.

  • Symmetrically, when IN ŌåÆ OUT (JMP OUT inside IN), label OUT saves INŌĆÖs rA to HOLDIN, restores OUTŌĆÖs rA from HOLDOUT, and resumes OUT. When OUT yields back, label IN reloads HOLDIN. So rA in IN is likewise unchanged across its yield/return.

This is exactly the two-sided generalization of the one-sided example on p. 198 (ŌĆ£if OUT would destroy rA, save at IN and restore at OUTŌĆØ), but now with two save cells so we can save the callerŌĆÖs A and simultaneously restore the calleeŌĆÖs A at each transfer.

  • From inside A, jump to AB to activate B, or to AC to activate C.
  • From inside B, jump to BA (activate A) or BC (activate C).

What the original program did (we must invert it)

  • Input format (2): a stream of characters ending with a period ŌĆ£.ŌĆØ; if a digit nŌłł{0,ŌĆ”,9} appears, it means ŌĆ£repeat the next character n+1 times,ŌĆØ even if that next character is itself a digit. Nondigits simply denote themselves.
  • Output format (3): the fully expanded character stream, printed in groups of three separated by blanks, 16 groups per card; processing stops at the period.

Our task is the inverse: read the grouped stream (3), ignore blanks, stop at the first period, and re-encode it into the compact code (2) with the fewest characters (e.g., never emit the redundant 0Z, just Z).

Minimal re-encoding strategy (correctness & optimality)

LetŌĆÖs call the flat stream (with blanks removed) S, ending at the first .. We want the shortest string over the alphabet ╬Ż Ōł¬ {0,ŌĆ”,9} whose expansion (by ŌĆ£digitŌåÆrepeat next char n+1 timesŌĆØ) equals S.

Process S run by run: for each maximal block of identical characters c of length L, output the shortest code for that block, then continue.

  • If c is a nondigit (letter, punctuation, etc.):

    • We may either write the character itself once (cost 1) or use a ŌĆ£digit + cŌĆØ pair (cost 2) to say multiple copies.

    • Optimal encoding is greedy by tens:

      • Let q = ŌīŖL/10Ōīŗ and r = L mod 10.

      • Output q pairs 9c (each pair contributes 10 copies).

      • For the remainder:

        • If r = 0: done.
        • If r = 1: just output c (better than 0c).
        • If 2 Ōēż r Ōēż 10: output (rŌłÆ1)c.

  • If c is a digit (one of 0..9 appearing as data), we cannot write it bare (because bare digits in (2) are operators). We must always use a pair:

    • Again q = ŌīŖL/10Ōīŗ, r = L mod 10.
    • Output q times 9c; if r>0, output (rŌłÆ1)c.

Why this greedy scheme is optimal:

  • Each ŌĆ£digit+charŌĆØ token has fixed cost 2 and yields between 1 and 10 copies (specifically n+1 copies). This is a canonical coin system: ŌĆ£coinsŌĆØ of values 1..10 with costs 2 each, except a free single-copy option of cost 1 for nondigits. This guarantees ŌĆ£the zero before the Z ŌĆ” would not really be producedŌĆØ (we would output just Z) since 0Z (cost 2) is strictly worse than Z (cost 1).

         JMP     QLOOP

         JE      ERRET         * r==0
         J1Z     SINGLE        * r==1 -> single CUR

         JMP     PUTCH
         JMP     ERRET

         STX     R

         JMP     PUTCH
         LDA     CCHAR
         JMP     PUTCH
         DEC     Q,1
         JMP     QDLOOP

         JE      ERRET
         DEC     R,1
         ADD     29            * 0..8 -> 0..8 digits (codes 30..38); r=1 -> '0'
         JMP     PUTCH
         LDA     CCHAR
         JMP     PUTCH
ERRET    JMP     *
