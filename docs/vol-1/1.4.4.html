1.4.4 Input and Output

1 [05]

Context you need

On MIX, IN initiates an input transfer and returns immediately; the program may continue executing, but it must not rely on the input buffer’s contents until the device is no longer busy (e.g., until JBUS shows “ready”). Sequence (3) relies on that rule: it waits with JBUS, then copies from the just-filled buffer, and finally starts the next input into that same buffer.

  1. Putting MOVEs before JBUS

Incorrect. If you copy from locations 2000–2099 before JBUS has confirmed completion, you may be moving words that haven’t arrived yet (or are only partially written). The model explicitly warns that input completes only after a subsequent I/O op or a successful “not busy” test; otherwise the buffer contents are not guaranteed. So the copied block could be incomplete/corrupted.

Verdict: No. The input may not yet be complete.

  1. Putting MOVEs after IN

Also incorrect (race hazard). If you start the next IN 2000(5) first and only then perform MOVE 2000(50); MOVE 2050(50), you are copying from a buffer that the device is simultaneously overwriting with the next block. If the input device outpaces the CPU even slightly, some words you copy will belong to the next block rather than the previous one, yielding a mixed/corrupted result.

Verdict: No. The device might outrun the MOVE, so the copy can be contaminated by the new transfer.

Final answers

(a) No. You might move data before the current input has finished. (b) No. Starting IN first risks the device overwriting the buffer while you copy, so the MOVE can pick up the wrong data.

2 [10]

Here’s a buffered-output scheme that mirrors Knuth’s buffered-input setup (his items (2) and (3)) but for writing tape unit 6. We’ll stage each 100-word block for output in an auxiliary buffer at addresses 2000–2099, while your program prepares the next 100-word block at 1000–1099. This overlaps CPU work with device I/O, just as his input pipeline does. The exercise statement appears at the end of §1.4.4; Knuth’s earlier unbuffered pair for output is OUT 1000(6); JBUS *(6) and his input buffering sequences (2)–(3) are the model we’re copying.

Setup and notation (MIX reminders)

  • A tape OUT A(6) starts writing 100 words beginning at A to unit 6.
  • JBUS *(6) waits until unit 6 is no longer busy.
  • MOVE A(50) copies 50 words starting at A into the destination pointed to by rI1, then advances rI1 by 50 (so two MOVE’s copy 100 words). Knuth’s input example sets rI1 to the destination just before the two MOVE’s.

Buffered output: two phases

  1. Prime the pipeline (first block)

Before the device can “trail” your computation, you must place the first block into the staging buffer and start the first write:

ENT1 2000          ; rI1 ← 2000 (destination = buffer)
MOVE 1000(50)      ; (1000–1049) → (2000–2049)
MOVE 1050(50)      ; (1050–1099) → (2050–2099)
OUT  2000(6)       ; start writing the staged block to tape unit 6

Explanation: you’ve copied the current block (prepared at 1000–1099) into 2000–2099 and kicked off the OUT. While the device is busy writing that buffer, your program is now free to compute/fill the next 100-word block at 1000–1099. (This mirrors Knuth’s “read-ahead” input, but here it’s “write-behind”: the device is finishing the previous buffer while the CPU prepares the next one.)

  1. Steady-state cycle (each subsequent block)

Whenever the next 100-word block at 1000–1099 is ready to be sent:

ENT1 2000          ; rI1 ← 2000 (dest = buffer), harmless if repeated
JBUS *(6)          ; wait until the prior OUT from 2000–2099 has finished
MOVE 1000(50)      ; (1000–1049) → (2000–2049)
MOVE 1050(50)      ; (1050–1099) → (2050–2099)
OUT  2000(6)       ; start writing this staged block

Why this order? You must not overwrite 2000–2099 while the device is still taking data from it; therefore the JBUS precedes the MOVE instructions, just as in Knuth’s input sequence (3) the JBUS precedes the MOVE to ensure the fresh input has arrived before moving it out of the staging area. This is the exact analog of his (2)–(3), but with roles reversed.

Finishing up

After you’ve initiated the final OUT 2000(6), wait for the device to complete before you stop or reuse the buffer:

JBUS *(6)          ; ensure the last block has been fully written

(Stopping early risks data loss; the same caveat applies to buffered input and is discussed around Knuth’s examples.)

Why it works

  • Safety. The JBUS *(6) before each pair of MOVEs prevents clobbering the very buffer the device is still using, exactly like waiting for input completion before moving data out of the input buffer in (3).
  • Overlap. While the device streams the previous block from 2000–2099, your program is free to prepare the next block at 1000–1099, so computation and I/O overlap. This is the write-side counterpart of Knuth’s “buffering” idea that overlaps I/O with computation.

3 [22]

Design notes

  • Two 100-word buffers live in memory, each followed by a single “link” word that holds the address of the other buffer. This is exactly analogous to the input routine in the text (which links its two buffers in lines 07, 11, 14 of (4)).
  • We keep the base address of the current output buffer in CUR.
  • Index register I5 holds the current position in the buffer as a negative count: it starts at −100 and is incremented each word; when it hits 0 the buffer is full. This lets us test fullness with a single J5N (jump if negative).
  • When a buffer fills, we self-modify the address field of a single OUT instruction to the “old” buffer’s base, execute it, then swap CUR to the other buffer and reset I5 to −100. (This is the same self-modifying trick used for subroutine linkage and in many MIX examples; see the standard STJ EXIT … EXIT JMP * pattern. )
  • No JBUS is needed during steady-state operation (same reason as for input (4)): we start the next I/O before we touch the previous buffer again.
  • At program end we pad with zeros if needed, OUT the final partial block, and then do a single JBUS *(V) to wait for the last I/O to complete.
* ====== Constants / device selection ======
V     EQU 1              * Tape unit to use (pick any valid unit id)
ZERO  EQU 0

* ====== Buffer areas (100 words each) + one link word ======
OUTBUF1 ORIG *+100       * 100 data words
LINK1   CON OUTBUF2      * address of the other buffer

OUTBUF2 ORIG *+100
LINK2   CON OUTBUF1

Why this works

  • “Buffer-swapping … analogous to (4)”: Two buffers linked to each other; when one fills we OUT it and immediately continue filling the other, with no busy-wait in the steady state.

  • “Index register 5 … current buffer position”: rI5 is the running position (negative count). We store at CUR+100,5 and detect a full buffer with a single J5N/rI5==0 test.

  • “Write 100 words onto tape unit V”: The OUT instruction is self-modified to the old buffer base so we write exactly that 100-word block on device V.

  • “Layout of buffer areas”: As shown—each 100-word area followed by a 1-word link holding the other buffer’s base (same linking idea as the text’s WORDIN).

  • “Beginning and end … first and last blocks properly written”:

    • Beginning: set CUR to your first buffer and rI5 := -100. That’s all.
    • End: call FLUSHOUT. It pads with zeros to the next 100-word boundary (if needed), OUTs the final block, then does a single JBUS *(V) to wait for completion before halting. (Padding is explicitly required.)

4 [M20]

Consider processing \(B\) blocks. For block \(k\) let

  • \(c_k\) = CPU time spent computing on that block’s data,
  • \(i_k\) = device time for the I/O of that block (read or write).

Define totals

\[ C=\sum_{k=1}^B c_k,\qquad I=\sum_{k=1}^B i_k. \]

With unbuffered I/O the CPU waits during each I/O, so the elapsed time is

\[ T_{\text{unbuf}}=C+I. \]

With buffered I/O (double buffering, for example), computing on one buffer can overlap I/O on the other. There is only one CPU and one device, so at any instant at most one CPU task and at most one device task can be active.

Fundamental lower bound (why speedup ≤ 2)

In any schedule (buffered or not):

  • You cannot finish earlier than \(C\), because the CPU must perform \(C\) units of work.
  • You cannot finish earlier than \(I\), because the device must perform \(I\) units of I/O.

Hence every buffered schedule satisfies

\[ T_{\text{buf}} \;\ge\; \max\{C,I\}. \]

Comparing to the unbuffered time \(C+I\),

\[ \frac{T_{\text{unbuf}}}{T_{\text{buf}}} \;\le\; \frac{C+I}{\max\{C,I\}} \;\le\; 2, \]

with equality only when \(C=I\) and overlap is essentially perfect. Therefore no single-device buffering scheme can yield more than a 2× speedup relative to unbuffered execution.

Constructive schedule (why ~2× is attainable)

Assume steady per-block times \(c_k=c\) and \(i_k=i\) and use two buffers:

  • Start I/O for block 1 (takes \(i\)).
  • While that I/O runs, compute block 0’s results or prepare block 2, etc. After the pipeline fills, each additional block takes \(\max\{c,i\}\) time because CPU and device run in parallel.
  • The exact makespan is

\[ T_{\text{buf}} = i + (B-1)\max\{c,i\} + c, \]

(first and last blocks incur the pipeline fill/drain). For large \(B\),

\[ T_{\text{buf}} \approx B\max\{c,i\}. \]

Thus the asymptotic speedup is

\[ \frac{T_{\text{unbuf}}}{T_{\text{buf}}} \approx \frac{B(c+i)}{B\max\{c,i\}} = \frac{c+i}{\max\{c,i\}} \le 2. \]

Best case. If \(c=i\) (balanced compute and I/O), then

\[ T_{\text{unbuf}}=2Bc,\qquad T_{\text{buf}}\approx Bc \quad(\text{plus small }O(c)\text{ edge effects}), \]

so the speedup approaches \(2\) as \(B\) grows. This is the “favorable circumstance” where buffering nearly halves total time.

Other cases. If \(c\ll i\) (I/O-bound) or \(i\ll c\) (compute-bound), then \((c+i)/\max\{c,i\}\) is close to \(1\); buffering still helps by overlapping the smaller resource with the larger, but the improvement is strictly \(<2\) and can be modest.

5 [M21]

Let

  • \(C\): total time the program spends computing (CPU time) not counting any I/O wait.
  • \(D_i\): total active time needed by I/O device \(i\) (sum of the times of all blocks handled by device \(i\)), for \(i=1,\dots,n\).

Assumptions are exactly those of §1.4.4: Each device works independently once an I/O is initiated; with buffering, the CPU can run while a device is busy; multiple devices may be busy concurrently. (This is the same setting used for MIX’s buffered I/O with concurrent devices.)

Two totals to compare:

  • Unbuffered baseline. I/O is blocking; devices don’t overlap with the CPU or each other.

    \[ T_{\text{unbuf}}=C+\sum_{i=1}^n D_i. \]

  • Buffered/overlapped. With sufficient buffering, CPU and devices can run concurrently; multiple devices can be active at once. Any schedule must last at least as long as the heaviest-used resource (CPU or any one device), hence

    \[ T_{\text{buf}} \;\ge\; \max\!\bigl(C,\,D_1,\dots,D_n\bigr). \]

(You can view this as a simple “makespan” lower bound: each of the \(n{+}1\) resources—CPU and the \(n\) devices—must process its assigned load; the overall time cannot be less than the largest load.)

General bound on speedup

Define the speedup \(S\) from buffering as

\[ S \;=\; \frac{T_{\text{unbuf}}}{T_{\text{buf}}} \;\le\; \frac{C+\sum_i D_i}{\max(C,\max_i D_i)}. \tag{★} \]

Since the maximum of \(n{+}1\) nonnegative numbers is at least their average,

\[ \max(C,D_1,\dots,D_n)\;\ge\;\frac{C+\sum_i D_i}{n+1}, \]

and plugging into (★) gives the universal bound

\[ S \;\le\; n+1. \]

This generalizes Exercise 4’s “at most 2×” for one device (\(n=1\Rightarrow S\le 2\)).

Tightness (when can we reach \(n{+}1\)?)

The bound is tight in “balanced” cases where

\[ C=D_1=\cdots=D_n=T. \]

Then \(T_{\text{unbuf}}=(n+1)T\) and the lower bound for any overlapped schedule is \(T\). With well-designed multiple buffering (as in §1.4.4’s coroutine-based control) you can keep all \(n\) devices busy continuously while the CPU also computes for the full \(T\). Thus

\[ T_{\text{buf}}=T,\qquad S=\frac{(n+1)T}{T}=n+1. \]

This is the natural multi-device analogue of the single-device “at most two, achieved when compute time equals I/O time” result.

Intuition and scheduling sketch

  • Lower bound: You cannot finish earlier than the longest of \(C,D_1,\dots,D_n\).

  • Achieving near the bound: Use multiple buffers per device and the coroutine-style controller (§1.4.4) so that:

    1. Each device is kept busy end-to-end (initiate the next transfer as soon as the previous completes), and
    2. The CPU has work to do at all times (no waiting on I/O), by computing against already-filled input buffers or producing output into free buffers. This is exactly the kind of overlap Knuth’s buffer-control algorithms are designed for, and the time–action charts in the section illustrate how the overlap hides I/O behind computation (and vice versa).

What the formula tells you in general

For an arbitrary workload, the best possible (ideal) speedup is bounded by

\[ S\;\le\;\frac{C+\sum_i D_i}{\max(C,\max_i D_i)}. \]

  • If one device dominates (say \(D_k\gg C\) and \(D_k\gg D_j\) for \(j\neq k\)), then \(S\approx (C+\sum_i D_i)/D_k\), which is much smaller than \(n{+}1\).
  • If CPU dominates (\(C\gg D_i\)), speedup is near 1 (little to hide).
  • If loads are well balanced (\(C\approx D_i\)), you approach the ideal \(n{+}1\).

This fully generalizes the single-device “≤2× speedup” to \(n\) devices.

You have two symmetric choices (pick either):

IN   INBUF1(U)
ENT6 INBUF2+99

or

IN   INBUF2(U)
ENT6 INBUF1+99

Optionally precede with IOC 0(U) if you might need to rewind/ready the unit.

Why this is the right start (step-by-step)

Recall the subroutine (4) (WORDIN) uses rI6 as a pointer and begins with:

  1. INC6 1
  2. LDA 0,6 and compare to SENTINEL If sentinel → refill this buffer: IN -100,6(U), then switch buffers: LD6 1,6 (loads rI6 from the “other-buffer” link), and loop to fetch the word.

Important details from that listing:

  • Each buffer reserves 100 data words; at offset +100 sits a sentinel constant; at +101 sits a link to the other buffer.
  • rI6 “addresses the last word of input” when you enter WORDIN; the subroutine immediately does INC6 1, then fetches.

Now follow the first call if you used the first init (IN INBUF1(U); ENT6 INBUF2+99):

  1. Before the first call:

    • Buffer 1 has just been filled by IN INBUF1(U) (block #1).
    • rI6 = INBUF2+99 (the last data word slot of Buffer 2).

  2. On the first WORDIN call:

    • INC6 1 ⇒ rI6 becomes INBUF2+100 (the sentinel slot of Buffer 2).
    • LDA 0,6 ⇒ you read the sentinel, so WORDIN concludes “this buffer is exhausted.”
    • IN -100,6(U) ⇒ reads the next block (#2) into Buffer 2.
    • LD6 1,6 ⇒ rI6 ← contents at INBUF2+101, which is the address of the other buffer (i.e., INBUF1).
    • Loop back to fetch the word: LDA 0,6 now loads the first actual data word of Buffer 1 — exactly what you want for the very first word delivered to the program — while the tape is already busy reading the subsequent block into Buffer 2.

Net effect: the very first call returns word 0 from Buffer 1, and you’re already overlapped with I/O that fills Buffer 2. From that point on, WORDIN alternates smoothly between the buffers, always keeping the device busy when possible (the “pipeline is primed”). This is exactly the behavior intended by (4).

What you must have in the data area

Your buffers must be laid out exactly as in (4):

  • INBUF1 and INBUF2 each reserve 100 data words.
  • At INBUF1+100 and INBUF2+100 place SENTINEL.
  • At INBUF1+101 put the address of INBUF2, and at INBUF2+101 put the address of INBUF1 (the links).

7 [22]

Instead of testing for a sentinel to detect “end of buffer”, keep a running count. A neat MIX trick is to keep rI6 negative and let it run from −100, −99, …, −1 inside each block:

  • Addressing: load the next word from BASE+100+rI6. With rI6 negative, this indexes positions 0..99 of the current buffer.

  • After each fetch, increment rI6. If it’s still negative, we’re still in the block; if it becomes 0, we’ve just consumed the last word, so we:

    1. start an IN on the buffer we just finished (to read the next block into it),
    2. reset rI6 to −100, and
    3. switch to the other buffer.

Using a tiny coroutine hand-off lets the subroutine “enter” at the correct buffer each call without extra state variables, which is exactly the style Knuth points to as one good way to solve this exercise.

MIX program (sentinel-free WORDIN)

* Sentinel-free WORDIN: double-buffered, coroutine style.
* U = the input device number (e.g., tape 5). rI6 is our negative index counter.

INBUF1     ORIG *+100           * 100-word buffer #1
INBUF2     ORIG *+100           * 100-word buffer #2

WORDIN     STJ  NEXT            * Patch NEXT with caller's return address.
NEXT       JMP  USE2            * First entry jumps into buffer #2’s path (can choose either).

* --- Use buffer #2 until it’s exhausted, then refill #2 and switch to #1 ---
USE2       LDA  INBUF2+100,6    * A ← INBUF2[100 + rI6]  (rI6 is negative ⇒ 0..99)
           JMP  RET             * Return this word to caller.
           INC6 1               * Advance position in the current block.
           J6N  USE2            * Still negative? More words in this block → stay.
           IN   INBUF2(U)       * rI6 just hit 0 ⇒ start reading next block into buffer #2.
           ENN6 100             * rI6 ← −100  (start of next block)
           JMP  USE1            * Switch to buffer #1 for subsequent words.

* --- Use buffer #1 until it’s exhausted, then refill #1 and switch to #2 ---
USE1       LDA  INBUF1+100,6
           JMP  RET
           INC6 1
           J6N  USE1
           IN   INBUF1(U)
           ENN6 100
           JMP  USE2

* --- Return site. STJ above patched NEXT to point back to the proper entry on the next call. ---
RET        STJ  NEXT            * Store rJ into the address field of NEXT (classic MIX return trick).
           JMP  *               * Jump to that saved address (i.e., return to caller).

How to bootstrap (same spirit as Ex. 6)

Before the first call to WORDIN, do the usual startup so the first fetch is correct and I/O is overlapped:

  • Initialize the counter: ENN6 100 (so rI6 = −100).
  • Prefetch one buffer (either is fine), e.g.: IN INBUF2(U).
  • Leave NEXT as shown (JMP USE2) so the first call uses buffer #2’s path. (You could swap the roles of 1/2 if you prefer.) These are exactly the kind of “get it started right” steps noted next to program (4).

Why it works (and why it’s fast)

  • Correctness. Each call returns exactly one word (from the current buffer at offset 100+rI6). When rI6 reaches 0, we (i) initiate an input on the buffer we just finished, (ii) reset rI6 to −100, and (iii) continue from the other buffer. That reproduces the overlapped “read-ahead” behavior of (4), but without testing a sentinel; the sign of rI6 is our end-of-buffer test. The coroutine hand-off (STJ NEXT + JMP USE1/USE2) selects the correct entry for the next call, so the caller remains oblivious.
  • Cost. As in the sentinel version, each word costs one indexed load plus a couple of very cheap instructions, and at block boundaries we pay for one IN and one ENN6. With read-ahead active, the per-block time is max(C, T) (C = compute per block, T = device time), just like the buffered methods in the text.

Notes

  • The routine assumes full 100-word blocks from the device, as in (4). If your data format has a separate notion of logical end-of-input (e.g., a trailing card with a special mark), handle that in the surrounding control coroutine/program exactly as in the section’s examples.
  • If you prefer not to use coroutines, you can keep a “current-buffer base” pointer in memory and branch between two short “use buffer” paths with an explicit test on rI6; the coroutine version above simply removes that extra pointer by letting the entry point itself encode which buffer is current. (Knuth’s official answer illustrates the coroutine approach.)

What the colors mean for output

  • Green (G) = a free buffer that can be ASSIGNed and filled with the next line image.
  • Yellow (Y) = the current buffer the program is building between ASSIGN and RELEASE.
  • Red (R) = a buffer that has been RELEASEd and now contains a complete line image, ready for the printer (or actively being printed if it’s the one pointed to by NEXTR).

The pointers are as in the text: NEXTG → “next green” (where the next ASSIGN will hand you a free buffer), NEXTR → “next red” (what the output device will print next), CURRENT → the yellow (assigned) buffer, if any.

Interpreting the figure sequence for output

State shown in Fig. 23 (before we start the three snapshots in Fig. 24)

At that instant there are several green buffers (free space is available) and two red buffers; the red buffer indicated by NEXTR is currently being printed by the line printer. At the same time, the program is computing between RELEASE and ASSIGN; i.e., it has finished one line (so that buffer turned red) and hasn’t yet requested a fresh buffer for the next line.

  1. After ASSIGN

The program calls ASSIGN. The buffer pointed to by NEXTG turns yellow and becomes CURRENT; NEXTG advances clockwise to the next green. The program now begins to compose the next line image into this yellow buffer while the printer continues with its current red buffer.

  1. After output completion (I/O complete)

When the printer finishes with the red buffer it was printing, that buffer turns green (it’s now free). NEXTR advances clockwise to the next red buffer (if any). If another red buffer exists, the printer immediately starts on it; otherwise the device becomes idle and will resume only when a red buffer appears later.

  1. After RELEASE

When the program RELEASEs the yellow buffer (line image completed), that buffer turns red (ready to print). If the device had been idle (no red buffers), it can now begin printing this newly red buffer (NEXTR will pick it up next); otherwise it joins the queue of red buffers that the printer will reach when NEXTR cycles around.

Model of operation (quick recap)

  • Buffers cycle through colors: green (free) → yellow (current/being filled) → red (ready for output) → green.

  • ASSIGN picks the next green buffer (if none are green, ASSIGN must wait).

  • RELEASE turns the current yellow buffer red and increments the count n of red buffers.

  • The CONTROL coroutine:

    • If the device is idle and n>0, it initiates OUT on the buffer pointed to by NEXTR (taking 7500u), then advances NEXTR and decrements n when output completes.

  • All “compute for T” durations are CPU time only; if ASSIGN has to wait for a green buffer, the CPU is idle during that wait (this is explicitly part of the exercise).

Deriving the schedule (walkthrough)

Let buffers be BUF1 → BUF2 → BUF3 in the ring. Initially all are green; NEXTR = NEXTG = BUF1; device idle.

t = 0: ASSIGN(BUF1) → BUF1 turns yellow. Compute 1000u.

t = 1000: RELEASE → BUF1 becomes red (n=1). Device is idle and n>0, so OUT BUF1 starts (finishes at 8500). Compute next 1000u.

t = 2000: ASSIGN(BUF2) (green) → yellow. Compute 1000u.

t = 3000: RELEASE → BUF2 red (n=2). Device busy (till 8500). Compute 1000u.

Minimal algorithmic change (Algorithm B → B′)

Add a single global flag EOF (initially 0). Then:

  • B2′ (gate before initiating I/O): If EOF = 1 or n = 0, go to B1 (compute). Otherwise proceed to B3.
  • B4′ (just after I/O completes, before advancing NEXTR): Inspect the 80th column of the buffer that was just filled (i.e., at address NEXTR + 15, byte 5). If it is '.', set EOF ← 1. Then continue with B5/B6 (advance NEXTR; n ← n − 1).

What changes conceptually

In the textbook, the CONTROL coroutine (Algorithm B / Program B) repeatedly:

  • waits until there is at least one red buffer (n > 0),
  • starts an IN on the buffer at NEXTR,
  • yields back to computation,
  • when the I/O finishes, advances NEXTR and decrements n.

For card input, each card image occupies 16 MIX words (5 bytes/word = 80 columns); column 80 is byte 5 of word 16, i.e., byte (5:5) of address NEXTR+15.

We’ll add a single sticky flag EOF (=0 initially). CONTROL will:

  1. After each input completes (just before advancing NEXTR), inspect the last byte of the just-read card ((NEXTR+15)(5:5)).
  2. If it equals the MIX character code for ‘.’, set EOF ← 1.
  3. At the top of the loop, treat EOF exactly like “no red buffers”: if EOF=1, do not start more IN operations ever again.

This leaves already-anticipated (green) cards, including the sentinel card itself, available for the COMPUTE coroutine to consume; it simply prevents any further anticipation past the end. (Knuth explicitly leaves such end-handling details to the reader; this is precisely the point of Ex. 12.)

Recap of the buffering model

  • Buffers are arranged in a circle; each buffer is colored green, yellow, or red. • Green = ready to be ASSIGNed (for input: already anticipated; for output: free). • Yellow = currently ASSIGNed (the program may read/write it). • Red = RELEASEd (for input: free area; for output: filled, awaiting device).
  • Pointers: NEXTG points to the next green buffer; NEXTR to the next red buffer; CURRENT names “the” yellow buffer when there is one. Two processes run as coroutines: Algorithm A (ASSIGN) makes CURRENT ← NEXTG and advances NEXTG. Algorithm R (RELEASE) increments the count n of red buffers. Algorithm B (Buffer control) starts I/O on NEXTR when n>0, then advances NEXTR and decrements n when I/O completes.

What if we do ASSIGN, ASSIGN, then RELEASE, RELEASE?

Think in terms of counts. At any instant:

#green + #yellow + #red = N ,

and (by definition of R1) n = #red. Every ASSIGN consumes a green and creates a yellow; every RELEASE turns a yellow into a red (and raises n by 1). If we perform two ASSIGNs in a row while at least two greens exist, we simply end up with two yellow buffers (two outstanding assignments). Later RELEASEs convert those yellows to reds one by one; Algorithm B then services those reds in FIFO pointer order. The control logic keeps working because B only depends on n (reds) and the NEXTR ring, not on how many yellows exist. This can actually be useful when the computation wants to look at two buffers at once (e.g., a boundary-crossing comparison, merging adjacent blocks, etc.), at the cost of tying up an extra buffer that is temporarily unavailable for anticipation/output.

Important edge case and the safe bound

  • N = 1 (one buffer). A burst ASSIGN, ASSIGN has no second distinct green to consume. The written Algorithm A’s gate (“wait for n < N”) isn’t sufficient to guarantee a green exists when we allow multiple simultaneous ASSIGNs; with one buffer you can wind up re-assigning the same area or, worse, overlapping program use with I/O. In short, the scheme breaks down for N=1.
  • General bound. Let y = (#ASSIGNs − #RELEASEs) = “#yellow at the moment.” Necessarily
0 ≤ y ≤ N .

(If y exceeded N, there would be no green left to assign; if y were negative, we’d be releasing more than we assigned.) Practically, keep the excess of ASSIGNs over RELEASEs nonnegative and ≤ N.

  • With N ≥ 2, allowing ASSIGN bursts just produces two (or more) simultaneous yellow buffers; the control coroutine continues to function, and this can be handy when the computation needs concurrent access to several buffers. It does, however, tie up buffer space, leaving fewer greens available for anticipation.
  • With N = 1, the idea is unsafe/inapplicable; the multiple-buffer algorithm as given relies on there being enough buffers that an extra ASSIGN can legitimately consume another green.

Here’s a fast, 3-buffer MIX solution for (“copy 100 blocks from tape unit 0 to tape unit 1, using just three buffers”). The trick is to pipeline I/O so the input and output units stay busy almost all the time, with only a one-time “prime” and a one-time “drain.” This is exactly the “same-circle input+output” idea behind Fig. 26 and the 1.4.4 exercises on pp. 225–228.

MIX program (3-buffer tape-to-tape copy)

* Tape-to-tape copy with triple buffering (fast)
INUNIT  EQU 0              * input tape unit
OUTUNIT EQU 1              * output tape unit

BUF1    ORIG *+100         * three 100-word buffers
BUF2    ORIG *+100
BUF3    ORIG *+100

START   IN   BUF1(INUNIT)  * prime the pipeline: read block #0 -> BUF1
        ENT1 99            * 99 more blocks to output after the prime

LOOP    IN   BUF2(INUNIT)  * queue read of next block -> BUF2
        OUT  BUF1(OUTUNIT) * write the block that was in BUF1
        IN   BUF3(INUNIT)  * queue read -> BUF3
        OUT  BUF2(OUTUNIT) * write the block now in BUF2
        IN   BUF1(INUNIT)  * queue read -> recycle BUF1
        OUT  BUF3(OUTUNIT) * write the block now in BUF3
        DEC1 3
        J1P  LOOP

        JBUS *(INUNIT)     * wait for final input to finish
        OUT  BUF1(OUTUNIT) * write the very last block
        HLT
        END  START

Why this is correct and fast

  1. Pipeline structure.
  • We “prime” with one input (BUF1).
  • Each loop body initiates three inputs and three outputs in a fixed rotation (…B2 in, B1 out; B3 in, B2 out; B1 in, B3 out…). This schedules operations so that every OUT uses a buffer whose IN started two I/O steps earlier, i.e., long enough before to have finished by the time the device consumes it (CPU time is negligible compared to device time).
  • We decrement by 3 because each loop iteration produces 3 completed outputs. Starting from 99, we do 33 iterations (99/3), then drain the final pending input and write the final block.

  1. No buffer hazards. Each buffer is reused only after its previous read and subsequent write have had time to complete; the triple spacing prevents “read-while-write” on the same memory. We never need to poll during the pipeline; a single JBUS *(INUNIT) at the end synchronizes the very last read before the last write.

  2. Minimal idle time ⇒ “as fast as possible.” After the one-time prime, both devices are kept busy continuously (as in the multiple-buffer timelines and exercises in §1.4.4), so throughput is device-limited. Only the leading and trailing edges incur unavoidable bubbles. This matches the “same circle” buffering method discussed around Fig. 26/27 and the surrounding exercises.

  3. Why ENT1 99 and DEC1 3? We must output 100 blocks total. One block is primed into BUF1 before the loop, so 99 outputs remain; each loop does 3 outputs, hence 33 passes. After the loop, the pending final read completes (JBUS) and we perform the 100th output.

Here’s a clean formulation of the “green–yellow–red–purple” multiple-buffer algorithm Knuth asks for, written in the same style as Algorithms A/R/B in §1.4.4, but now with three coroutines: one for input, one for output, and one for the computation. I stick to his color semantics and ring-of-buffers pointers.

Model and invariants

  • We have a single circle of N equal buffers.

  • Colors (states) and their meaning (same naming as the text and Fig. 26): Green = anticipated input is ready to be ASSIGNed to the program; Yellow = currently assigned to the program (between ASSIGN and RELEASE); Red = RELEASEd by the program; block is ready to be output; Purple = output finished; buffer is free for new input. (Purple is turned to green by input; yellow is turned to red by release; red is turned to purple by output.)

  • Pointers on the ring:

    • NEXTG → first G (next buffer the program may assign)
    • NEXTR → first R (next buffer the output device will transmit)
    • NEXTP → first P (next buffer the input device will fill)
    • CURRENT → Y (present only while the program holds a buffer)

  • Counters we maintain:

    • g = # of green buffers
    • r = # of red buffers
    • p = # of purple buffers
    • y ∈ {0,1} (0 if no current assignment; 1 if a buffer is yellow)
    • Invariant: g + r + p + y = N.

  • The ring is always segmented in the cyclic order P* G* (Y?) R* with the pointers at the segment heads; this ensures none of NEXTG, NEXTR, NEXTP “passes” another. (This is the three-pointer analogue of the “no passing” story for two pointers in Fig. 24.)

Initial/terminal conditions

  • Start: all buffers purple (the natural “free for input” state). So p = N, g = r = y = 0, and set NEXTP = NEXTG = NEXTR to some buffer. Input begins immediately and produces the first G’s for the program.
  • End: after the program’s last RELEASE, we must allow output to drain until r = 0 (all finished) before declaring completion (the analogue of the drain step discussed for two-color algorithms).

Program-side primitives (as in §1.4.4)

We keep ASSIGN and RELEASE as program-visible actions; their algorithms mirror Knuth’s A and R, but they operate on g, r, p, y and NEXTG:

Algorithm A′ (ASSIGN, inside the COMPUTE coroutine)

A1′. Wait for g > 0. (Stall the program until some green buffer exists.) A2′. CURRENT ← NEXTG. (The chosen G becomes the current buffer; color changes G→Y.) A3′. Advance NEXTG clockwise to the next G. A4′. Update counts. g ← g − 1, y ← 1. ■

Algorithm R′ (RELEASE, inside the COMPUTE coroutine)

R1′. Finish the buffer. (Color Y→R.) R2′. Update counts. y ← 0, r ← r + 1. ■

These are exactly the same “shape” as Knuth’s A/R; only the bookkeeping differs (his single counter n was the number of red buffers in the two-party case).

Device controllers as coroutines

The two devices now have independent controllers (each analogous to Knuth’s Algorithm B, but specialized to one color transition). Each controller runs “in parallel” with COMPUTE via coroutine linkage (COMPUTE periodically yields with JRED … as in the text).

Coroutine IN (input controller; turns P→G)

I1. Compute (yield to COMPUTE) for a short while; re-enter here later at a device-ready instant. I2. If p = 0, go to I1. (No purple → nothing to input.) I3. Initiate input from the input device into the buffer at NEXTP. I4. Compute (yield to COMPUTE) while the device is busy. I5. On input completion: advance NEXTP clockwise to the next P; update counts: p ← p − 1, g ← g + 1. Go to I2. ■

Coroutine OUT (output controller; turns R→P)

O1. Compute (yield to COMPUTE) for a short while; re-enter here later at a device-ready instant. O2. If r = 0, go to O1. (No red → nothing to output.) O3. Initiate output from the buffer at NEXTR to the output device. O4. Compute (yield to COMPUTE) while the device is busy. O5. On output completion: advance NEXTR clockwise to the next R; update counts: r ← r − 1, p ← p + 1. Go to O2. ■

Coroutine linkage (skeleton)

Exactly as Knuth does, insert “jump-when-ready” instructions inside the COMPUTE code so the device controllers can preempt when their unit becomes ready. One practical arrangement is:

  • COMPUTE periodically executes JRED INCTRL(U_in) and JRED OUTCTRL(U_out) (once every ~50 instructions as he suggests), letting each controller run steps I1/I4 and O1/O4 at device-ready times; each controller returns to COMPUTE after its brief action.

Why the pointers never “pass”

At every transition we convert the head element of one contiguous color segment to the head of the next segment in the cyclic order:

  • IN does P→G at NEXTP and then advances NEXTP to the next P; thus the P-segment shrinks at its head, the G-segment grows at its tail, and no pointer can jump over another.
  • ASSIGN does G→Y at NEXTG, then advances NEXTG to the next G; the single-element Y sits between G and R, preserving order.
  • RELEASE does Y→R (no pointer move).
  • OUT does R→P at NEXTR and advances NEXTR to the next R; the R-segment shrinks at its head and the P-segment grows at its tail.

Hence the circular order P* G* (Y?) R* is invariant and pointers cannot “lap” each other (the three-party generalization of the two-pointer races discussed with Fig. 24).

Liveness (no deadlock)

  • If the program needs a buffer and g = 0, then either some output is pending (r > 0, so OUT will run and eventually increase p), or some input is possible (p > 0, so IN will run and increase g).
  • If an output is needed but r = 0, eventually a RELEASE will occur (COMPUTE makes Y→R) or new input will feed the pipeline to produce future R’s.
  • If input is needed but p = 0, eventually an output will complete (R→P) or the program advances and releases something.

Provided each coroutine is scheduled fairly (as in Knuth’s set-up with periodic JRED), progress continues.

Data structures and invariants

Buffers: Each buffer b has fields (state, dev), where state ∈ {FREE, GREEN, YELLOW, RED} and dev is a device id.

Global pool/list:

  • FREE: list of free buffers.
  • For each input device d: queue G[d] of GREEN buffers (ready for compute), and counter g[d] = |G[d]|.
  • For each output device o: queue R[o] of RED buffers (ready for device), and counter r[o] = |R[o]|.
  • Device status idle(d) / idle(o) and an “in-flight” counter for I/O requests in progress.

Anti-flood control (the key to the exercise):

  • Per-input token bucket. For each input device d, maintain tok[d] with bounds 0 ≤ tok[d] ≤ cap[d] (small cap, e.g., 2). – Consume one token when scheduling an IN for d. – Produce one token when computation takes a GREEN buffer of d (i.e., on ASSIGN_IN(d)), reflecting actual consumption. This ensures anticipation never runs more than cap[d] blocks ahead of use.
  • Free-buffer reserve. Maintain reserve_free ≥ 1 (e.g., reserve_free = #output_devices, one per output). Do not start new input if |FREE| ≤ reserve_free, unless compute is currently blocked waiting for that specific input device d (the text’s exception).
  • Priority: Output drains (RED→device) always take precedence over more input anticipation.

These three rules are sufficient to keep pooled buffers from stalling due to over-prefetch.

Algorithms (pooled variants of A/R/B)

Algorithm A_pool — ASSIGN (normal state)

There are two entry points:

A_pool.IN(d): hand compute the next input block from device d. A1. If G[d] nonempty: pop head b from G[d], set state(b)←YELLOW, return b. Also tok[d] ← min(tok[d]+1, cap[d]) (credit for consumption). A2. Otherwise: mark need[d]←true, invoke B\_pool (control), and wait until G[d] becomes nonempty, then go to A1.

A_pool.OUT(o): obtain a workspace buffer destined for output device o. A3. If FREE nonempty: take b from FREE, set state(b)←YELLOW, dev(b)←o, return b. A4. Otherwise: invoke B\_pool and wait until FREE nonempty, then A3.

Algorithm R_pool — RELEASE (normal state)

R1. If b was obtained by ASSIGN_IN, computation has consumed it: set state(b)←FREE, push to FREE. R2. If b was obtained by ASSIGN_OUT(o), computation has produced it: set state(b)←RED, enqueue to R[o]. R3. Optionally call B\_pool (helps prompt draining).

Algorithm B_pool — CONTROL (runs on I/O completion or when A/R asks)

B1. Drain outputs first (priority). For each idle output device o with R[o] nonempty:  pop b from R[o], start OUT(o, b) (device becomes busy).

B2. Start essential inputs (to unblock compute). For each input device d with need[d]==true and idle(d)==true:  If FREE nonempty (even if |FREE| ≤ reserve_free), take b from FREE, set state(b)←(in-flight IN for d), tok[d]−− (if tok[d]==0, set to 0; we allow this unblocking read even if tokens are 0), start IN(d, b), clear need[d]. (Exactly the exception suggested in the text: we may use the final buffer for input only if compute is stalled for that input.)

B3. Conservative anticipation for inputs. For each idle input device d, if tok[d] > 0 and |FREE| > reserve_free:  take b from FREE, set (in-flight IN for d), tok[d]−−, start IN(d, b).

B4. On I/O completion (interrupt or poll): – If an OUT(o, b) completes: set state(b)←FREE, push FREE. – If an IN(d, b) completes: set state(b)←GREEN, enqueue to G[d]. Then re-enter B1 (outputs may now drain again).

This is the entire pooled-buffer controller. It’s short because the token bucket + reserve encode the needed policy compactly.

Why this prevents slowdown

  • No green-pool flood. Tokens ensure each input device anticipates at most cap[d] blocks beyond what compute actually consumes; you can’t fill the pool with “anticipated input” because consumption is the only way to mint new tokens.
  • Output always proceeds. Output drains have strict priority (B1) and at least reserve_free buffers are kept out of anticipatory reads, so produced blocks can turn RED and get scheduled without starvation.
  • Progress when blocked. If compute stalls waiting for d, B2 violates both the token and reserve limits only for that d, matching Knuth’s exception: you may use the last buffer to get unstuck for the specific input that blocks compute.

Tuning notes

  • Typical choices: reserve_free = #output_devices and cap[d] ∈ {1,2}. With fast outputs or many outputs, increase reserve_free. With bursty inputs, cap[d]=2 smooths latency while still preventing flood.
  • You may add a small hysteresis (high/low watermark) around reserve_free to avoid thrashing (start anticipation only when |FREE| ≥ reserve_free+H, stop when |FREE| ≤ reserve_free).

  • A_* calls may block, which triggers B_POOL and CHECK_IO until the requested list becomes nonempty.

  • B_POOL is also called:

    • right after REL_OUT? (to drain RED → device),
    • from CHECK_IO when completions happen,
    • and opportunistically anywhere you’d like to make the system responsive.
  1. Policy knobs (the “anti-flood” part)
  • Token buckets: TOK[d] minted only when compute actually consumes a GREEN buffer (in A_IN?, after successful dequeue). One token is spent when scheduling an anticipatory IN in B3. Essential reads in B2 don’t spend tokens (that’s the “exception when blocked”). Keep CAP[d] small (1–2).
  • Reserve: RESV = #outputs is a good default; increase if outputs can bunch up or are slow. The FREE_COUNT > RESV guard must hold before B3 starts more anticipation.
  • Output priority: Encoded by B1 running before B2/B3. This prevents the pool from filling with GREEN while outputs starve.

Below is a clean, interrupt-driven rewrite of Knuth’s multiple-buffer scheme (Algorithms/Programs A, R, B in §1.4.4) for MIX, using the proposed control-state “interrupt” extension described right after Exercise 18. I first restate the interrupt model very briefly, then give the modified algorithms and a compact MIX sketch, and finally explain why it works and how it maps to the original buffering logic.

Interrupt model (per the exercise)

  • MIX gains control memory locations −3999..−0001 and two internal states: normal and control. On an interrupt, registers rA, rI1..rI6, rX, and (rJ, OV, CI, “next-inst”) are saved into −0009..−0001, and MIX enters control state at an address determined by the interrupt type. There is also a “clock” at −0010 that ticks each 1000u and, on zero, interrupts to −0011. The instruction INT: in normal state, forces an interrupt to −0012; in control state, it restores registers from −0009..−0001 and returns to normal state. I/O completion interrupts vector to −(0020+unit). (All as specified in the exercise statement.)

Variables and notation

We keep Knuth’s §1.4.4 notation:

  • N = total buffers in the circle.
  • n = number of green buffers (assignable).
  • NEXTG, NEXTR, CURRENT = ring pointers as in the text.
  • New: t = 0 if the device is idle, 1 if a transfer is in progress.
  • Device unit number is U.

(Initial conditions are as in §1.4.4: for input, start with all buffers green; for output, with all buffers red; and NEXTR = NEXTG at start. See the section text around Algorithms A/R/B.)

The modified algorithms

These are exactly the changes needed to eliminate any polling/JRED and run purely interrupt-driven. The outline matches Knuth’s official answer (“Partial solution”) but with full context and linkage.

A′ — ASSIGN (normal state, called by the compute routine)

Unchanged from Algorithm A in the text:

  • A1. Wait for n < N.
  • A2. CURRENT ← NEXTG.
  • A3. Advance NEXTG (clockwise).

(Identical to §1.4.4A.)

R′ — RELEASE (normal state, inlined where the compute routine is done with the current buffer)

  • R1′. Increase n by 1.
  • R2′. If t = 0 (device idle), force an interrupt with INT so that control jumps immediately to the control routine’s step B3′ (see below).

This is exactly the little “nudge” we need so that when the CPU frees a buffer and the device happens to be idle, the control routine can immediately kick off the next I/O without delay.

B′ — BUFFER CONTROL (runs in control state; it processes interrupts)

Think of B′ as the interrupt handler framework for buffering; it is entered:

  • on a software interrupt from R2′ (vector −0012),
  • or on an I/O completion interrupt (vector −(0020+U)).

It uses t to remember whether the device is currently busy.

  • B1′. Restart the main program (execute INT) if there’s nothing useful to do right now.
  • B2′. If n = 0, set t ← 0 and go to B1′ (no buffers to service, stay idle and let the CPU compute).
  • B3′. Set t ← 1 and initiate I/O on unit U using the buffer pointed to by NEXTR (e.g., IN for input flow, OUT for output flow).
  • B4′. Restart the main program (INT). An I/O completion will later interrupt and re-enter control state.
  • B5′. On the I/O completion interrupt, advance NEXTR to the next buffer in the ring.
  • B6′. Decrease n by 1 (we’ve consumed a green buffer for input, or emptied a red buffer for output), and go to B2′.

This is the full interrupt-driven analogue of the original B; no busy-waiting or JRED appears anywhere.

Why this is correct

  • No polling: The original B used a “wait until device ready” flavor (conceptually via JBUS/JRED). Here we never poll. The control logic either restarts the main program (B1′/B4′) or is re-entered only when something interesting happens (a release when idle, or I/O completion). This directly matches the exercise’s intent.

  • Immediate kick-off when possible: If the CPU frees a buffer while the device is idle (t=0), R2′ forces a software interrupt, which vectors straight to B3′ and starts I/O at once. This eliminates latent delays that might otherwise occur before the next timer/clock tick.

  • Invariant preservation: The same invariants from §1.4.4 hold:

    • n counts available (green) buffers; is incremented only by RELEASE (R1′) and decremented only when a transfer is actually begun (B6′).
    • t tracks device occupancy; set in B3′ and implicitly cleared by the path B2′ takes when n=0 (and by the fact that once completion fires, the next cycle will either start another I/O or idle again).
    • NEXTR advances only upon I/O completion (B5′), exactly mirroring the original control logic.

  • State transitions match hardware: INT in control state restores registers and resumes normal execution (B1′, B4′), and I/O completion vectors to −(0020+U) (B5′ path). That’s precisely how the exercise defines INT and the I/O interrupts.

Mapping to Knuth’s official answer

Knuth’s answer gives the same three changes concisely:

  • “In the algorithms below, t is 0 when idle, 1 when active.”
  • A′ unchanged.
  • R′: increment n; if t=0 force an interrupt to B3′.
  • B′: (i) restart; (ii) if n=0, set t ← 0 and restart; (iii) set t ← 1 and initiate I/O; (iv) restart; (v) on I/O complete, advance NEXTR; (vi) n ← n − 1; loop.

Notes and practical tips

  • If you use the “green–yellow–red–purple” combined input/output circle (Fig. 26) elsewhere in §1.4.4, the same interrupt framework applies; you just choose IN vs. OUT in B3′ based on which half (input or output) is being serviced at that moment.
  • You do not need the clock interrupt (−0011) for correctness; it can be ignored or used for other housekeeping. The key events are R2′’s software INT and device completion interrupts.

We process \(n\ge 2\) consecutive blocks from a rotating device. Block \(k\) can begin input only when its start passes the head, namely at times

\[ t_k \equiv (k-1)L \pmod P,\qquad t_1=0, \]

and it becomes available \(T\) time units later, i.e. it may be assigned at

\[ u_k\;\ge\; t_k+T, \]

then takes \(C\) time units to compute, so it is released at

\[ v_k = u_k + C . \]

Processing is sequential so \(u_k \ge v_{k-1}\) for \(k>1\). With only \(N\) buffers we also need \(t_k \ge v_{k-N}\) for \(k>N\).

Question: what is the least \(N\) that lets us achieve the theoretical optimum finishing time

\[ v_n^{\min}=T+C+(n-1)\max(L,C)? \]

Give a general rule and then work it out for \(L=1,\;P=100,\;T=0.5,\;n=100\) with several \(C\)’s.

Lower bound and target schedule

The lower bound arises because each additional block must wait at least \(L\) time units (until its start again appears) and at least \(C\) time units (to compute). Thus the best possible cadence is by \(\max(L,C)\), with the pipeline start-up cost \(T+C\). We therefore try to hit the bound by choosing

\[ v_k=\begin{cases} T+C, & k=1,\\[2pt] v_{k-1}+C,& C\ge L\quad\text{(compute-bound)},\\[2pt] v_{k-1}+L,& C\le L\quad\text{(I/O-bound)}. \end{cases} \]

Case A — I/O-bound (\(C\le L\))

Here we can target \(u_k=t_k+T=(k-1)L+T\) (use the earliest revolution), giving

\[ v_k=T+C+(k-1)L . \]

To be feasible with \(N\) buffers we must not start input before a buffer frees:

\[ t_k\;=\;(k-1)L\;\ge\;v_{k-N}=T+C+(k-N-1)L \;\;\Longleftrightarrow\;\; (N-1)L\;\ge\;T+C. \]

Hence the minimum number of buffers is

\[ \boxed{\,N_{\min}=\left\lceil 1+\frac{T+C}{L}\right\rceil\,,\qquad C\le L.} \]

Case B — Compute-bound (\(C\ge L\))

Now we target \(u_k=v_{k-1}\) so \(v_k=T+kC\). Block \(k\) must have finished input by \(v_{k-1}-T\). Among the allowed input start times for block \(k\),

\[ \mathcal{T}_k=\{(k-1)L+mP \mid m\in\mathbb{Z}_{\ge0}\}, \]

let \(t_k\) be the latest \(\le v_{k-1}-T\). The lag we need to “absorb” with buffering is the backward distance

\[ \delta_k \stackrel{\text{def}}{=}\bigl( (v_{k-1}-T) - t_k \bigr) = \bigl((k-1)(C-L)\bigr)\bmod P \;\in [0,P). \]

(That identity uses \(v_{k-1}-T=(k-1)C\).) With \(N\) buffers the available “window” before compute touches the block is

\[ (v_{k-1}-T)-v_{k-N} = (N-1)C - T, \]

so feasibility for all \(k\) requires

\[ \delta_k \le (N-1)C - T\quad\text{for }k=2,\dots,n. \]

Let

\[ \Delta_n \;=\;\max_{1\le j\le n-1}\; \bigl(j(C-L)\bmod P\bigr), \]

then a necessary and sufficient condition to attain the optimum is

\[ (N-1)C - T \;\ge\; \Delta_n . \]

Therefore the minimum number of buffers is

\[ \boxed{\,N_{\min}=\left\lceil 1+\frac{T+\Delta_n}{C}\right\rceil\,,\qquad C\ge L,} \]

with \(\displaystyle \Delta_n=\max_{1\le j\le n-1}\!\bigl(j(C-L)-P\lfloor j(C-L)/P\rfloor\bigr)\).

Remarks.

  • If \((C-L)/P\) is rational \(=p/q\) in lowest terms and \(n-1\ge q\), then \(\Delta_n=P-P/q\). If \(n\) is smaller, compute the max over \(j=1,\dots,n-1\) explicitly.
  • If \(C=L\), then \(\Delta_n=0\) and the formula gives \(N_{\min}=\lceil1+T/C\rceil\).

Worked examples (\(L=1,\;P=100,\;T=0.5,\;n=100\))

For each \(C\) we evaluate \(\Delta_{100}=\max_{1\le j\le 99}(j(C-1)\bmod 100)\) and then \(N_{\min}\).

Case \(C\) Regime \(\Delta_{100}\) \(N_{\min}\)
(a) 0.5 \(C\le L\) — \(\left\lceil 1+\frac{0.5+0.5}{1}\right\rceil=2\)
