1.4.1 Subroutines

1 [10]

Calling sequence (two entrances)

  • Special case \(n=100\): JMP MAX100 (sets \(rI3 \leftarrow 100\) internally via ENT3 100 in (5)).
  • General case \(n\) arbitrary: ENT3 n ; JMP MAXN (caller supplies \(n\) in \(rI3\), then enters at MAXN).

Entry conditions

  • For MAX100 entrance: No register precondition for \(n\); the subroutine itself performs ENT3 100.
  • For MAXN entrance: \(rI3 = n\) with \(n \ge 1\). (These mirror the role of \(n\) in the general MAXIMUM spec given in (4), but with \(rI3\) used as the parameter in this version.)

Exit conditions (functional result and register effects)

Let \(X\) be the base address used by the code (as in the MAX programs), and let the table be \(X+1,\dots,X+n\). Then on return:

  • Result: \(rA \;=\; \max_{1 \le k \le n} \mathrm{CONTENTS}(X+k)\) and \(\mathrm{CONTENTS}(X+rI2) = rA\) (so \(rI2\) gives a position at which the maximum occurs). (Same functional contract as in (4).)
  • Loop index cleanup: \(rI3 = 0\) on exit (as in (4)).
  • Other status: \(rJ\) (the jump register) and the comparison indicator (CI) are affected by the subroutine call/return sequence (as noted in (4)).
  • No external stores implied: The specification does not imply changes to memory outside the subroutine’s own locations; in (4) Knuth notes we may conclude nothing has been stored externally when the contract doesn’t say so.

Linkage notes (how (5) returns)

    1. saves the return address with STJ EXIT at entry (for MAXN) and returns with EXIT JMP *, which jumps back to the instruction following the caller’s JMP. For the MAX100 entrance, ENT3 100 is done first, but the same EXIT return is used; ENT3 doesn’t disturb \(rJ\).

Compact spec (in the exact style of (4))

  • Calling sequence: JMP MAX100  or  ENT3 n ; JMP MAXN.

  • Entry conditions: For MAX100: none (internally sets \(rI3 \leftarrow 100\)). For MAXN: \(rI3 = n\), assume \(n \ge 1\).

  • Exit conditions: \(rA = \displaystyle \max_{1 \le k \le n} \mathrm{CONTENTS}(X+k) = \mathrm{CONTENTS}(X+rI2)\); \(rI3 = 0\); \(rJ\) and CI are also affected; no other state is changed.

2 [10]

Replace this use of JSJ (jump without altering rJ) with ordinary instructions.

Solution

Use the same two-instruction prologue that label MAXN uses—save the return address and jump into the body—then set the parameter and continue:

MAX50  STJ  EXIT    ; save caller's return address (rJ) where MAXN would save it
       ENT3 50      ; set rI3 = 50 (parameter n = 50)
       JMP  2F      ; continue at the same point as other entries

Why this works

  • At entry MAXN, the subroutine begins by STJ EXIT then JMP 2F to the body. By duplicating those two actions at MAX50, we avoid needing a JSJ.
  • ENT3 does not change the J-register (Knuth notes this explicitly), so you may place ENT3 50 before or after STJ EXIT; either way the saved return address is the caller’s, as desired.
  • After JMP 2F, execution is identical to the other entrances; the subroutine eventually returns via the saved address at EXIT.

This exactly satisfies the exercise’s request to substitute for (6) without using JSJ.

3 [M15]

Knuth’s spec (4) for MAXIMUM is (summarizing):

Calling sequence: JMP MAXIMUM. Entry: rI1 = n (assume n ≥ 1). Exit: rA = max_{1≤k≤n} X[k] = X[rI2], and rI3 = 0. It notes that “rJ and CI are also affected” but leaves them unspecified.

The code pattern is the usual “find max” loop, with a return via EXIT JMP *. (See the MAX100 variant (1) and the generalized forms in the same section.)

Precisely: effect on rJ

  • On entry (caller does JMP MAXIMUM), rJ holds the caller’s return address (the location after the call), per MIX jump semantics.
  • On exit the subroutine executes JMP * at label EXIT. In MIX, JMP * returns to the address currently in rJ and stores the address of the instruction following EXIT (i.e., EXIT+1) into rJ as part of the jump.
  • Net effect seen by the caller: control returns to “after the call”, but rJ is clobbered to an internal address (EXIT+1 from inside the callee). Thus rJ is not preserved by the subroutine. (This is why (4) warned that rJ is affected.)

Precisely: effect on the comparison indicator (CI)

Look at the inner loop for MAXIMUM/MAX100 (labels M3–M5): each iteration does a CMPA X,3 and possibly updates the running maximum, then decrements the loop index and exits when it is no longer positive. Therefore, the last instruction that sets CI before return is the final CMPA in the loop.

  • At exit, rA holds the maximum m, and the final CI equals the result of comparing m with the last element that was compared (namely X[1] in the typical 1-downward loop).
  • Consequently, CI is data-dependent: it is “=” (equal) iff X[1] equals the maximum, otherwise “>” (greater) if the maximum is strictly larger than that last compared value. In general, the subroutine does not define/sanitize CI; it simply leaves whatever CI arose from the last CMPA.

(That is exactly why (4) omitted CI by default: it isn’t a stable, caller-useful postcondition unless explicitly specified.)

If the precondition n ≥ 1 is violated (rI1 not positive)

The code initializes with j ← n; m ← X[n]; k ← n−1; and then loops while k > 0. If n ≤ 0, the loop body is skipped immediately, and the routine returns without doing a proper “maximum of 1..n”:

  • It will have loaded rA from X[n] and set rI2 ← n, then exit with rI3 = n−1 ≤ 0 (so the advertised rI3 = 0 postcondition in (4) fails).
  • If n = 0, you actually get the value at X[0]; if n < 0, you fetch from X + n (a location before X[1]). This is precisely why the spec’s precondition “assume n ≥ 1” is essential; otherwise the behavior is out of spec and effectively undefined for the intended task.

Completed specification addendum to (4)

Add the following to the “Exit conditions” of (4):

  • rJ after return: contains the address EXIT+1 (inside the subroutine), not the caller’s return address; therefore rJ is modified.
  • Comparison indicator (CI): equals the result of the final CMPA executed in the loop (i.e., comparison of the final maximum with the last element tested). Its value is not fixed by the specification and must not be relied upon by callers.
  • If rI1 ≤ 0 at entry: the routine returns immediately after loading rA ← X[n], with rI2 ← n and rI3 = n−1 ≤ 0; the semantics “max of the first n elements” does not hold in this case.

4 [21]

Generalize the MAX subroutine so it can scan an array with an arbitrary stride r>0 instead of consecutive elements. Provide two entry points:

  • SMAX — general case (caller supplies r),
  • SMAX1 — fast special case for r=1.

Assume the array base is the memory label X, the caller supplies the length n>0 in index register I3, and (for SMAX) the stride r>0 in index register I1. On return put the maximum into A and the position where it occurs into I2. (Context: the fixed-length MAX100 subroutine and Knuth’s subroutine conventions appear in §1.4.1. )

Knuth’s official answer states the exit conditions precisely, including the final value of I3; we’ll derive those below.

The original MAX100 scans X+100, X+99, …, X+1 by keeping the current offset in I3 and doing DEC3 1 each pass. To scan every r-th element, do exactly the same but replace the decrement by DEC3 0,1, which subtracts r (the contents of I1) from I3. The loop terminates when I3 ≤ 0. (This is the classic one-entry, one-exit MIX subroutine scheme with STJ EXIT/EXIT JMP * linkage. )

* MAXIMUM OVER X[n], X[n−r], X[n−2r], ...
* Calling:
*   SMAX : rI3 = n (>0), rI1 = r (>0)
*   SMAX1: rI3 = n (>0), and r = 1 by convention
* Returns:
*   rA  = max of inspected elements
*   rI2 = offset k (so that element is X + k)
*   rI3 = ((n−1) mod r) + 1 − r   (derived below)

SMAX    STJ  EXIT          * Save return address
        JMP  INIT          * General case (stride in rI1)

SMAX1   STJ  EXIT          * Fast path when r = 1
        JMP  INIT1

* ---- initialization: A ← X + I3, I2 ← I3, then step once ----

INIT    ENT2 0,3           * rI2 ← I3
        LDA  X,3           * rA  ← X + I3
        DEC3 0,1           * I3 ← I3 − r   (r = rI1)
        J3P  LOOP          * if I3 > 0, examine more
        EXIT JMP *         * single element case

INIT1   ENT2 0,3
        LDA  X,3
        DEC3 1             * I3 ← I3 − 1   (r = 1)
        J3P  LOOP1
        EXIT JMP *

* ---- main loop: compare, possibly update (A,I2), then step ----

LOOP    CMPA X,3           * compare A with X + I3
        JGE  STEP
        ENT2 0,3           * remember where the new max is
        LDA  X,3           * take the new max into A
STEP    DEC3 0,1           * I3 ← I3 − r
        J3P  LOOP
        EXIT JMP *

LOOP1   CMPA X,3
        JGE  STEP1
        ENT2 0,3
        LDA  X,3
STEP1   DEC3 1
        J3P  LOOP1
        EXIT JMP *

This mirrors the pattern of MAX100 (initialize by loading the first candidate, then loop), but changes the decrement to r when entered via SMAX. (Compare with the MAX100 listing shown in the text. )

Preconditions and postconditions

  • Entry I3 = n > 0. For SMAX, I1 = r > 0; for SMAX1, r is understood to be 1.

  • Exit

    • A = max { CONTENTS(X + n − k r) : k = 0,1,2,…, ⌊(n−1)/r⌋ }. Equivalently, the maximum over X+n, X+n−r, X+n−2r, … while the offset stays positive.
    • I2 is the offset at which that maximum was found (so the element is CONTENTS(X + I2)).
    • I3 = n − r ⌈n/r⌉ = ( (n−1) mod r ) + 1 − r = −((−n) mod r). (Knuth states this compactly in the official answer; the algebra appears below. )

  • Registers preserved As with MAX100, the code doesn’t touch X or I1 (except that SMAX1 never needs to set I1). The return address is handled by STJ EXIT/EXIT JMP * per §1.4.1 conventions.

Correctness (loop invariant)

Let k denote the current value of I3 at the top of LOOP/LOOP1.

Invariant: After each comparison step, A = max { CONTENTS(X + t) : t ∈ { n, n−r, n−2r, …, k+r } } and I2 is a value t realizing that maximum.

  • Initialization (INIT/INIT1): We set A ← CONTENTS(X + n) and I2 ← n; the set {n} makes the invariant true.
  • Preservation: If A ≥ CONTENTS(X + k), we keep (A,I2); otherwise we set A ← CONTENTS(X + k), I2 ← k. Either way, after stepping k ← k − r we’ve incorporated the element at offset k into the set.
  • Termination: The loop stops when k ≤ 0. Then all offsets n, n−r, …, >0 have been examined, so A is the desired maximum and I2 its position.

Final value of I3

Starting with I3 = n and repeatedly doing I3 ← I3 − r until I3 ≤ 0 gives, after q = ⌈n/r⌉ decrements,

I3_final = n − q r = n − r ⌈n/r⌉.

Using ⌈n/r⌉ = ⌊(n−1)/r⌋ + 1,

I3_final = n − r ( ⌊(n−1)/r⌋ + 1 )
         = (n − r ⌊(n−1)/r⌋) − r
         = ((n−1) mod r + 1) − r,

which matches Knuth’s succinct form −((−n) mod r).

Cost

Let m = ⌈n/r⌉ be the number of elements inspected. Each pass is a constant number of MIX instructions (one compare, a conditional 2-instruction update, one decrement, one test), so the running time is Θ(m) = Θ(⌈n/r⌉), reducing work by a factor of about r relative to a unit-stride scan.

Sanity check (small example)

Take n=7, r=3. The offsets visited are 7, 4, 1; the next would be −2, so we stop. The final I3 equals −2, and the formula gives

((7−1) mod 3) + 1 − 3 = (6 mod 3) + 1 − 3 = 0 + 1 − 3 = −2.

Consistent.

5 [21]

  1. Calling code

Load the return address (the word immediately after the call) into a non-J register—say rA—then jump to the subroutine:

      ENTA *+2        ; rA ← address of the next instruction (the return point)
      JMP  MAX100     ; enter the subroutine
; ... execution resumes here after the subroutine returns

*+2 is the location two words ahead: the word after JMP MAX100 is the resume point, and MIX’s address computation makes ENTA *+2 exactly that. (We use two words because the JMP itself occupies one word.) The subroutine does not rely on A’s initial value, since it initializes A before the first compare (see below), so there are no entry conditions to worry about.

  1. Subroutine prologue/epilogue

Replace the STJ prologue with a store from the register you chose (here rA) into the address field of EXIT:

* MAXIMUM OF X[1..100]  (abbreviated body)
MAX100  STA  EXIT(0:2)   ; linkage via rA: store sign+2-byte address into EXIT
        ENT3 100         ; M1. Initialize
        JMP  2F

1H      CMPA X,3         ; M3. Compare
        JGE  *+3
2H      ENT2 0,3         ; M4. Change m
        LDA  X,3         ;     New maximum found
        DEC3 1           ; M5. Decrease k
        J3P  1B          ; M2. All tested?
EXIT    JMP  *           ; return using the address we stored
  • EXIT(0:2) targets the address part (sign + two bytes) of the JMP * at EXIT, leaving its index/field/opcode bytes undisturbed—exactly what STJ would have done with rJ, but now with rA.
  • The rest of MAX100 is unchanged; crucially, it re-initializes rA before the first CMPA, so seeding rA with the return address in the caller does not affect correctness.

You could equally well choose another register: STX EXIT(0:2), or ST1 EXIT(0:2), … provided the subroutine uses that register to write the address field at entry and doesn’t need its incoming value for anything else.

Why this works

  • In MIX, the instruction word at EXIT is JMP *. The address field of that instruction is where the return address lives during the call.
  • The standard scheme writes rJ there with STJ EXIT; our scheme writes whatever register we like (here rA) using a store with field spec (0:2), so that only the address field is changed.
  • Control returns by executing EXIT JMP *, which now jumps to the address we just stored.

Knuth’s answer for this exercise confirms the key point—any other register can be used, and one simple way to arrange the call is:

ENTA *+2
JMP  MAX100

with no entry conditions. Our small prologue tweak (STA EXIT(0:2)) is the corresponding change inside the subroutine that makes rA the active linkage register instead of rJ.

Below is a clean MIXAL subroutine that emulates the hardware operator MOVE using the calling sequence

JMP  MOVE
NOP  A,I(F)

It copies exactly F words from the source block starting at the effective address M(A,I) to the destination block starting at rI1, word-by-word, and when finished increments rI1 by F. All other registers (A, X, I2–I6, overflow toggle, comparison indicator) are restored to their entry values, so the only externally visible differences from a real MOVE are (1) register J (as with any subroutine call) and (2) time/space. This matches the definition of MIX’s MOVE (C=7: “the number of words specified by F is moved… and rI1 is increased by the value of F at the end”), including the documented “one-word-at-a-time” behavior on overlap.

The subroutine also follows the Section 1.4.1 convention of passing arguments in the word after the jump (we read the NOP A,I(F) via rJ, which points at that word), then returning to that NOP so execution continues with the caller’s next instruction.

MOVE (software) — MIXAL

* MOVE — software emulation of hardware MOVE
* Call:   JMP MOVE
*         NOP A,I(F)
* Effect: Copy F words from M(A,I) -> [rI1 .. rI1+F-1]; then rI1 := rI1 + F.
* Notes:  Preserves rA, rX, rI2..rI6, CI, overflow; only rI1 (and rJ) differ.

MOVE    STJ     RET           Save return (address of the NOP A,I(F))

        * Save registers we will clobber
        STA     SAVEA
        ST1     SAVE1
        ST2     SAVE2
        ST3     SAVE3
        ST4     SAVE4

        * rJ points to the parameter word (NOP A,I(F)).
        STJ     JADDR
        LD2     JADDR         rI2 <- rJ  (an address we can index from)

        * Decode A, I, F from that instruction word.
        LDA     0,2(0:2)      rA <- A (±AA)
        STA     SRC           save base address A
        LD4     0,2(3:3)      rI4 <- I (0..6)
        LD3     0,2(5:5)      rI3 <- F (count)

        * Build source pointer: rI2 := A + rIi (using index arith. only;
        * INCi does not affect the overflow toggle).
        LD2     SRC           rI2 <- A
        J4Z     SRC_OK        if I = 0, no indexing needed

        * Add the selected index register to rI2
ISELECT DEC4    1
        J4Z     ADDI1
        DEC4    1
        J4Z     ADDI2
        DEC4    1
        J4Z     ADDI3
        DEC4    1
        J4Z     ADDI4
        DEC4    1
        J4Z     ADDI5
        * must be 6
        ST6     TMP
        JMP     ADDI

ADDI1   ST1     TMP
        JMP     ADDI
ADDI2   ST2     TMP
        JMP     ADDI
ADDI3   ST3     TMP
        JMP     ADDI
ADDI4   ST4     TMP
        JMP     ADDI
ADDI5   ST5     TMP
ADDI    INC2    TMP           rI2 += rIi
SRC_OK  NOP

        * Main loop: move F words, advancing both pointers
LOOP    J3Z     DONE          if F = 0, nothing to do
        LDA     0,2           rA := CONTENTS(rI2)
        STA     0,1           CONTENTS(rI1) := rA
        INC2    1             rI2++
        INC1    1             rI1++
        DEC3    1             F--
        JMP     LOOP

DONE    * rI1 has been advanced by original F (as required)

        * Restore preserved registers (except rI1)
        LDA     SAVEA
        LD2     SAVE2
        LD3     SAVE3
        LD4     SAVE4
        LD1     SAVE1

RET     JMP     *             Return to the NOP; it executes and control
                              * falls through to the caller’s next instr.

* ------- locals -------
SAVEA   CON     0
SAVE1   CON     0
SAVE2   CON     0
SAVE3   CON     0
SAVE4   CON     0
JADDR   CON     0
SRC     CON     0
TMP     CON     0

Why this matches the hardware MOVE

  1. Source/destination and count. We take A and I from the parameter and compute M(A,I) for the source; F is the count. The loop copies one word at a time in increasing address order, exactly like the hardware (so overlapping ranges behave the same), and rI1 advances by F total.

  2. Register effects. MOVE shouldn’t alter rA, rX, rI2–rI6, CI, or the overflow toggle. We therefore save/restore those we touch and use INCi (not ADD/INCA) to avoid toggling overflow. Only rI1 changes (by F), and rJ is different because of the call/return—exactly as the exercise requests.

  3. Calling/returning. Section 1.4.1’s “arguments after the jump” convention is used: we read the NOP A,I(F) via rJ; returning with JMP * lands on that NOP, which executes (doing nothing) and then the caller proceeds to the next instruction.

7 [20]

Here’s a concise, exam-style answer with enough depth for TAOCP.

Knuth’s MIX used self-modifying code for subroutine linkage (e.g., storing the return address into the instruction stream), but he already notes that “newer computers…keep [such] information in registers or in a special array, not within the program itself” (see the note in §1.4.1 that points you to this exercise). The reasons modern systems discourage self-modification include:

  1. Security policies (W⊕X). Operating systems increasingly enforce write-xor-execute: a page may be writable or executable, not both. That prevents code injection and many exploit techniques; it also means ordinary programs can’t (and shouldn’t) patch their own instruction bytes.

  2. OS/ABI design: text is shared and read-only. Executable code (“text”) pages are mapped read-only and commonly shared among processes for memory efficiency. Letting programs write into those pages would break sharing and isolation.

  3. Performance costs on modern CPUs. Pipelines and split instruction/data caches mean that when code is changed, the CPU must perform special synchronization/serialization and cache/TLB maintenance to make the modification visible to instruction fetch. That’s slow and error-prone if mishandled. Vendors explicitly warn about penalties and coherence requirements for self-modifying code.

  4. Concurrency and reentrancy hazards. Multiple threads may execute the same code simultaneously. Patching instructions in place introduces race conditions, tears, and undefined behavior unless you add heavyweight synchronization—further eroding any perceived benefit. (Read-only text plus normal call/return linkage avoids this.)

  5. Optimization, tooling, and maintainability. Compilers, linkers, profilers, verifiers, and JITs assume code is immutable at run time; immutability enables aggressive optimization, code signing, control-flow integrity, and reliable debugging. Self-modification undermines those assumptions and complicates reasoning about correctness.

  6. Better idioms exist. Modern linkage stores return addresses in a register/stack frame (or a link register), not in the instruction stream; parameter passing and multiple exits are handled by calling conventions and structured control flow, so the MIX-style “patch the code to return here” is unnecessary. (Knuth foreshadows exactly this shift.)

Bottom line. What was once a practical trick on simple machines is now a liability: it conflicts with OS security and memory models, harms performance on cached/pipelined CPUs, breaks reentrancy, and fights today’s toolchains. Hence modern advice—and practice—is to avoid self-modifying code. The exercise list where this appears is on p. 193.
