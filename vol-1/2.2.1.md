# 2.2.1 Stacks, Queues, and Deques

Notation used below:

- A **stack** supports `PUSH(x)` and `POP()` (last-in, first-out).
- A **queue** supports `ENQUEUE(x)` and `DEQUEUE()` (first-in, first-out).
- A **deque** supports insertion/deletion at both front and rear.
- An **input-restricted deque** allows insertions at only one end, but deletions at both ends.
- An **output-restricted deque** allows deletions at only one end, but insertions at both ends.

### 1 [06]
An input-restricted deque is a linear list in which all insertions are made at one end, and deletions are made at both ends. Show how to represent this structure and implement its basic operations.

A simple representation is a doubly linked list with pointers `F` (front) and `R` (rear). We allow insertion only at rear; deletion is allowed at both front and rear.

Operations:

- `INSERT(x)` (rear only): append node after `R`; update `R`; if empty, also set `F`.
- `DELETE-FRONT()`: remove node at `F`; advance `F`; if becomes empty, clear `R` too.
- `DELETE-REAR()`: remove node at `R`; move `R` backward; if becomes empty, clear `F` too.

All operations are `O(1)` because both ends are directly accessible.

### 2 [10]
Give an analogous implementation for an output-restricted deque.

Now deletions are allowed only at one end (say front), but insertions at both ends.

Use the same doubly linked list with `F` and `R`:

- `INSERT-FRONT(x)`: prepend before `F`.
- `INSERT-REAR(x)`: append after `R`.
- `DELETE-FRONT()`: remove `F`.

Again each operation is `O(1)`.

### 3 [08]
Explain how a queue can be implemented using a circular array.

Maintain:

- `Q[0..m-1]`
- index `head` (position of next item to remove)
- index `tail` (position where next item is inserted)
- count `n`

Rules:

- `ENQUEUE(x)`: if `n=m`, overflow; else `Q[tail]=x`, `tail=(tail+1) mod m`, `n++`.
- `DEQUEUE()`: if `n=0`, underflow; else `x=Q[head]`, `head=(head+1) mod m`, `n--`.

Circular indexing avoids shifting elements and makes each operation `O(1)`.

### 4 [12]
Show how two stacks can share one array efficiently.

Let array be `A[0..m-1]`.

- Stack 1 grows upward from left: top index `t1`, initially `-1`.
- Stack 2 grows downward from right: top index `t2`, initially `m`.

Push rules:

- Push to stack 1 if `t1+1 < t2`, then `A[++t1]=x`.
- Push to stack 2 if `t1+1 < t2`, then `A[--t2]=x`.

Pop rules:

- Pop stack 1 if `t1>=0`, return `A[t1--]`.
- Pop stack 2 if `t2<m`, return `A[t2++]`.

Overflow occurs exactly when `t1+1=t2`, i.e., all cells are used.

### 5 [12]
Compare linked and sequential representations for stacks and queues.

Sequential (array):

- Pros: compact, cache-friendly, no pointer overhead.
- Cons: fixed capacity unless resized; possible overflow despite free memory elsewhere.

Linked:

- Pros: dynamic growth/shrinkage; no global fixed bound.
- Cons: extra pointer fields; allocator overhead; poorer locality.

For predictable bounded workloads, arrays are often better. For highly dynamic workloads, linked structures are often safer.

### 6 [10]
For stack permutations, characterize when an output permutation is feasible from input `1,2,...,n`.

Classical criterion: feasible permutations are exactly those obtainable by a sequence of push/pop operations on one stack.

Equivalent pattern characterization: feasible permutations avoid pattern `231`.

Operational decision procedure:

1. Scan input `1..n`, pushing until top equals next required output.
2. Pop whenever top matches required output.
3. If you get stuck (input exhausted and top mismatch), permutation is impossible.

This gives a linear-time recognizer.

### 7 [14]
Describe how to test stack-permutation feasibility in linear time.

Use the simulation procedure directly:

- `nextIn = 1`, `nextOut = 1`.
- While `nextOut <= n`:
  - If stack top equals required output, pop and increment `nextOut`.
  - Else if `nextIn <= n`, push `nextIn` and increment `nextIn`.
  - Else fail.

If loop completes, succeed.

Complexity is `O(n)` time and `O(n)` auxiliary stack space.

### 8 [16]
Give a queue implementation using two stacks.

Let stacks be `S_in` and `S_out`.

- `ENQUEUE(x)`: push `x` onto `S_in`.
- `DEQUEUE()`:
  - If `S_out` nonempty, pop from `S_out`.
  - Else move all items from `S_in` to `S_out` by repeated pop/push, then pop from `S_out`.

Correctness intuition: transfer reverses order once, restoring FIFO at dequeue side.

Amortized cost is `O(1)` per operation, since each element moves stacks at most once before removal.

### 9 [12]
Implement a deque with a doubly linked list and sentinels.

Use sentinel nodes `H` (head) and `T` (tail) with invariant `H.next ... T.prev`.

Operations:

- `pushFront(x)`: insert after `H`.
- `pushBack(x)`: insert before `T`.
- `popFront()`: remove node after `H` (unless empty).
- `popBack()`: remove node before `T` (unless empty).

Sentinels eliminate special-case pointer code at boundaries.

### 10 [18]
Discuss underflow/overflow handling policies for linear-list primitives.

Good practice:

- Underflow (remove from empty): explicit error code/exception.
- Overflow (insert into full bounded array): explicit error; or resize and retry.
- Keep operation contracts precise: either total (return status) or partial (signal failure).
- Never silently ignore failures; that corrupts higher-level algorithms.

### 11 [20]
Show how to reverse a queue using one stack.

Algorithm:

1. While queue nonempty: dequeue item and push onto stack.
2. While stack nonempty: pop item and enqueue back into queue.

First pass reverses order into LIFO form; second pass emits that reversed order back to FIFO container.

Time `O(n)`, extra space `O(n)`.

### 12 [20]
Show how to reverse a stack using one queue.

One method:

1. Pop all stack items and enqueue them (top to bottom order).
2. Repeat until queue empty:
   - rotate queue `k-1` steps to bring last element to front,
   - dequeue that element and push to stack,
   - decrement `k`.

This reconstructs stack in reversed order. Straightforward version is `O(n^2)`; with extra structures, better bounds are possible.

### 13 [24]
Suppose you are allowed to use only queues as data structures. Implement a stack.

Use two queues `Q1, Q2`.

Push-heavy version:

- `PUSH(x)`: enqueue to `Q1` (`O(1)`).
- `POP()`:
  1. Move first `|Q1|-1` items from `Q1` to `Q2`.
  2. Dequeue last item from `Q1` (this is stack top).
  3. Swap names of `Q1` and `Q2`.

This gives `PUSH=O(1)`, `POP=O(n)`.

(Alternate design gives `PUSH=O(n)`, `POP=O(1)`.)

### 14 [26]
Suppose you are allowed to use only stacks as data structure. Implement a queue.

Use two stacks `S1` and `S2`.

Operations:

- `ENQUEUE(x)`: `PUSH(S1, x)`.
- `DEQUEUE()`:
  1. If `S2` is nonempty, `POP(S2)`.
  2. Else move everything from `S1` to `S2` by repeated pop/push.
  3. If `S2` still empty, underflow; else pop from `S2`.

Why FIFO holds:

- Items enter `S1` newest-on-top.
- Transfer to `S2` reverses order, making oldest item topmost in `S2`.
- Dequeue from `S2` therefore removes oldest remaining element.

Complexity:

- Worst-case single `DEQUEUE` can be `O(n)` during transfer.
- Amortized over long sequences, each element is moved at most once between stacks, so average per operation is `O(1)`.
