# 2.5 Dynamic Storage Allocation

This section studies how memory blocks are reserved and released efficiently.

For beginners, always track three things:

1. **Correctness invariant** (free/used blocks partition memory, no overlap).
2. **Fragmentation** (internal vs external).
3. **Cost model** (worst-case vs amortized).

### 1 [06]
Core specification for any allocator:

- request format,
- block metadata format,
- reserve/release semantics,
- failure behavior.

Without this contract, correctness cannot be checked.

### 2 [08]
First-fit policy.

Algorithm: scan free list from start, pick first block big enough.

Pros: fast to implement.
Cons: front of list gets fragmented.

### 3 [08]
Best-fit policy.

Choose smallest block that satisfies request.

Pros: small immediate leftovers.
Cons: expensive search and many tiny unusable fragments over time.

### 4 [08]
Next-fit policy.

Like first-fit but resume scan where previous scan ended.

Pros: avoids repeatedly rescanning list head.
Cons: allocation quality may worsen under skewed patterns.

### 5 [08]
Worst-fit policy.

Choose largest free block.

Idea: preserve medium blocks by carving from largest.
Often unstable in mixed-size workloads.

### 6 [10]
Boundary tags.

Store size/state at both block header and footer.

Benefit: on release, neighbors are found in O(1), so coalescing is immediate and local.

### 7 [10]
Immediate vs deferred coalescing.

- Immediate: merge on every release (better space, extra per-release cost).
- Deferred: merge later (faster release path, possible transient fragmentation).

### 8 [10]
External fragmentation definition.

Total free memory is enough, but no single contiguous free block is large enough.

### 9 [10]
Internal fragmentation definition.

Allocated block contains unused bytes because allocator granularity exceeds request.

### 10 [12]
Split rule.

When reserving from a larger free block, split only if remainder can form valid free block (>= minimum block size + metadata). Otherwise hand out whole block.

### 11 [12]
Fundamental partition invariant.

At all times:

- every byte belongs to exactly one block,
- each block is either free or allocated,
- adjacency/size metadata is consistent.

### 12 [12]
Compaction.

Compaction relocates live blocks together to eliminate holes, then creates one large free region.

Requires pointer-fixup mechanism.

### 13 [14]
Mark-sweep GC.

1. Mark all nodes reachable from roots.
2. Sweep heap: unmarked blocks go to free list; marked blocks survive.

Simple and robust, but may pause for full-heap scan.

### 14 [14]
Reference counting.

Immediate reclamation when count hits zero.

Limitation: cycles are never reclaimed unless cycle handling is added.

### 15 [14]
Copying collection.

Copy only live objects to to-space, then swap spaces.

Good locality, natural compaction, cost proportional to live data.

### 16 [16]
Buddy allocation overview.

Memory is managed in powers-of-two blocks.

Reserve: split larger blocks recursively.
Release: merge with free buddy of same order.

### 17 [16]
Buddy merge condition.

Blocks `B` and buddy(`B`) merge iff both are free and same size order.
Then attempt recursive upward merge.

### 18 [12]
Binary buddy internal waste bound.

A request is rounded to next power of two, so waste is strictly less than 50% of granted block.

### 19 [12]
Choosing minimum block size.

Too small → metadata overhead dominates.
Too large → internal fragmentation dominates.
Choose based on request-size distribution.

### 20 [14]
Segregated free lists.

Maintain one free list per size class.

Average reserve/release becomes near O(1) for common sizes.

### 21 [14]
No universally best policy.

Allocator performance is workload-dependent (request size/lifetime pattern), so benchmark-guided choice is essential.

### 22 [12]
Real-time constraint.

Need bounded worst-case latency: fixed-size pools/slabs often preferred to variable-size global heaps.

### 23 [16]
Pool allocation.

Great for fixed object sizes: fast, predictable, minimal external fragmentation.
Tradeoff: poor fit for variable sizes.

### 24 [16]
Slab allocation.

Adds object caches and constructor reuse; improves locality and repeated allocation/deallocation throughput.

### 25 [14]
Batching reclamation.

Deferring expensive maintenance and doing it in batches can improve throughput, though latency spikes may increase.

### 26 [14]
Relocation requirement.

If objects move, all pointers to moved objects must be updated (precise maps, forwarding pointers, or conservative techniques).

### 27 [12]
Allocator diagnostics.

Useful checks:

- double free detection,
- corrupted free-list links,
- leak counters,
- block-accounting consistency.

### 28 [14]
Graph/game-state storage view.

Retrograde win/loss marking naturally uses dynamic node pools plus predecessor lists; free-list recycling controls memory growth.

### 29 [12]
Mass erase operation.

Releasing an entire linked structure is linear in number of nodes. Implementation can splice released chain to free list in O(1) once chain endpoints are known.

### 30 [12]
Amortized perspective.

Occasional expensive global operations (repack/compact) are acceptable when triggered infrequently enough that average cost per request remains small.

### 31 [18]
Fibonacci buddy idea.

Potentially lowers internal waste versus binary buddy, but split/merge arithmetic and buddy lookup are more complex.

### 32 [M20]
Analytical occupancy problems.

Expected waste/throughput must be derived from explicit probabilistic model of requests; no single closed form applies universally.

### 33 [20]
Linear-time mark-compact outline.

1. Mark live objects.
2. Compute new addresses by prefix sums.
3. Rewrite pointers.
4. Move objects.

Done carefully, each phase is linear in heap size or live size.

### 34 [20]
Why naive compaction becomes quadratic.

If each moved object triggers a full scan to update references, total work is `O(n^2)`. Use forwarding tables or two-pass pointer rewrite to retain linear complexity.

### 35 [14]
Relation to §2.2.2 reallocation.

§2.2.2 reallocates contiguous table segments; §2.5 generalizes to arbitrary object lifetimes and pointer-rich structures, introducing fragmentation and GC concerns.

### 36 [22]
Seating problem as allocation analogy.

This models online placement with mixed request sizes. A good strategy reserves patterns that avoid dead-space residues, preserving future feasibility.

### 37 [16]
Why `22` seats can fail when `23` can succeed.

Capacity thresholds can be parity/mod-pattern sensitive; one missing slot may remove all valid placements for a particular arrival sequence.

### 38 [16]
`N(n,2)` style guarantees.

These values characterize guaranteed accommodation under adversarial unit/double requests; derive by extremal configurations and invariant arguments.

### 39 [M24]
Higher-order bounds.

Typically require advanced combinatorics/probabilistic methods; constructive proofs are uncommon for general `k`.

### 40 [M24]
Open problems.

Exact formulas for several online packing guarantees remain difficult; empirical/simulation support is often used.

### 41 [22]
Buddy-system worst-case bound intuition.

Free space distribution across orders forms a geometric hierarchy; summing maximal unusable residues yields linear bound in highest order scale.

### 42 [M24]
Refined adversarial analysis.

Requires detailed state-transition modeling of order lists and split/merge events.

### 43 [M24]
Expected-case buddy utilization.

Can be modeled via Markov chains over free-block count vectors by order.

### 44 [M24]
Nonbinary buddy families.

Generalized buddy systems may improve fit but significantly complicate correctness proofs and policy tuning.
