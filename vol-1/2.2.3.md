# 2.2.3 Linked Allocation

This section uses linked-list storage (records with link fields) and, later, successor/predecessor lists for directed graphs.

Notation:

- `Λ` = null link.
- `AVAIL` = head of free-list.
- `TOP`, `F`, `R` = conventional stack/queue pointers.
- For topological sorting, `COUNT(v)` = number of predecessors of `v` not yet removed.

### 1 [06]
Why doesn’t a simple stack push operation always mention overflow?

In linked allocation, stack growth depends on free nodes, not on a fixed array bound. So the primitive `PUSH` typically acquires one cell from `AVAIL`; overflow occurs only when `AVAIL = Λ`.

### 2 [08]
Give linked-allocation insertion for a list.

Insert new node `P` after node `X`:

1. `P ← AVAIL`; if `P=Λ` overflow.
2. `AVAIL ← LINK(P)`.
3. `INFO(P) ← Y`.
4. `LINK(P) ← LINK(X)`.
5. `LINK(X) ← P`.

### 3 [08]
Give linked-allocation deletion for a list.

Delete node after `X`:

1. `P ← LINK(X)`; if `P=Λ` underflow/not found.
2. `LINK(X) ← LINK(P)`.
3. Return cell to free list: `LINK(P) ← AVAIL`, `AVAIL ← P`.

### 4 [10]
How to handle overflow in a system with fixed memory (`SEQMIN`-style)?

When `AVAIL=Λ`:

- either signal failure immediately,
- or invoke compaction/reclamation (if available), then retry once.

Correct policy must be explicit; silent failure is invalid.

### 5 [12]
Modify queue operations to support deque-like behavior with linked nodes.

With singly linked nodes plus both `F` and `R` pointers:

- rear insertion: `O(1)`.
- front deletion: `O(1)`.
- front insertion can be `O(1)` (update `F`).
- rear deletion is `O(n)` unless doubly linked list or predecessor tracking is used.

Hence full efficient deque needs doubly linked representation.

### 6 [10]
Can we avoid forcing the last link to null in queue conventions?

Yes, if empty/nonempty is represented by pointer-state invariants (e.g., both `F` and `R` null when empty, otherwise consistent reachability), not by testing a terminal null in data path. But null-terminated chains are simpler and less error-prone.

### 7 [14]
How to invert links of a singly linked list in place?

Standard three-pointer method:

- `prev ← Λ`, `curr ← FIRST`.
- While `curr ≠ Λ`:
  - `next ← LINK(curr)`
  - `LINK(curr) ← prev`
  - `prev ← curr`
  - `curr ← next`
- `FIRST ← prev`.

Time `O(n)`, space `O(1)`.

### 8 [12]
State correctness idea for list inversion algorithm.

Loop invariant: `prev` heads already-reversed prefix; `curr` heads not-yet-reversed suffix. Each iteration moves one node from suffix to reversed prefix. On termination suffix is empty, so `prev` is full reversed list.

### 9 [14]
Which relations are partial orderings?

Check reflexive, antisymmetric, transitive.

- On numbers, `x ≼ y` defined as `x ≥ y` is a partial order.
- “is a multiple of” on all integers fails antisymmetry (`1` and `-1`).
- Dependency relation on proofs is a partial order if self-dependence/cycles are excluded.
- Parity relation `x+y even` is equivalence, not partial order.
- Call-graph relation `x calls y` is not reflexive/transitive in general and may cycle.

### 10 [10]
Can subset relation be viewed as a partial ordering?

Yes. `⊆` on sets is reflexive, antisymmetric, and transitive; hence a partial order.

### 11 [16]
Enumerate topological sorts of a given DAG.

Method: recursively pick any zero-indegree vertex, output it, delete its outgoing edges, and continue; backtrack over all choices. This generates exactly all topological orderings.

### 12 [16]
Give two useful ways to order all subsets of an `n`-set (under inclusion constraints).

Two classic arrangements:

1. By nondecreasing cardinality (`|S|=0,1,...,n`).
2. Lexicographic/Gray-code within each layer, preserving layer order.

Layered ordering respects inclusion direction (a set can only include sets of smaller/equal size).

### 13 [20]
Count topological orders of the Boolean lattice (all subsets under inclusion).

Exact closed forms are nontrivial for general `n`; counting linear extensions of this poset is a hard combinatorial problem. Practical approach: dynamic programming over antichains/ideals for moderate `n`.

### 14 [14]
When is a partial order actually total?

Exactly when every pair is comparable: for all `x,y`, either `x≼y` or `y≼x`.
Then there is a unique relative position between any pair, so linear extension is forced (up to ties if indistinguishable elements are allowed).

### 15 [16]
Show the covering relation (Hasse edges) uniquely determines a finite partial order.

Take transitive closure of cover edges. This reconstructs all order relations, and transitive reduction of a finite poset is unique. Hence cover relation characterizes the poset.

### 16 [18]
Incidence matrix and diagonal form under topological renumbering.

If vertices are renumbered in topological order, then no edge goes from higher index to lower. Therefore incidence/adjacency entries below diagonal become zero (upper-triangular form, excluding diagonal conventions).

### 17 [18]
Why does topological sorting exist iff no directed cycle exists?

- If a DAG: always at least one zero-indegree vertex; remove repeatedly.
- If cycle exists: every vertex on cycle has indegree ≥1 within cycle, so removal process eventually blocks.
Thus acyclicity is necessary and sufficient.

### 18 [18]
Compare queue-based vs stack-based zero-indegree selection in Algorithm T.

Both are correct (produce valid topological orders). They differ only in which valid order appears:

- queue tends toward breadth-like selection,
- stack tends toward depth-like selection.

### 19 [12]
How do repeated relations affect Algorithm T?

Repeated edge `(u,v)` increments/decrements `COUNT(v)` multiple times consistently; still correct if duplicates are intended multiarcs. Usually input should deduplicate or treat duplicates as single relation.

### 20 [12]
What about self-reference `(v,v)`?

A self-loop is a directed cycle of length 1, so topological ordering is impossible. Algorithm T will fail to process all vertices (nonzero counts remain).

### 21 [14]
How should Program T validate malformed input?

Checks:

- vertex labels in range,
- nonnegative counts,
- relation count within declared bounds,
- no illegal memory links,
- optional rejection of duplicates/self-loops per spec.

### 22 [14]
How to report presence of a loop after topological sort attempt?

After processing queue empties, if output count `< n`, graph has cycle. Any remaining vertex with positive indegree lies in or downstream of cycles; use DFS on remaining subgraph to extract an explicit cycle.

### 23 [16]
How to print an actual cycle efficiently?

Run DFS with colors (`white/gray/black`) on remaining vertices.
Upon finding edge to gray vertex, unwind parent pointers to print cycle path.

### 24 [16]
Program-level strategy for loop detection in linked representation.

Maintain predecessor array while DFS traverses successor lists (linked nodes). When back-edge found, follow predecessor chain to repeated vertex and emit nodes in reverse, then reverse output.

### 25 [24]
Topological sorting when graph exceeds memory (tape/external storage model).

Use external-memory passes:

1. Build indegree file and edge file sorted by source.
2. Maintain frontier of zero-indegree vertices in memory buffer blocks.
3. Stream edges for removed vertices, decrementing indegrees via batched external updates.
4. Repeat rounds until frontier empty.

This minimizes random tape seeks by block-wise sequential passes.

### 26 [20]
Design relocating algorithm for linked objects.

Compute relocation map `old→new`, copy records, then rewrite every link field by map lookup. Finally update roots (`FIRST`, `AVAIL`, etc.). This is the standard compacting/relocating pass.

### 27 [20]
What invariant guarantees relocation correctness?

For every original node `u`:

- payload preserved at `map(u)`;
- each pointer field formerly to `v` now points to `map(v)`;
- null links remain null.

By structural induction over reachability, all list/graph shapes are preserved.

### 28 [24]
Classify game positions as win/loss using predecessor/successor lists.

Retrograde analysis:

- terminal losing positions marked `L`.
- predecessor of any `L` is `W`.
- position with all successors `W` is `L`.

Implement with queue of newly classified nodes and counters of unresolved nonwinning successors.

### 29 [18]
Mass erase of a list given (a) `FIRST` only, (b) `FIRST`/`SECOND`.

(a) `FIRST` only:

Traverse entire list; append chain to `AVAIL` in `O(n)`.

(b) `FIRST`/`SECOND` given (e.g., two-node access):

Still `O(n)` total, but can optimize pointer updates by preserving tail while traversing; asymptotically unchanged.

### 30 [20]
Queue representation where empty is `F=0` and `R` undefined: write insert/delete.

Insert `x`:

- if `F=0`: allocate node `P`, set `F←R←P`.
- else append after `R`, set `R←new`.

Delete:

- if `F=0`: underflow.
- else remove `F`, set `F←LINK(F_old)`.
- if new `F=0`, also clear `R` (queue became empty).

This preserves invariant: `F=0` iff queue empty; otherwise `R` points to last node.
