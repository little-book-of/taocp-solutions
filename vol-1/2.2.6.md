# 2.2.6 Arrays and Orthogonal Lists

Notation:

- `A` is an array with base address `LOC(A[l1,...,lk])`.
- Bounds in dimension `r`: `l_r..u_r`.
- Each element occupies `c` words.
- Row-major order is assumed unless stated otherwise.

### 1 [06]
For an `m×n` matrix (1-indexed), each entry using 2 words, location of `A[j,k]` is

\[
LOC(A[1,1]) + 2\big((j-1)n + (k-1)\big).
\]

### 2 [10]
General location formula for `k`-dimensional array with arbitrary bounds:

\[
LOC(A[i_1,\dots,i_k]) = LOC(A[l_1,\dots,l_k]) + \sum_{r=1}^{k} \alpha_r(i_r-l_r),
\]

with

\[
\alpha_r = c\prod_{s=r+1}^{k}(u_s-l_s+1).
\]

### 3 [08]
Lower-triangular storage (1-indexed, `1\le k\le j\le n`) in packed row-major form:

\[
LOC(A[j,k]) = LOC(A[1,1]) + \frac{(j-1)j}{2} + (k-1).
\]

### 4 [10]
Upper-triangular packed formula (`1\le j\le k\le n`):

\[
LOC(A[j,k]) = LOC(A[1,1]) + (j-1)n - \frac{(j-1)(j-2)}{2} + (k-j).
\]

### 5 [12]
Using indirect/indexed addressing for one-instruction element access:

Use a precomputed row-base table `B[j]=LOC(A[j,1])`. Then

\[
LOC(A[j,k]) = B[j] + (k-1)c.
\]

For triangular arrays, similarly precompute row starts in table `T[j]` and add intra-row offset.

### 6 [14]
3D tetrahedral arrays (`1\le k\le j\le i\le n`) packed location:

\[
LOC(A[i,j,k]) = LOC(A[1,1,1]) + \binom{i+1}{3} + \binom{j}{2} + (k-1).
\]

(Equivalent shifted forms are fine.)

### 7 [14]
`k`-dimensional simplex (tetrahedral generalization):

For indices satisfying `i_1\ge i_2\ge\cdots\ge i_k\ge 1`, address uses cumulative binomial ranks:

\[
LOC = LOC_0 + \sum_{r=1}^{k}\binom{i_r + k-r-1}{k-r+1}
\]

(up to index-origin shifts).

### 8 [12]
Neat storage for all oriented tetrahedral variants:

Embed in a cube-like host and map each orientation by coordinate transforms (sign/permutation). This allows shared indexing machinery with small orientation-specific affine transforms.

### 9 [16]
One-pass intersection query for sparse attribute lists (e.g., female, blue-eyed, blond, ages 21..23):

Maintain pointers in each sorted list; repeatedly advance pointers at current minimum key until all active pointers match. Output matches; terminate when any mandatory list ends.

This is multiway sorted-list intersection, `O(total traversed nodes)`.

### 10 [10]
Better organization for search:

Use inverted indexes + hashing/bitmaps per attribute. Intersections become fast bitset ANDs or short-list intersections.

### 11 [10]
Storage for 200×200 matrix with ≤4 nonzeros per row, 3 words/node + row/column heads:

- nodes: `200*4*3 = 2400`
- row heads: `200`
- column heads: `200`

Total `2800` words.

### 12 [12]
At start of step S7 (symbolic references `a,b,c,d`), values are:

\[
VAL(Q_0)=c,\quad VAL(P_0)=b/a,\quad VAL(P_1)=d
\]

(as specified by preceding normalization/pivot steps).

### 13 [10]
Why circular lists in sparse matrix routines?

They remove null-end tests and simplify repeated cyclic scans over row/column chains; boundary logic becomes uniform.

### 14 [14]
Show sparse pivot savings can be achieved with sequential storage + auxiliary `LINK[j]`.

Store nonzeros sequentially, but maintain per-row (or per-column) linked index chains through `LINK`. Pivot operations iterate only listed nonzeros, skipping implicit zeros; therefore work scales with nonzero count, not full matrix size.

### 15 [18]
Pivot algorithm (high level):

1. choose pivot `(r,c)` with nonzero value.
2. scale pivot row/column as required.
3. for each nonzero in pivot row/column, update affected entries.
4. delete newly zero entries; insert newly nonzero entries into orthogonal structure.
5. update row/column head links.

Complexity depends on local sparsity around pivot.

### 16 [18]
Running-time estimate of sparse pivot:

If `d_r` and `d_c` are pivot row/column nonzero counts, dominant work is proportional to interactions among these sets; typical cost `O(d_r d_c)` plus list-maintenance overhead.

### 17 [20]
Sparse matrix multiplication (orthogonal-list form):

For each row `i` of `A`, accumulate products with columns reachable through nonzeros:

- traverse nonzeros `A[i,k]`,
- traverse row/column list of `B[k,*]`,
- accumulate into temporary map for row `i`,
- emit nonzero accumulators into result orthogonal lists.

This avoids touching zero entries.

### 18 [16]
Inverse of
\[
\begin{bmatrix}1&2&3\\0&1&2\\0&0&1\end{bmatrix}
\]

Since matrix is upper triangular unit-diagonal, inverse is
\[
\begin{bmatrix}
1 & -2 & 1\\
0 & 1 & -2\\
0 & 0 & 1
\end{bmatrix}.
\]

(Direct check by multiplication gives identity.)

### 19 [18]
How to invert sparse matrices with orthogonal lists:

Use sparse Gaussian elimination / sparse pivoting with fill-in management:

- choose pivots with heuristic minimizing fill,
- perform sparse row/column updates via linked nonzero structures,
- track permutations separately,
- apply recorded row/column permutations to final inverse factors.

### 20 [12]
Tridiagonal storage scheme:

Store three diagonals in separate vectors (or packed by rows):

- subdiag `L[2..n]`,
- diag `D[1..n]`,
- superdiag `U[1..n-1]`.

Addressing is `O(1)` with simple case split on `j-i` in `{ -1,0,1 }`.

### 21 [16]
Allocation for `N×N` matrix where `N` grows dynamically:

Use blocked growth (doubling or chunked square blocks). Keep current capacity `C`; when `N>C`, allocate larger block, copy old `C^2` entries, update base pointer. Amortized insertion/expansion cost remains bounded.

### 22 [24]
Find polynomial/ranking mapping from `k`-tuples of nonnegative integers to nonnegative integers, monotone by sum of indices.

Use graded lexicographic rank:

\[
rank(i_1,\dots,i_k)=\sum_{s=0}^{S-1}\binom{s+k-1}{k-1}+\text{offset within grade }S,
\]
where `S=i_1+...+i_k`.

First term counts all tuples with smaller sum; second term ranks within fixed-sum layer.

### 23 [20]
Storage function for extensible matrix indexed by expanding shells:

Let `x=max(i,j)`. Total cells before shell `x` is `(x-1)^2`. Then offset inside shell is affine in `(i,j)` by side (top/right/bottom/left depending layout). So

\[
LOC(A[i,j]) = LOC(A[1,1]) + (x-1)^2 + \delta(i,j,x),
\]

with `\delta` piecewise by shell side.

### 24 [18]
How to access sparse-array elements in contiguous memory without initializing all zeros:

Use lazy initialization with timestamp/version arrays:

- data array stores values,
- tag array stores “epoch when set”.

Element is logically zero unless `tag[idx]==current_epoch`. Clearing whole array becomes `current_epoch++` (O(1)); physical zero-fill avoided.
