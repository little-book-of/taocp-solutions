# 2.4 Multilinked Structures

These exercises center on hierarchical name tables, scope/reference resolution, and `MOVE CORRESPONDING`-style matching.

### 1 [08]
COBOL data descriptions are naturally ordered by tree traversal. In practice they are processed in preorder (visit declaration, then descendants). This supports single-pass stack-based construction of nesting context.

### 2 [10]
Algorithm A runs in linear time in number of declarations `N` (amortized): every item is pushed once and popped at most once. Thus total stack adjustment work is `O(N)`, and overall pass is `O(N)`.

### 3 [12]
To adapt Algorithm A to PL/I-like semantics, same-level/ancestor transitions must be detected by comparing lexical levels against stack top and popping until the correct parent level is found, then linking as sibling or child accordingly.

### 4 [12]
Ambiguity with repeated names is handled by retaining parent/sibling context chains and requiring fully qualified resolution when multiple candidates survive lookup. A robust rule is: if two visible declarations with same name are reachable without one hiding the other, reject as ambiguous.

### 5 [10]
Algorithm B (reference check) with linked parent pointers:
1. parse qualified name components,
2. locate rightmost component in symbol bucket,
3. walk parent chain backward against remaining qualifiers,
4. accept iff all components match exactly.

### 6 [10]
Under strict PL/I complete qualification, abbreviated references are invalid: every ancestor component must be supplied, so matching is exact path-equality in the declaration tree.

### 7 [08]
`MOVE CORRESPONDING SALES TO PURCHASES` copies values between elementary items with matching names under compatible substructures; unmatched names are ignored. So only intersecting leaf-name set is moved.

### 8 [06]
`MOVE CORRESPONDING A TO B` equals `MOVE A TO B` only when both denote fully corresponding elementary items (or structurally identical groups with one-to-one leaf order and names).

### 9 [16]
Correctness of Algorithm C (`MOVE CORRESPONDING` matching):
- soundness: output pair only when names/path constraints match,
- completeness: DFS/BFS through both subtrees visits every candidate leaf,
- uniqueness: each output pair produced once by deterministic traversal order.

### 10 [14]
Without `NAME` field in data table, use symbol-table row ids/hashes as surrogate identifiers. Matching becomes equality on symbol id plus qualification constraints.

### 11 [14]
Speedups for B/C:
- hash by rightmost component,
- cache ancestor signatures (rolling hashes of parent chain),
- keep per-scope dictionaries of visible names,
- short-circuit by depth bounds.
These reduce average lookup from linear in scope chain to near constant expected.

### 12 [12]
If scopes are explicitly nested blocks, one can maintain display/stack of active scopes and perform nearest-scope resolution first, then outward fallback; this yields standard lexical binding behavior.

### 13 [14]
For repeated large `MOVE CORRESPONDING`, precompute correspondence maps between record types once, then execute moves via compiled map. Build cost amortizes across many runtime calls.

### 14 [12]
A formal invariant for scope table construction: after processing first `t` declarations, table encodes exactly the visible declaration tree for prefix `1..t`, with stack equal to active ancestor chain at declaration `t`.

### 15 [16]
Overall chapter takeaway: multilinked structures trade extra links (parent/child/sibling/prev) for asymptotically better query/update operations in compilers and record-processing systems, especially under nested qualification semantics.
