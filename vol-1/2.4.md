# 2.4 Multilinked Structures

This section is about data structures where each record has **multiple links** (for example: parent, first-child, next-sibling, previous sibling, etc.).

For beginners, the most useful mindset is:

1. First identify the exact relation each pointer encodes.
2. Write the invariant (what must always be true after every update).
3. For each algorithm, prove two things: it never produces invalid links, and it eventually finds/updates what it should.

### 1 [08]
COBOL data descriptions are naturally processed in **preorder** (visit an item, then its nested items).

Step by step:

1. A declaration line introduces a data item and its level.
2. Any immediately following more-indented lines are descendants of that item.
3. Therefore building the structure in one scan means: create node now, attach children later as we descend.

That is exactly preorder behavior.

### 2 [10]
Running time of Algorithm A (scope table construction) is linear in declarations.

Reasoning:

1. Each declaration is read once.
2. Each declaration is pushed to stack once.
3. Each declaration can be popped at most once.

So total stack operations are `O(N)`; plus one pass over input, still `O(N)`.

### 3 [12]
Adapting Algorithm A to PL/I-style semantics.

Difference from simple level processing: equal names and level transitions require stricter qualification logic.

Practical adaptation:

1. While current level is less than stack top level, pop.
2. If equal level, link as sibling under same parent context.
3. If deeper level, link as child of current stack top.
4. Record full qualification path so later references require full match (PL/I mode).

### 4 [12]
How to guard against ambiguous references with repeated names.

Use a two-stage lookup:

1. Find all visible declarations with target leaf name.
2. Filter candidates by qualification path (parent chain).
3. If exactly one remains: success.
4. If more than one remains: ambiguity error.

This is safer than “first match wins.”

### 5 [10]
Algorithm B with linked parent references (reference checking).

Reference `A.B.C` is checked from right to left:

1. Find candidate nodes named `C` in visible scope set.
2. For each candidate, check parent is `B`, grandparent is `A`.
3. Accept candidate if full chain matches.

Complexity depends on candidate count and average ancestor depth.

### 6 [10]
PL/I complete-reference rule.

If PL/I policy requires complete qualification, then shorthand like `C` where `A.B.C` is required is invalid even if unique.

So algorithm must compare full path length and full parent chain, not only suffix.

### 7 [08]
`MOVE CORRESPONDING SALES TO PURCHASES` behavior.

Interpretation:

1. Enumerate elementary leaves of both records.
2. Match leaves by name (and compatible type rules, if enforced).
3. Move values only for matched pairs.

Unmatched names are ignored.

### 8 [06]
When is `MOVE CORRESPONDING A TO B` identical to `MOVE A TO B`?

Only when structures are effectively already one-to-one aligned for all leaves moved by plain `MOVE` (same layout semantics).

If structural differences exist, corresponding move and whole-record move differ.

### 9 [16]
Correctness sketch for Algorithm C (`MOVE CORRESPONDING` pairing).

Need three properties:

1. **Soundness**: every produced pair truly corresponds.
2. **Completeness**: every corresponding pair is eventually found.
3. **No duplicates**: no pair emitted twice.

Why these hold:

- Tree traversal visits every relevant leaf.
- Matching predicate is exact and deterministic.
- Traversal order is fixed, so each pair has one encounter point.

### 10 [14]
Implement B and C without explicit NAME field.

Replace textual names with symbol-table ids:

1. Parser interns each identifier in symbol table.
2. Nodes store symbol id instead of literal string.
3. Equality test uses id equality.

This improves speed and reduces repeated string comparison.

### 11 [14]
How to speed up B/C in practice.

Strong options:

- hash maps from `(scope, symbol-id)` to declaration,
- cached ancestor signatures,
- memoized correspondence maps between record types.

Main gain: avoid repeated parent-chain scans for frequently used paths.

### 12 [12]
Scope-stack invariant for lexical blocks.

At any parsing point:

- stack bottom = global scope,
- stack top = current innermost scope,
- visible names are union of stack scopes with nearest-scope precedence.

Maintaining this invariant makes reference resolution straightforward.

### 13 [14]
Batch optimization for repeated `MOVE CORRESPONDING`.

If two record types appear repeatedly, precompute mapping once:

`[(source-offset, target-offset), ...]`

Then runtime move is a simple loop over that map, much faster than repeated tree matching.

### 14 [12]
Formal invariant for Algorithm A table construction.

After processing first `t` declarations:

1. Every processed declaration appears exactly once in table.
2. Parent-child relations reflect nesting levels.
3. Stack encodes active ancestor path for parser position.

Induction on `t` proves construction correctness.

### 15 [16]
Why multilinked structures are worth the extra pointers.

Compared with single-link structures, extra links provide:

- O(1) upward movement (parent link),
- O(1) delete from middle with prev link,
- fast sibling navigation.

So we spend memory to save repeated scans and simplify update logic.
