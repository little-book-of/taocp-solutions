# 2.2.5 Doubly Linked Lists

Notation:

- `LLINK(P)` is left predecessor of node `P`.
- `RLINK(P)` is right successor of node `P`.
- `LEFT`, `RIGHT` are deque endpoints.
- `Λ` denotes null link.

### 1 [06]
Give insertion and deletion specifications at the left end of node `P`.

For insertion of value `Y` at left end:

1. `P ← AVAIL`; if `P=Λ`, overflow.
2. `AVAIL ← RLINK(P)` (or free-list successor convention).
3. `INFO(P) ← Y`.
4. `LLINK(P) ← Λ`.
5. `RLINK(P) ← LEFT`.
6. If `LEFT ≠ Λ`, set `LLINK(LEFT) ← P`; else set `RIGHT ← P`.
7. `LEFT ← P`.

For deletion at left end:

1. If `LEFT=Λ`, underflow.
2. `P ← LEFT`; `Y ← INFO(P)`.
3. `LEFT ← RLINK(P)`.
4. If `LEFT ≠ Λ`, set `LLINK(LEFT) ← Λ`; else set `RIGHT ← Λ`.
5. Return node `P` to `AVAIL`.

All operations are `O(1)`.

### 2 [08]
Why can’t a singly linked list support all deque operations efficiently?

Because deleting from the rear needs predecessor access. In singly linked lists, predecessor is not directly stored, so rear deletion requires scanning from front, which is `O(n)`. Therefore one-ended efficiency is possible, but fully efficient deque operations at both ends are not.

### 3 [10]
How can we justify that `CALLUP`, `CALLDOWN`, and `CALLCAR` are independent state variables in elevator simulation?

Each indicates a distinct physical event:

- hall up request,
- hall down request,
- in-car destination request.

Combinations like hall-up + hall-down at same floor are possible, and they do not imply in-car requests. Hence one bit/variable cannot encode all three without losing states; separate variables are required.

### 4 [10]
What changes if step E9 is deleted from the elevator algorithm?

E9 resolves decision timing when loading/unloading overlaps with direction choice. Removing it delays or misorders decisions, potentially causing direction choices that ignore newly established in-car calls and reducing responsiveness/correctness.

### 5 [12]
In the given scenario (new user appears while another wants opposite direction), why does direction choice still follow algorithm priority?

Because direction policy is not “last request wins”; it is governed by decision rules (current state + call priorities + neutral transitions). The algorithm first satisfies its deterministic selection criteria, then serves opposite-direction requests later.

### 6 [12]
How to change coroutine model to support directional indicator lights?

Maintain separate wait structures for up/down hall calls at each floor.

- Arrival/loading in upward mode consumes only upward hall queue.
- Arrival/loading in downward mode consumes only downward hall queue.
- Button lights are cleared only when corresponding directional request is actually serviced.

This preserves user-visible indicator semantics.

### 7 [10]
What error appears if a branch that rechecks waiting status is redirected to main cycle prematurely?

A waiting user can be dropped from the local handling path without being properly requeued or reconsidered, creating starvation/ghost-wait behavior (request exists physically but is no longer represented in active queue state).

### 8 [14]
Implement step E8 (decision-to-move transition) in structured pseudocode.

```text
E8:
if no pending calls relevant to current floor/direction:
    remain/return to HOLD state
else:
    choose next direction by priority policy
    set next instruction/state to movement coroutine
```

Key requirement: E8 must inspect hall calls and car calls consistently before committing direction.

### 9 [16]
Implement decision subroutine for elevator direction.

Inputs: current floor, current direction, pending call vectors.

Output: next state (`UP`, `DOWN`, `HOLD`).

Algorithm:

1. If no pending calls anywhere ⇒ `HOLD`.
2. If calls exist in current direction beyond current floor ⇒ continue same direction.
3. Else if calls exist opposite direction ⇒ reverse.
4. Else if call exists at current floor ⇒ load/unload and re-evaluate.

This deterministic policy avoids oscillation and ensures progress.

### 10 [18]
How should sparse-memory simulation updates be organized?

Use event/update lists instead of full-state rescans:

- table A: changed variables (dirty set),
- table B: dependent equations/actions.

Process only affected dependencies, then clear dirty markers. This yields sparse-update efficiency and avoids `O(N)` sweeps when only few values changed.

### 11 [12]
Why are doubly linked lists useful in this simulation context?

Because entities (users/events) may need removal from the middle of waiting lists (timeouts, cancellations, priority shifts). Doubly linked lists support middle deletion in `O(1)` with node pointer, while singly linked lists require predecessor search.

### 12 [14]
What invariant guarantees correctness of middle-node deletion in doubly linked lists?

For node `P` to be deleted:

- if `LLINK(P)=A`, then set `RLINK(A)=RLINK(P)`;
- if `RLINK(P)=B`, then set `LLINK(B)=LLINK(P)`;
- adjust boundary pointers if `P` was `LEFT` or `RIGHT`.

Invariant after deletion: for every remaining adjacent pair `(X,Y)`, `RLINK(X)=Y` iff `LLINK(Y)=X`.
