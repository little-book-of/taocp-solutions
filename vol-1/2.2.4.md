# 2.2.4 Circular Lists

We use circular singly linked lists with pointer `PTR` and link field `LINK(·)`. In a nonempty circular list, following links eventually returns to starting node.

### 1 [06]
If emptiness is represented by `PTR = LOC(PTR)` (self-reference), what changes?

This is a sentinel-style empty encoding. It works, but logic becomes less transparent than `PTR=Λ` because operations must distinguish “pointer to self” from ordinary nodes. Functional power is similar; clarity is usually worse.

### 2 [08]
Describe concatenation of two circular lists.

Let `PTR1`, `PTR2` reference representative nodes of two nonempty circles. Swap successors of these representatives:

- `a ← LINK(PTR1)`, `b ← LINK(PTR2)`
- `LINK(PTR1) ← b`, `LINK(PTR2) ← a`

This splices circles into one circle in `O(1)`.

### 3 [08]
What if we concatenate a list with itself?

If representatives are same node, swap is a no-op. If two representatives are different nodes in same circle, swapping their successors splits one circle into two circles; then losing one pointer may lose access to one component.

### 4 [10]
Implement stack behavior with representation (4).

Take `PTR` as top reference:

- Push: insert new node immediately after `PTR`, then set `PTR` to that node.
- Pop: remove `PTR` node (or its successor depending convention), update `PTR` accordingly.

Both operations `O(1)`.

### 5 [12]
Reverse a circular singly linked list.

Use three pointers (`prev`, `curr`, `next`) as in ordinary reversal, stopping when `curr` returns to start. After loop, reconnect old start to new successor and update `PTR` to old start (or chosen representative). Time `O(n)`, space `O(1)`.

### 6 [10]
Draw examples for polynomial circular-list representation (e.g., `xz-3` and `0`).

Representation principle:

- Each term node stores coefficient and exponent triple.
- Nodes are ordered by decreasing exponent tuple.
- Zero polynomial is represented by a single header/sentinel node linking to itself.

### 7 [10]
Why maintain exponent fields in decreasing order?

It enables single forward merge for addition:

- compare current terms once,
- advance one/both lists monotonically,
- no backtracking or repeated searches.

Hence linear-time merge behavior.

### 8 [10]
Why keep trailing pointer `Q1` in polynomial addition?

When a term cancels to zero, deletion of current node in `Q` needs predecessor access. `Q1` provides constant-time deletion without rescanning from list head.

### 9 [16]
Would addition/multiplication algorithms still work when pointers coincide (`P=Q`, `P=M`, etc.)?

- Addition with identical inputs is safe (effectively doubling coefficients in place) if algorithm avoids structural mutation hazards.
- Multiplication is unsafe when source and destination alias in ways that new inserted terms are later re-read as inputs; this can corrupt semantics or explode work.

So aliasing restrictions are needed for multiplication.

### 10 [12]
If exponent tuple is packed into one field instead of three, what changes?

Algorithms are structurally unchanged. Only comparison/extraction routines change (single-key compare instead of lexicographic on three fields).

### 11 [14]
Design a copy routine for circular list polynomials.

Traverse source once, allocate one node per source node, copy payload, wire links in same order, close final link back to first copied node. Complexity `O(n)`.

### 12 [14]
Compare copy time with add time when `poly(Q)=0`.

Copy is linear with one allocation+copy per node. Add with zero destination can still incur compare/control overhead and possible insertion path costs; same asymptotic `O(n)` but larger constant in typical implementations.

### 13 [12]
Design an erase routine.

Traverse circle, returning each node to `AVAIL`; finally set list pointer to empty representation (null or self-sentinel per convention). Time `O(n)`.

### 14 [10]
Design a `ZERO` routine.

If using explicit zero sentinel, free all non-sentinel nodes and leave single sentinel linking to itself. If using null-empty representation, erase whole list and set pointer null.

### 15 [18]
Design multiply routine for circular-list polynomials.

For each term `p` in first polynomial:

1. scale second polynomial by `p` (add exponents, multiply coefficients),
2. merge-add into accumulator polynomial.

Use ordered representation so merge remains linear in current sizes. Straightforward sparse multiplication.

### 16 [20]
Estimate running time of multiplication.

If first polynomial has `m` terms and second has `n`, naive sparse multiply forms `mn` candidate terms, then combines like terms via ordered merges. Typical complexity ranges from `Θ(mn)` (favorable combine) to higher constants depending on insertion/merge strategy; worst practical bound often near `O(mn + K)` where `K` is combine overhead.

### 17 [10]
What advantages do circular lists have over linear null-terminated lists?

- No special end null; traversal cycles naturally.
- Concatenation/splice can be `O(1)` with representative pointers.
- Some operations avoid boundary branches.

Tradeoff: termination conditions are trickier (must detect return to start).

### 18 [24]
Can we traverse efficiently in both directions with only one link field?

Yes, with XOR-linked technique: store `link = prev XOR next`. Given one neighbor, recover the other by XOR again. Traversal requires remembering previous node address. This saves space but complicates debugging, portability, and safety.
