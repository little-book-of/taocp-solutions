# 2.1 Introduction

Notation used in this section (same model as Knuth’s card example):

- `TOP` is a link (pointer) to the first card in the deck.
- `Λ` means “null link” (no card).
- Each card record has four relevant fields: `TAG`, `SUIT`, `RANK`, `NEXT`.
- `NEXT(X)` is the link field of card `X`, i.e., where the next card is.
- `CONTENTS(V)` means “the value currently stored in memory cell `V`.”
- `LOC(E)` means “the address/location where object `E` is stored.”

For beginners: think of a deck as a **chain of boxes**. Each box (card record) stores card data plus an arrow (`NEXT`) to the next box.

### 1 [04]
In the situation depicted in (3), determine:
1) `SUIT(NEXT(TOP))`, and
2) `NEXT(NEXT(NEXT(TOP)))`.

Step-by-step:

1. `TOP` points to the first card in the deck.
2. `NEXT(TOP)` means: follow one link from the first card. So this is the **second card**.
3. Looking at diagram (3), the second card’s suit is Spades.
4. In Knuth’s coding for suits, Spades is coded as `4`.
5. Now evaluate `NEXT(NEXT(NEXT(TOP)))`:
   - first `NEXT`: first card → second card;
   - second `NEXT`: second card → third card;
   - third `NEXT`: third card → beyond last card.
6. Beyond the last card, the link is null (`Λ`).

**Answer:**
- `SUIT(NEXT(TOP)) = Spades = 4`.
- `NEXT(NEXT(NEXT(TOP))) = Λ`.

### 2 [08]
We always have `CONTENTS(LOC(V)) = V`. Under what condition is `LOC(CONTENTS(V)) = V` also true?

Step-by-step:

1. `LOC(V)` is “where `V` lives.”
2. `CONTENTS(LOC(V))` then means “read the location where `V` is stored,” so of course you get `V`. That direction is always true.
3. The reverse direction starts with `CONTENTS(V)`: this is whatever value happens to be stored in memory cell `V`.
4. Then `LOC(CONTENTS(V))` asks for the location of that value.
5. For this to equal `V`, the value stored in cell `V` must be an object whose own location is exactly `V`.
6. That is a special/self-referential case, not generally true for arbitrary memory data.

**Answer:** `LOC(CONTENTS(V)) = V` only in the special case where the contents of address `V` denote an object located at `V`.

### 3 [12]
Give an algorithm to draw one card from the **top** of the deck.

Idea for beginners: “draw from top” means remove the first node of a linked list.

Step-by-step:

1. If `TOP = Λ`, there is no card to draw; return empty (`Λ`).
2. Otherwise store the current top card in a variable `CARD`.
3. Move `TOP` forward to the next card: `TOP ← NEXT(CARD)`.
4. Optional cleanup: set `NEXT(CARD) ← Λ` so the drawn card is detached from the list.
5. Return `CARD`.

Pseudocode:

```text
DRAW-TOP:
if TOP = Λ then return Λ
CARD ← TOP
TOP ← NEXT(CARD)
NEXT(CARD) ← Λ
return CARD
```

Why correct:
- If deck is empty, correctly returns empty.
- If deck is nonempty, exactly one card (the first) is removed, and the rest of the deck remains linked in original order.

Cost:
- Constant time `Θ(1)`.

### 4 [12]
Give an algorithm to place a card on the **bottom**, face down.

Suppose the new card is `X`.

Step-by-step:

1. Mark the card face down: `TAG(X) ← 1`.
2. Since this will be the new last card, force `NEXT(X) ← Λ`.
3. If deck is empty (`TOP = Λ`), set `TOP ← X` and stop.
4. Otherwise start from `TOP` and walk through links until you find last card `P` with `NEXT(P)=Λ`.
5. Link old last card to new one: `NEXT(P) ← X`.

Pseudocode:

```text
PUT-ON-BOTTOM(X):
TAG(X) ← 1
NEXT(X) ← Λ
if TOP = Λ then
    TOP ← X
    return
P ← TOP
while NEXT(P) ≠ Λ do
    P ← NEXT(P)
NEXT(P) ← X
```

Why correct:
- Empty deck case creates a one-card deck.
- Nonempty deck case appends after current tail, so old order is preserved and `X` becomes new tail.

Cost:
- `Θ(n)` because of tail search (unless a separate tail pointer is maintained).

### 5 [15]
Give an algorithm to draw one card from the **bottom**.

For beginners: removing the last node of a singly linked list requires finding the **second-last** node.

Step-by-step:

1. If `TOP = Λ`, deck is empty; return `Λ`.
2. If `NEXT(TOP)=Λ`, there is exactly one card:
   - `CARD ← TOP`
   - `TOP ← Λ`
   - return `CARD`.
3. Otherwise use two pointers:
   - `PREV` starts at first card,
   - `CUR` starts at second card.
4. Move both until `CUR` is the last node:
   - while `NEXT(CUR) ≠ Λ`:
     - `PREV ← CUR`
     - `CUR ← NEXT(CUR)`.
5. Now `CUR` is bottom card and `PREV` is second-last.
6. Remove bottom card: `NEXT(PREV) ← Λ`.
7. Return `CUR`.

Pseudocode:

```text
DRAW-BOTTOM:
if TOP = Λ then return Λ
if NEXT(TOP) = Λ then
    CARD ← TOP
    TOP ← Λ
    return CARD
PREV ← TOP
CUR ← NEXT(TOP)
while NEXT(CUR) ≠ Λ do
    PREV ← CUR
    CUR ← NEXT(CUR)
NEXT(PREV) ← Λ
return CUR
```

Why correct:
- Handles empty and one-card decks explicitly.
- In longer decks, loop invariant is: `PREV` is immediately before `CUR` in the chain.
- At loop end, `CUR` is tail; cutting `NEXT(PREV)` removes exactly that tail.

Cost:
- `Θ(n)`.

### 6 [10]
Why are `SUIT(LOC(CARD))` and `SUIT(TOP)` meaningful, while `SUIT(CONTENTS(CARD))` is generally not?

Step-by-step:

1. The operator `SUIT(·)` expects as input a **card location/address**.
2. `LOC(CARD)` is a location, so `SUIT(LOC(CARD))` is type-correct.
3. `TOP` is also a card location (if non-null), so `SUIT(TOP)` is type-correct.
4. `CONTENTS(CARD)` is just the raw value stored in memory cell `CARD`.
5. A raw stored value is not automatically a valid card address.
6. Therefore `SUIT(CONTENTS(CARD))` is generally an address/type mismatch.

**Answer:** `SUIT` needs a card address; `LOC(CARD)`/`TOP` provide one, but `CONTENTS(CARD)` usually does not.

### 7 [12]
In MIX-style linked access, explain why we first load the pointer into an index register, then use indexed access like `0,1(NEXT)`.

Step-by-step:

1. A link field (`NEXT`) stores an **address** of another card.
2. To access fields of that next card, you must dereference that address.
3. In MIX assembly style, that is commonly done by loading the address into an index register (say `I1`).
4. Then `0,1(NEXT)` means: base `0` + index `I1`, field `NEXT`.
5. This computes “field `NEXT` of the card whose address is in `I1`.”
6. If you skip this indirection pattern, you may read from wrong addresses.

**Answer:** this two-step method is the proper explicit indirection required by linked structures in MIX.

### 8 [16]
Give an algorithm to count how many cards are in the deck.

Step-by-step:

1. Initialize count: `N ← 0`.
2. Initialize pointer `P ← TOP`.
3. While `P` is not null:
   - increment `N` by 1,
   - move forward: `P ← NEXT(P)`.
4. When loop ends, `N` is total number of cards.

Pseudocode:

```text
COUNT-DECK:
N ← 0
P ← TOP
while P ≠ Λ do
    N ← N + 1
    P ← NEXT(P)
return N
```

Why correct:
- Each loop iteration counts exactly one node.
- Traversal ends exactly when no nodes remain.
- So returned `N` equals number of reachable cards from `TOP`.

Cost:
- Time `Θ(n)`, extra space `Θ(1)`.

### 9 [23]
Write a MIX program that prints alphabetic card names (example: `TEN OF HEARTS`).

A beginner-friendly design is to separate the task into three parts:

1. **Traversal** of the linked list from `TOP`.
2. **Decoding** numeric `RANK` and `SUIT` via tables.
3. **Output formatting** as `RANK-NAME + " OF " + SUIT-NAME`.

Detailed plan:

1. Set traversal pointer `P ← TOP`.
2. If `P = Λ`, stop.
3. Read fields from card `P`:
   - `t ← TAG(P)`
   - `r ← RANK(P)`
   - `s ← SUIT(P)`
4. Use lookup tables:
   - `rank_name[r]` from 13-entry rank table (`ACE` ... `KING`),
   - `suit_name[s]` from 4-entry suit table (`CLUBS` ... `SPADES`).
5. Emit the text for one card.
6. If `t=1` (face down), apply the text convention required by the exercise/text (for example, wrapping marker).
7. Print newline.
8. Advance `P ← NEXT(P)` and repeat.

Program skeleton:

```text
PRINT-NAMES:
  P ← TOP
LOOP:
  if P = Λ then return
  t ← TAG(P)
  r ← RANK(P)
  s ← SUIT(P)
  emit rank_name[r]
  emit " OF "
  emit suit_name[s]
  if t = 1 then decorate_as_facedown
  newline
  P ← NEXT(P)
  jump LOOP
```

Why this works:

- Traversal is standard linked-list traversal, so every reachable card is visited once.
- Table lookup gives deterministic textual names from numeric codes.
- One output line is generated per visited card.
- Therefore the program prints exactly the alphabetic name for each card in deck order.
