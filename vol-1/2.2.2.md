# 2.2.2 Sequential Allocation

We use Knuth’s standard array-based notation for linear lists.

- For a queue in array `Q[1..M]`, let `F` be the front index and `R` the next insertion position.
- In circular form, indices wrap: if index exceeds `M`, continue at `1`.
- For shared-memory dynamic schemes, queue `k` occupies a segment with lower bound `L_k` and upper bound `U_k`, adjusted during repacking.

### 1 [06]
In the queue operations from the text, how many items can be in the queue at one time without overflow?

In Knuth’s circular queue convention, one slot is intentionally left empty so that `F = R` can mean “empty” unambiguously. Therefore the maximum number of stored items is `M-1`.

**Answer:** `M-1`.

### 2 [10]
Add operations for a general deque in sequential (circular-array) form.

Let deque array be `X[1..M]`, with front index `F` at first occupied item and rear index `R` at one position beyond the last occupied item.

- Empty iff `F = R`.
- Full iff `next(R) = F`, where `next(i)= i+1` or `1` if `i=M`.

Operations:

- `insertRear(Y)`: if full, overflow; else `X[R] ← Y`, `R ← next(R)`.
- `insertFront(Y)`: if full, overflow; else `F ← prev(F)`, `X[F] ← Y`.
- `deleteFront()`: if empty, underflow; else `Y ← X[F]`, `F ← next(F)`, return `Y`.
- `deleteRear()`: if empty, underflow; else `R ← prev(R)`, `Y ← X[R]`, return `Y`.

All run in `O(1)`.

### 3 [08]
Give compact addressing formulas (with indirect addressing available) for relative access and table-of-tables access.

Let `BASE` be address of current array base and `I` desired offset. With indexed/indirect mechanisms, relative lookup can be reduced to one effective-address computation:

- `A ← CONTENTS(BASE + I)`

For table-of-tables: if `B[j]` stores base of table `j`, then element `i` in table `j` is:

- `A ← CONTENTS(B[j] + i)`

So one level of indirection chooses table base; ordinary indexing chooses element.

### 4 [12]
Evaluate special indirect-addressing patterns and explain behavior.

Key idea: repeated indirection can create chains (or loops). If an address points (directly or through modifiers) back into the same chain, effective-address resolution can become nonterminating unless hardware or ISA rules forbid it.

Practical conclusions:

1. Two-hop link access (`link(link(x))`) is expressible by one extra dereference level.
2. Three-or-more hops require either repeated instructions or richer addressing modifiers.
3. If self-referential address words are allowed, infinite effective-address loops are possible.

So any realistic design needs either:
- restricted modifier combinations, or
- explicit hop limits/traps.

### 5 [18]
Why does unrestricted nested indirect addressing (notably a `7:7`-style form) complicate hardware?

Because effective-address computation stops being a simple linear pipeline and becomes a tree/stack process:

- one indirection may expose another modifier,
- each branch may require deferred index additions,
- hardware must remember pending contexts while descending/returning.

Hence additional state (stack-like control, cycle detection, trap logic) is required. Restricting modifier combinations keeps address calculation deterministic and cheap.

### 6 [10]
For a bounded sequential queue, classify each operation as legal or overflow/underflow.

Decision rules:

- Insert when full ⇒ overflow.
- Delete when empty ⇒ underflow.
- Otherwise operation is legal.

Applying these criteria to any specific sequence gives the required labels.

### 7 [10]
In repacking-based sequential allocation, can the increment parameter be zero?

No. If increment were zero, a structure that triggered repacking would receive no additional room, immediately risking repeated repacks on the next insertion. Standard amortized analyses require positive growth on expansion events.

**Answer:** increment must be at least 1.

### 8 [14]
How should repacking handle circular queues?

Treat each circular queue as one logical sequence that may be physically split into two blocks:

- block A: `F..M`
- block B: `1..R-1`

During repack:

1. compute new segment bounds for each queue;
2. copy logical order contiguously into the new segment;
3. reset `F` to segment start and `R` to one past last copied element.

This preserves queue order and restores contiguous representation after each repack.

### 9 [20]
Analyze expected relocation/work for the sequential-allocation policy.

Let total occupied cells be `N` and total slack from expansion policy be proportional to `N` (e.g., geometric or additive-by-load policy). Then:

- occasional repacks cost `Θ(N)` moves,
- but occur only after `Θ(N)` aggregate growth.

Therefore amortized move cost per successful insertion is `O(1)` under standard growth assumptions.

### 10 [20]
Continue the analysis for multiple queues sharing one array.

With `q` queues and total occupancy `N`, if repack redistributes free space proportionally to recent growth rates, then no single hot queue immediately re-triggers repack unless its growth significantly exceeds forecast. This stabilizes behavior and keeps long-run amortized insertion cost bounded by a constant independent of `N`.

### 11 [20]
Give a worst-case upper bound per insertion under repacking.

Worst case for one insertion is when it triggers a full repack: `Θ(N)` time, since potentially all occupied cells move.

So:

- worst-case single operation: `Θ(N)`;
- amortized over long sequence: `O(1)` (from Exercises 9–10 assumptions).

### 12 [20]
How does deletion affect amortized cost in the same model?

Deletions usually cost `O(1)` directly. They also create slack, which delays future repacks. So deletions do not worsen amortized insertion bounds; if anything, they improve average spacing and reduce repack frequency.

### 13 [22]
Discuss policy choice: equal slack vs growth-proportional slack among queues.

- Equal slack is simple/fair in static use, but poor if one queue is much hotter.
- Growth-proportional slack better tracks workload and reduces repeated repacks by active queues.

Hence adaptive proportional slack is usually superior for performance.

### 14 [22]
What invariant should be preserved during any repack?

For each queue `k`:

1. logical order of elements is unchanged;
2. element count is unchanged;
3. front and rear pointers identify same logical ends as before;
4. queues occupy disjoint valid segments.

Proving these invariants is enough to show repack correctness.

### 15 [22]
How can we verify repack correctness mechanically?

Use a two-part check:

- **local check** per queue: compare pre/post sequence snapshots.
- **global check**: ensure all occupied cells belong to exactly one queue and pointers are in range.

This can be implemented as linear scans over old/new arrays.

### 16 [24]
Can two queues (or a queue and a stack) safely grow toward each other in one array without repacking?

Not reliably. Even with low occupancy, adversarial insert/delete patterns can make boundaries collide early (fragmentation by motion), causing overflow despite enough total free space elsewhere.

So direct “grow toward each other forever” is unsafe without relocation strategy.

### 17 [24]
Why does an initial balanced placement help?

If queues begin evenly spaced, first collision time is delayed under symmetric workloads. More importantly, after each repack, near-proportional spacing reduces immediate conflict probability. This improves practical throughput even though adversarial sequences can still force early repacks.

### 18 [24]
Explain the time behavior of adaptive repacking policies.

As structures grow, allocated slack also grows; therefore repacks become less frequent relative to operation count. This is the same phenomenon as dynamic arrays/hash tables: expensive global reorganizations happen, but their frequency drops enough to keep amortized cost low.

### 19 [26]
Adapt stack/queue index conventions to arrays whose lower bound is `-1` (instead of `1`).

For stack in `S[-1..M-1]`:

- empty top sentinel can be `T = -2` (or choose convention `T=-1` with first push incrementing first).
- push: check overflow against upper bound, then increment/store.
- pop: check underflow against sentinel, then read/decrement.

For circular queue in `Q[-1..M-1]`:

- define `next(i) = -1` when `i = M-1`, else `i+1`.
- keep same empty/full logic (`F=R` empty; `next(R)=F` full if using one-slot-empty convention).

Only boundary arithmetic changes; abstract queue/stack semantics are unchanged.
