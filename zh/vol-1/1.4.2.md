## 1 [10]
A *coroutine* is symmetric: two (or more) routines repeatedly transfer control to each other, each time resuming exactly where it last left off (state must persist across transfers). This is unlike a subroutine, where control flows down and then returns once.&#x20;

Why “short and simple” is elusive

1. One transfer ⇒ not really a coroutine.
   If A calls B only once (and B never resumes A mid-run), the interaction is observationally identical to an ordinary subroutine call. So the “smallest” control exchange doesn’t demonstrate anything essentially coroutine-specific. To *show* coroutine behavior, you need multiple alternations (A↔B↔A↔B…), which already lengthens the example.

2. State must be preserved between activations.
   Real coroutine examples must keep local state alive across yields. On a bare machine (like MIX in the text) that means explicit linkage and register saves/restores—extra instructions that swell even toy programs. Knuth’s own coroutine linkage shows saving/restoring registers and jump handshakes, which are inherently not “tiny.”&#x20;

3. Nontrivial data flow is needed to be meaningful.
   To motivate *why* coroutines help, you typically show a producer/consumer, scanner/parser, or “turn an n-pass pipeline into a one-pass process” example. But as soon as you pass realistic data (buffers, termination conditions, etc.), the code grows beyond a few lines. (Knuth’s figure contrasting a four-pass pipeline with a one-pass chain of coroutines illustrates the typical use case.)&#x20;

4. Toy examples are either contrived or misleading.
   Extremely short “ping-pong” snippets (e.g., print A/B alternately) either hide the crucial save/restore machinery or rely on features (like a built-in `yield`) that the reader hasn’t met yet, making them pedagogically suspect on a machine-language canvas.

Minimal illustration (conceptual)

* Subroutine pattern: `main → sub() → return` (one descent, one return).
* Coroutine pattern: `A → B → A → B → …` (multiple *resumptions*), plus explicit context preservation. Without at least two transfers, A↔B collapses to subroutine semantics and doesn’t illuminate coroutines.

To genuinely demonstrate coroutines you need (i) more than one control transfer, (ii) preserved execution context, and (iii) meaningful data exchange. Each of these adds lines and machinery, so examples that are both *short* and *truly illustrative* are rare; once they’re short enough, they stop being distinctly coroutine examples and look like plain subroutines instead.

1. Initialization still sets `rI6 ← 0` and `rX ← 0` (lines 58–59), so `IN1` will read the first input word if needed and isolate the next character into `rA`.&#x20;

2. At the end of its step, `IN` yields to `OUT` via the coroutine linkage (`JMP OUT`). Control lands at the `OUT` linkage stub, which first saves IN’s resume point and then transfers to `OUT1`.&#x20;

3. `OUT1` begins by preparing a fresh output card/buffer: it sets up indices and executes a `MOVE` that blanks the output area (“Start new output card… Set output area to blanks”), then it asks `IN` for the “next translated character” by jumping back via the linkage (`JMP IN`). Those prologue instructions overwrite or otherwise invalidate the character that `IN` had already placed in `rA`. Net effect: the very first character fetched by `IN` is lost before `OUT` stores anything.&#x20;

This is exactly why the text starts with `JMP OUT1`: the OUT coroutine’s prologue should run before the first request to `IN`, so that the buffer is ready and the first character fetched by `IN` is immediately consumed instead of being clobbered by OUT’s setup. The official answer summarizes this as “the first character found by IN would be lost,” and notes that lines 58–59 were chosen to initialize `IN`; starting with `IN` would require compensating initialization for `OUT`.&#x20;

How to make “start with IN” work (one correct fix)

If you truly want to begin at `IN1`, adjust the first handoff into `OUT` so OUT’s prologue does not run after `IN` has already put a character in `rA`:

* Pre-initialize OUT’s state that the prologue would otherwise need. Concretely, set `rA ← −16` so the initial `MOVE` count is already in place, and clear the output buffer if it isn’t known to be blank.
* Change the first transfer from `IN` to `OUT` to jump directly to the code that stores `rA` into the buffer (the code around line 39) rather than to `OUT1` (the prologue). In the book’s notation: modify the first `OUTX` jump target so it goes to the “store character” logic rather than the “start new card” logic. With that, the very first character produced by `IN` will be preserved and packed before any further OUT setup runs.&#x20;

Bottom line

* If you simply replace `JMP OUT1` with `JMP IN1`, the first input character disappears.
* To start with `IN` safely, you must (i) pre-initialize OUT’s prologue effects and (ii) route the first transfer from `IN` straight into OUT’s “store character” code rather than OUT’s prologue.&#x20;

## 4 [20]
What “linkage (1)” is doing in MIX

Knuth’s linkage (labels rearranged slightly for clarity) is the tiny handshake at the end of his program:

```
60  JMP OUT1        ; start with OUT  (exercise 2)
61  OUT  STJ INX    ; store our resume point "OUTX" into IN's jump word
62  OUTX JMP OUT1   ; jump to OUT's entry (will be patched by IN)
63  IN   STJ OUTX   ; ditto, in the other direction
64  INX  JMP IN1    ; jump to IN's entry (will be patched by OUT)
```

Each coroutine owns one *indirect jump word* (`INX`, `OUTX`). When a coroutine yields, it stores the address of its own resume label (e.g., `OUTX`) into the *other* coroutine’s jump word, then jumps to the other’s entry (`IN1` ⇄ `OUT1`). The first time, `INX` and `OUTX` contain the entry addresses; thereafter they contain the saved resume addresses. (Knuth later shows a variant “(4)” that also saves/restores register A, but the basic idea above is “(1)”.)&#x20;

Below are two concrete, modern implementations that are directly analogous but avoid self-modifying code by using *indirect branches through writable data*.

x86-64 (NASM/Intel syntax; position-independent)

```nasm
; Assemble with: nasm -felf64 coro.asm && ld -o coro coro.o
default rel

section .data
INX:   dq IN1        ; IN’s current continuation (entry at first)
OUTX:  dq OUT1       ; OUT’s current continuation (entry at first)

section .text
global _start

_start:
    jmp OUT1         ; Like line 60: begin in OUT

; --- IN coroutine -------------------------------------------------

; Entry for IN: always tail-jump to whatever OUT stored into INX
IN1:
    jmp qword [INX]

; Somewhere inside IN, to yield to OUT do:
;   1) store our resume address into OUTX
;   2) jump to OUT’s entry
IN_yield_to_OUT:
    lea  rax, [rel IN_resume]  ; “return address” for IN
    mov  [OUTX], rax
    jmp  OUT1
IN_resume:
    ; ... execution continues here when OUT yields back ...

; --- OUT coroutine ------------------------------------------------

; Entry for OUT: always tail-jump to whatever IN stored into OUTX
OUT1:
    jmp qword [OUTX]

; To yield back to IN:
OUT_yield_to_IN:
    lea  rax, [rel OUT_resume] ; “return address” for OUT
    mov  [INX], rax
    jmp  IN1
OUT_resume:
    ; ... execution continues here when IN yields back ...
```

Why this is exactly like (1):

* `IN1: jmp [INX]` and `OUT1: jmp [OUTX]` play the role of the “indirect jump words” (`INX JMP IN1`, `OUTX JMP OUT1`).
* `IN_yield_to_OUT` stores `IN_resume` into `OUTX` before jumping to `OUT1`; `OUT_yield_to_IN` stores `OUT_resume` into `INX` before jumping to `IN1`.
* No stacks or OS help are required; you’re literally passing continuations in memory, just as MIX does by patching a jump word.

If you need register/save semantics like Knuth’s variant (4), push/pop (or move to callee-saved registers/stack slots) around the `mov`/`jmp` pair, corresponding to `STA HOLDA` … `LDA HOLDA` in his text.&#x20;

Portable C (GCC/Clang) with “labels as values”

This uses the compiler extension for *computed goto*. It mirrors MIX even more transparently:

```c
// gcc -std=gnu11 coro.c -o coro
#include <stdio.h>

static void *INX, *OUTX;

int main(void) {
  INX  = &&IN1;      // initial entries
  OUTX = &&OUT1;

  goto OUT1;         // start with OUT (like line 60)

IN1:
  goto *INX;         // always jump to IN’s current continuation

  // ... IN’s code ...
IN_yield_to_OUT:
  OUTX = &&IN_resume;  // store our resume label into OUT’s jump slot
  goto OUT1;           // jump to OUT’s entry
IN_resume:
  // ... resumes here ...

OUT1:
  goto *OUTX;        // always jump to OUT’s current continuation

  // ... OUT’s code ...
OUT_yield_to_IN:
  INX = &&OUT_resume;  // store our resume label into IN’s jump slot
  goto IN1;            // jump to IN’s entry
OUT_resume:
  // ... resumes here ...
}
```

* `INX`/`OUTX` are the indirect jump words.
* `goto *INX` / `goto *OUTX` replace the “`JMP IN1`/`JMP OUT1` words patched by STJ”.
* Assigning `&&label` is the exact analogue of `STJ otherX`.

This version is often the simplest way to prototype Knuth’s linkage today, and it avoids any system-dependent assembly.

## 5 [15]

Whenever OUT executes `JMP IN`, rA in OUT must be exactly what it was when control returns to the next OUT instruction; likewise for IN when it executes `JMP OUT`. So each coroutine must (1) save its own rA at exit, and (2) restore its own rA at re-entry, without constraining the other coroutine’s use of rA while it runs.

Knuth already shows the one-sided variant (“if OUT would destroy A, save/restore A around the transfer”), right above Fig. 22; we just symmetrize that pattern.
We splice this linkage where the book places the standard two-coroutine linkage (lines 57–65), replacing those four short lines with the version below.

MIX code (two-sided A preservation)

```mix
* --- Linkage data (two save slots, one per coroutine) ---
HOLDIN  CON  0        * last-saved rA for IN
HOLDOUT CON  0        * last-saved rA for OUT

* --- Initialization and linkage ---
START   ENT6 0        * as in text: init rI6 for NEXTCHAR
        ENTX 0        * as in text: init rX for NEXTCHAR
        JMP  OUT1     * start with OUT (exercise 2 discusses starting with IN)

OUT     STJ  INX      * standard coroutine bookkeeping (return for IN)
        STA  HOLDIN   * save caller's A (leaving IN -> coming here)
        LDA  HOLDOUT  * restore OUT's own A for re-entry
OUTX    JMP  OUT1     * enter OUT proper

IN      STJ  OUTX     * standard coroutine bookkeeping (return for OUT)
        STA  HOLDOUT  * save caller's A (leaving OUT -> coming here)
        LDA  HOLDIN   * restore IN's own A for re-entry
INX     JMP  IN1      * enter IN proper

        END  START
```

Why this works

* When OUT → IN (`JMP IN` inside OUT): control lands at label `IN`. We immediately

  1. save the caller’s A (`STA HOLDOUT`), then
  2. restore IN’s previously saved A (`LDA HOLDIN`),
     and jump to `IN1`. While IN runs, it may freely clobber rA. When IN later yields back to OUT (`JMP OUT`), label `OUT` restores OUT’s A from `HOLDOUT` before resuming at `OUT1`. Thus rA in OUT is unchanged across the yield/return pair.

* Symmetrically, when IN → OUT (`JMP OUT` inside IN), label `OUT` saves IN’s rA to `HOLDIN`, restores OUT’s rA from `HOLDOUT`, and resumes OUT. When OUT yields back, label `IN` reloads `HOLDIN`. So rA in IN is likewise unchanged across its yield/return.

This is exactly the two-sided generalization of the one-sided example on p. 198 (“if OUT would destroy rA, save at IN and restore at OUT”), but now with two save cells so we can save the caller’s A and simultaneously restore the callee’s A at each transfer.

## 6 [22]
Here’s a clean coroutine linkage for three MIX coroutines $A,B,C$ so that any one can transfer to either of the other two, and on activation each resumes exactly where it last left off. (This is the Exercise 6 setup at the end of §1.4.2.)&#x20;

First, recall the 2-coroutine pattern (called “(1)” in the text): each coroutine has a *resume cell* holding a `JMP` whose address field gets patched by the other coroutine via `STJ`. Then you enter a coroutine by *executing* that stored jump (i.e., `JMP OUTX` executes the word at `OUTX`).

We generalize that idea by giving each coroutine its own resume cell (`AX`, `BX`, `CX`) and by providing two entry labels per coroutine—one entry for each possible caller—so we know which caller’s resume cell to update.

Linkage (MIX)

```asm
* Resume cells (initial jumps to first-time entries A1, B1, C1)
AX      JMP A1
BX      JMP B1
CX      JMP C1

* Entries to B
AB      STJ AX       * called B from A: remember A’s return point in AX
        JMP BX       * now execute B’s stored jump (first time → B1)

CB      STJ CX       * called B from C: remember C’s return point in CX
        JMP BX

* Entries to C
AC      STJ AX       * called C from A: remember A’s return point
        JMP CX

BC      STJ BX       * called C from B: remember B’s return point
        JMP CX

* Entries to A
BA      STJ BX       * called A from B: remember B’s return point
        JMP AX

CA      STJ CX       * called A from C: remember C’s return point
        JMP AX
```

How you use it

* From inside A, jump to `AB` to activate B, or to `AC` to activate C.
* From inside B, jump to `BA` (activate A) or `BC` (activate C).
* From inside C, jump to `CA` (A) or `CB` (B).

Why it works (invariants)

* Each coroutine Q has a resume cell QX that always contains a `JMP <where Q should resume>`.
* When caller P transfers to callee Q, control arrives at entry label QP, which executes `STJ PX`, storing the caller’s *return address* (the instruction after the `JMP QP`) into PX. Thus the caller’s resume cell is freshly patched to “where to continue when we next execute `JMP PX`.”
* The entry then `JMP`s to QX, *executing* Q’s stored `JMP`—first time it jumps to `Q1`; thereafter it jumps to Q’s last suspended point.

You can verify a round-trip: A → `JMP AB` → `AB: STJ AX; JMP BX` (B begins); later B → `JMP BA` → `BA: STJ BX; JMP AX` which executes the patched word at AX and resumes A precisely after its `JMP AB`.

## 7 [30]

What the original program did (we must invert it)

* Input format (2): a stream of characters ending with a period “.”; if a digit `n∈{0,…,9}` appears, it means “repeat the next character `n+1` times,” even if that next character is itself a digit. Nondigits simply denote themselves.&#x20;
* Output format (3): the fully expanded character stream, printed in groups of three separated by blanks, 16 groups per card; processing stops at the period.&#x20;

Our task is the inverse: read the grouped stream (3), ignore blanks, stop at the first period, and re-encode it into the compact code (2) with the fewest characters (e.g., never emit the redundant `0Z`, just `Z`).&#x20;

Minimal re-encoding strategy (correctness & optimality)

Let’s call the flat stream (with blanks removed) `S`, ending at the first `.`. We want the shortest string over the alphabet `Σ ∪ {0,…,9}` whose expansion (by “digit→repeat next char `n+1` times”) equals `S`.

Process `S` run by run: for each maximal block of identical characters `c` of length `L`, output the shortest code for that block, then continue.

* If `c` is a nondigit (letter, punctuation, etc.):

  * We may either write the character itself once (cost 1) or use a “digit + c” pair (cost 2) to say multiple copies.
  * Optimal encoding is greedy by tens:

    * Let `q = ⌊L/10⌋` and `r = L mod 10`.
    * Output `q` pairs `9c` (each pair contributes 10 copies).
    * For the remainder:

      * If `r = 0`: done.
      * If `r = 1`: just output `c` (better than `0c`).
      * If `2 ≤ r ≤ 10`: output `(r−1)c`.
* If `c` is a digit (one of `0..9` appearing as data), we cannot write it bare (because bare digits in (2) are operators). We must always use a pair:

  * Again `q = ⌊L/10⌋`, `r = L mod 10`.
  * Output `q` times `9c`; if `r>0`, output `(r−1)c`.

Why this greedy scheme is optimal:

* Each “digit+char” token has fixed cost 2 and yields between 1 and 10 copies (specifically `n+1` copies). For nondigit characters we also have a special 1-copy option of cost 1 (just the character). This is a canonical coin system: “coins” of values `1..10` with costs `2` each, except a free single-copy option of cost 1 for nondigits. Greedy by 10s, then one optimal remainder, is optimal under such costs; any non-greedy split that uses more tokens (or replaces a single bare `c` by `0c`) cannot be shorter.

This guarantees “the zero before the Z … would not really be produced” (we would output just `Z`) since `0Z` (cost 2) is strictly worse than `Z` (cost 1).&#x20;

MIX solution architecture

We mirror Knuth’s i/o organization:

1. NEXTCHAR: reuse the input subroutine from the text that returns the next nonblank character from card reader into `rA`; it handles card boundaries and blanks. (Exactly as in the example: read card, scan words, extract next character.)&#x20;

2. PUTCH: an output subroutine that appends one character to an 80-col output buffer; when full, punch the card and clear the buffer (similar to OUT’s buffering, but without inserting blanks every three chars). The last card is punched when we see `.`.

3. MAIN (reverse encoder):

   * Read first nonblank char `x` via `NEXTCHAR`.
   * If `x='.'`: output `.` and finish.
   * Set `CUR=x`, `CNT=1`.
   * Loop: get next `x` via `NEXTCHAR`.

     * If `x='.'`: flush run `(CUR, CNT)` using EmitRun (below), then output `.` and finish.
     * Else if `x==CUR`: `CNT++`.
     * Else: flush `(CUR, CNT)`, then `CUR=x`, `CNT=1`.
   * EmitRun(CUR, CNT) implements the minimal strategy above, using helper predicates “is digit?” (compare `rA` with codes 30..39) and a small loop to emit `9CUR` `q` times; remainder as per rules.
