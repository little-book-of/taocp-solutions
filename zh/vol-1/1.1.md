### 1 [10]
Minimal in-place algorithm (5 replacements)

```
t ← a
a ← b
b ← c
c ← d
d ← t
```

Walkthrough.
Let the initial values be `a0, b0, c0, d0`.

1. `t ← a` → `t = a0`
2. `a ← b` → `a = b0`
3. `b ← c` → `b = c0`
4. `c ← d` → `c = d0`
5. `d ← t` → `d = a0`

Final state is `(a, b, c, d) = (b0, c0, d0, a0)`, exactly the requested left rotation.

Why 5 replacements is optimal

1. All four variables change. In a 4-cycle `(a b c d)`, none of `a, b, c, d` keeps its original value, so each of the four must be assigned at least once → ≥ 4 replacements.

2. You must preserve one original value while overwriting.
   As soon as you assign, say, `a ← b`, the original `a0` is lost unless it was saved. But the final `d` must become `a0`. Therefore at least one extra “save” is necessary (e.g., `t ← a`) → ≥ 4 + 1 = 5 replacements.

Thus 5 is a lower bound, and the procedure above achieves it, so it’s minimal.

Generalization

For an n-cycle rotation `(x1 x2 … xn)`, the same argument shows a lower bound of n + 1 replacements (n to set each variable once, plus 1 to save a value), and the standard scheme

```
t ← x1;  x1 ← x2;  x2 ← x3;  …;  xn−1 ← xn;  xn ← t
```

meets that bound.

### 2 [15]
Consider any time we return to E1 after completing an earlier pass through E1/E2/E3.

* On the previous pass through E1 we computed a remainder $r$ with $0\le r<n$. If $r=0$ we would have halted in E2 and would not reach E1 again. Hence whenever we *do* reach E1 again we must have $r\neq 0$, i.e., $0<r<n$.&#x20;
* Then E3 set $m\leftarrow n$ and $n\leftarrow r$. Therefore, at the start of the next E1, we have

  $$
  m_{\text{new}} = n_{\text{old}} \quad\text{and}\quad n_{\text{new}} = r_{\text{old}}\,,
  $$

  with $0<r_{\text{old}}<n_{\text{old}}$. Hence $m_{\text{new}} = n_{\text{old}} > r_{\text{old}} = n_{\text{new}}$, i.e., $m>n$ strictly.

This shows that from the second visit to E1 onward, the strict inequality $m>n$ always holds.

For the first visit to E1, no such guarantee exists: the inputs might have $m\le n$ (indeed $m<n$ is allowed), so the statement allows the exception “except possibly the first time.” (If one prepends the optional “ensure $m\ge n$” swap step suggested in the text, the property would also hold the first time.)

### 3 [20]
Algorithm F (Euclid, without “trivial replacements”)

Goal. Given positive integers `m` and `n`, compute `gcd(m, n)`.

- F1. \[Remainder m / n] Divide `m` by `n` and replace `m` with the remainder. (So `0 ≤ m < n`.)
- F2. \[Is it zero?] If `m = 0`, halt with answer `n`.
- F3. \[Remainder n / m] Divide `n` by `m` and replace `n` with the remainder. (So `0 ≤ n < m`.)
- F4. \[Is it zero?] If `n = 0`, halt with answer `m`; otherwise go to F1.

This version never does “`m ← n, n ← r`”; we overwrite the dividend directly with the remainder each time and alternate roles. 

Why this is correct

1) Invariant (gcd is preserved)

At every step,
`gcd(m, n) = gcd(n, m mod n)` and `gcd(m, n) = gcd(m, n mod m)`.
That’s the fundamental property behind Euclid’s method; F1/F3 are exactly these replacements, so the gcd never changes. (This is the same reasoning used to justify Algorithm E; see §1.1. ([PagePlace][1]))

2) Termination

Each remainder is strictly smaller than its divisor. After F1 we have `m < n`; after F3 we have `n < m`. Thus the maximum of the pair `{m, n}` strictly decreases on every pass through F1–F4. Since the maximum is a non-negative integer, it can decrease only finitely many times, so the algorithm halts.

3) Upon halting, the answer is correct

If we halt at F2, then `m = 0`, so `gcd(m, n) = gcd(0, n) = n`.
If we halt at F4, then `n = 0`, so `gcd(m, n) = m`.
Combined with the invariant, this equals the original `gcd`.


Worked example

Compute `gcd(2166, 6099)` using F.

- F1: `m ← 2166 mod 6099 = 2166` (since 2166 < 6099)
  F2: `m ≠ 0` → continue
- F3: `n ← 6099 mod 2166 = 1767`
  F4: `n ≠ 0` → loop
- F1: `m ← 2166 mod 1767 = 399`
  F2: `m ≠ 0`
- F3: `n ← 1767 mod 399 = 171`
  F4: `n ≠ 0`
- F1: `m ← 399 mod 171 = 57`
  F2: `m ≠ 0`
- F3: `n ← 171 mod 57 = 0`
  F4: `n = 0` → halt with answer `m = 57`.
  
  
### 4 [16]
We repeatedly divide and keep the remainder; the gcd is the last non-zero remainder.

1. $6099 = 2\cdot 2166 + 1767$ → $r_1=1767$
2. $2166 = 1\cdot 1767 + 399$   → $r_2=399$
3. $1767 = 4\cdot 399 + 171$    → $r_3=171$
4. $399 = 2\cdot 171 + 57$      → $r_4=57$
5. $171 = 3\cdot 57 + 0$

The last non-zero remainder is $57$, so

$$
\gcd(2166,6099)=57.
$$

Quick checks

- Divisibility: $2166/57=38$ and $6099/57=107$ are integers.
- No larger common factor: Euclid’s process ensures optimality.

### 7 [M21]
Let $A(m,n)$ be the number of times step E1 (“divide $m$ by $n$; let $r$ be the remainder”) is executed by Euclid’s algorithm when it is started on the ordered pair $(m,n)$ of positive integers.

Knuth defines $T_n$ as the average of $A(m,n)$ when $n$ is fixed and $m$ is allowed to range over all positive integers; because only the residue of $m\bmod n$ matters after the first division, the average is taken over $m=1,2,\dots,n$ (uniformly). He notes that for large $n$, $T_n$ is $\asymp c\ln n$ with $c=12\ln 2/\pi^2$.&#x20;

The exercise asks to define $U_m$ analogously, with $m$ fixed and $n$ “ranging over all positive integers,” and to relate $U_m$ to $T_m$.

We will prove that $U_m$ is well defined (the limit average exists and is finite) and that

$$
\boxed{\,U_m = T_m + 1\,}.
$$

Two simple identities for $A(m,n)$

(I) Swap once when the first number is smaller.
If $m<n$, then the very first division in Euclid’s algorithm leaves remainder $r=m$; the next state is $(n,m)$. Thus

$$
A(m,n) \;=\; 1 + A(n,m)\qquad(m\ne n).
\tag{1}
$$

(II) Periodicity in $n \pmod m$ when $n>m$.
Write $n=qm+r$ with $0\le r<m$. The first step on $(m,n)$ gives remainder $r=m$ (since $n>m$), so after one step we are at $(n,m)$. One more step divides $n$ by $m$ and leaves remainder $r$; from then on we continue with $(m,r)$. Hence

$$
A(m,n) \;=\;
\begin{cases}
2, & r=0,\\
2 + A(m,r), & 1\le r\le m-1.
\end{cases}
\tag{2}
$$

So for $n>m$, $A(m,n)$ depends only on $n \bmod m$ (and equals a constant $2$ when $m\mid n$).

Existence (and value) of $U_m$

Define the Cesàro average

$$
U_m \;:=\; \lim_{N\to\infty}\frac{1}{N}\sum_{n=1}^N A(m,n),
$$

if the limit exists.

Partition $1,\dots,N$ into complete blocks of length $m$ plus a final partial block. Using (II), each complete block contributes

$$
\sum_{r=0}^{m-1} A\bigl(m,\,qm+r\bigr)
= 2 + \sum_{r=1}^{m-1}\bigl(2 + A(m,r)\bigr)
= 2m + \sum_{r=1}^{m-1} A(m,r).
$$

Thus the block average is

$$
\frac{1}{m}\left(2m + \sum_{r=1}^{m-1} A(m,r)\right)
= 2 + \frac{1}{m}\sum_{r=1}^{m-1} A(m,r).
\tag{3}
$$

The final partial block contributes $O(m)$, which is negligible after division by $N$. Therefore the limit defining $U_m$ exists and equals the block average:

$$
U_m \;=\; 2 + \frac{1}{m}\sum_{r=1}^{m-1} A(m,r).
\tag{4}
$$

Relation to $T_m$

By definition (see the paragraph introducing $T_n$ and how the average is taken over a complete set of residues), we have

$$
T_m \;=\; \frac{1}{m} \sum_{x=1}^{m} A(x,m)
= \frac{1}{m}\Bigl(A(m,m) + \sum_{x=1}^{m-1} A(x,m)\Bigr).
$$

Now use (I) with $x<m$: $A(x,m)=1+A(m,x)$. Also $A(m,m)=1$. Hence

$$
T_m
= \frac{1}{m}\Bigl(1 + \sum_{x=1}^{m-1} (1 + A(m,x))\Bigr)
= \frac{1}{m}\Bigl(m + \sum_{x=1}^{m-1} A(m,x)\Bigr)
= 1 + \frac{1}{m}\sum_{x=1}^{m-1} A(m,x).
\tag{5}
$$

Comparing (4) with (5) immediately yields

$$
\boxed{\,U_m = T_m + 1\,}.
$$

Sanity check on a small example

Take $m=5$. One can compute:

$$
A(5,1)=1,\;
A(5,2)=2,\;
A(5,3)=3,\;
A(5,4)=3.
$$

Then

$$
T_5 = 1 + \tfrac{1}{5}(1+2+3+3)=1+\tfrac{9}{5}= \tfrac{14}{5},
\qquad
U_5 = 2 + \tfrac{1}{5}(1+2+3+3) = 2+\tfrac{9}{5}=\tfrac{19}{5}
= T_5+1.
$$

(Manually checking a long run of $n$ values shows the block average equals $19/5$, as predicted.)

Consequence

Since $T_n \sim \frac{12\ln 2}{\pi^2}\ln n$ for large $n$, we get

$$
U_m \sim \frac{12\ln 2}{\pi^2}\ln m + 1,
$$

so asymptotically $U_m$ and $T_m$ differ by exactly $1$.&#x20;

That proves $U_m$ is well defined and establishes its precise relationship to $T_m$.

### 8 [M25]
Design a set of *string-rewriting* rules in Knuth’s $(T_j,\;s_j,\;a_j,\;b_j)$ format that, starting from the input string $a^m b^n$ (i.e., $m$ copies of `a` followed by $n$ copies of `b`), halts with the string $a^{\gcd(m,n)}$. The rules must be expressed as a Markov algorithm in the tabular style used in §1.1. 

Quick refresher: what $(T_j,s_j,a_j,b_j)$ means

A program state is a pair $(\sigma, j)$: current string $\sigma$ and current rule index $j$. Each table row $j$ has:

- $T_j$ (“pattern”): the substring to search for,
- $s_j$ (“replacement”): the string to substitute when $T_j$ is found,
- $b_j$: the next rule index if a replacement did occur,
- $a_j$: the next rule index if no $T_j$ is found in $\sigma$.

Operationally (Markov rules): if $T_j$ occurs, replace the *leftmost* occurrence by $s_j$ and jump to $b_j$; otherwise jump to $a_j$. One special index $N$ means “halt.” ([Stack Overflow][1])

A minimal working rule set

Use the alphabet $\{a,b,c\}$ and five rules (indices $0,1,2,3,4$); let $5$ be the halting index $N$.

| j | $T_j$   | $s_j$         | $b_j$ | $a_j$ | Intuition                                                        |
| - | ------- | ------------- | ----: | ----: | ---------------------------------------------------------------- |
| 0 | `ab`    | (empty)       |     1 |     2 | Cancel one `a` with one `b` (one subtraction step).              |
| 1 | (empty) | `c` (prepend) |     0 |     0 | After each cancellation, count it by sticking a `c` on the left. |
| 2 | `a`     | `b`           |     2 |     3 | Turn all remaining `a`’s into `b`’s.                             |
| 3 | `c`     | `a`           |     3 |     4 | Then turn all `c`’s into `a`’s.                                  |
| 4 | `b`     | `b`           |     0 |     5 | If any `b`’s remain, repeat another Euclid round; else halt.     |

Why this works (high-level):

- Rules 0↔1 simulate *repeated subtraction*: each `ab` removed corresponds to kubtracting the smaller number from the larger once; the number of times you can do this is $\min(m,n)$. We “count” those cancellations by accumulating `c`’s.
- When no `ab` remains, the string looks like `c^{min(m,n)} x^{|m-n|}`, where $x$ is whichever letter survived (`a` or `b`).
- Rules 2 and 3 normalize the string to `a^{min(m,n)} b^{|m-n|}`, i.e. we map the pair $(m,n)$ to $(\min(m,n),\,|m-n|)$.
- Rule 4 loops if the second component is nonzero; otherwise halt. This is exactly the subtraction-based Euclidean step, so repeated rounds end with `a^{gcd(m,n)}`. The superscript means “repeat `a` that many times.” ([Stack Overflow][2])

Correctness argument (brief but complete)

Invariant after finishing rule 0↔1 loop (before rule 2):
Let the current pair be $(m,n)$. Repeatedly removing the leftmost `ab` (rule 0) and then prepending a `c` (rule 1) exactly $\min(m,n)$ times yields a string of the form:

$$
c^{\min(m,n)}\; y^{|m-n|},
$$

where $y$ is `a` if $m>n$ and `b` if $n\ge m$. This encodes “we subtracted the smaller from the larger $\min(m,n)$ times.”

Normalization (rules 2→3):

- Rule 2 turns any stray `a`’s into `b`’s, so only `c`’s and `b`’s remain.
- Rule 3 turns each `c` into `a`.
  Thus the string becomes `a^{\min(m,n)} b^{|m-n|}` in canonical order.

Euclid step and termination (rule 4):

- If any `b` remains, rule 4 jumps back to rule 0, so the new pair is $(m',n')=(\min(m,n),|m-n|)$.
- Repeating transforms $(m,n)$ via the subtraction form of Euclid’s algorithm until $n'=0$; at that point rule 4 falls through to $5$ (halt), with the string `a^{gcd(m,n)}`. ([Stack Overflow][1])

Worked example

Take $m=6$, $n=10$; start with `aaaaaabbbbbbbbbb`.

- Round 1 (to exhaustion of rule 0): remove `ab` six times while accumulating `c`’s → `ccccccbbbb`.
  Normalize (rules 2→3) → already no `a`’s, then `c→a` ⇒ `aaaaaabbbb` = `a^6 b^4`.
- Round 2: cancel `ab` four times → `cccc a a` (i.e., `ccc caa` but order doesn’t matter until normalization) ⇒ normalize ⇒ `a^4 b^2`.
- Round 3: cancel twice ⇒ normalize ⇒ `a^2 b^0`.
- Round 4: rule 4 sees “no `b` left” ⇒ halt with `aa` = $a^{\gcd(6,10)}$.

Relation to Euclid’s Algorithm E

Each full pass of rules 0→…→4 implements the mapping

$$
(m,n)\;\mapsto\;(\min(m,n),\;|m-n|),
$$

i.e., Euclid’s algorithm with the division step replaced by *repeated subtraction*. That’s precisely the hint Knuth gives; the string system is just a Markov-algorithm rendering of that idea. 

Notes and common confusions

- The superscript in “$a^{\gcd(m,n)}$” means “`a` repeated $\gcd(m,n)$ times.” For example, if $\gcd(12,18)=6$, the output is `aaaaaa`.
- In the rule table, $b_j$ is the “branch if a replacement happened,” and $a_j$ is the “branch if not.” This branching scheme, together with leftmost replacement, is standard for Markov algorithms and matches Knuth’s §1.1 formalism.

### 9 [M30]
Let

$$
C_1=(Q_1, I_1, \Omega_1, f_1),\qquad
C_2=(Q_2, I_2, \Omega_2, f_2)
$$

be computational methods (state space $Q$, initial/allowable inputs $I\subseteq Q$, halting/“output” states $\Omega\subseteq Q$, and a (partial) next-state function $f$). A *computation sequence* of $C_i$ is the usual chain $x, f_i(x), f_i^{(2)}(x),\dots$ until (and if) a state in $\Omega_i$ is reached.

Intuitively, $C_2$ may do more, smaller steps and may keep extra bookkeeping in its states, yet it should faithfully realize $C_1$.

We say that $C_2$ represents $C_1$ iff there exist total maps

- an encoding $e:Q_1\to Q_2$ and
- a decoding/abstraction $\pi:Q_2\to Q_1$

such that $\pi\!\circ e=\mathrm{id}_{Q_1}$ (i.e., $e$ embeds $C_1$’s abstract state into $C_2$’s concrete state, and $\pi$ forgets the extra detail), and the following hold:

1. Input compatibility. $e(I_1)\subseteq I_2$.

2. Output compatibility. $\pi(\Omega_2)\subseteq \Omega_1$ and for every $\omega\in\Omega_1$ there exists some $\hat \omega\in\Omega_2$ with $\pi(\hat \omega)=\omega$.
   (Equivalently, $\pi^{-1}(\Omega_1)$ contains only states from which $C_2$ can reach $\Omega_2$ without changing $\pi$’s value.)

3. Step correspondence (stuttering simulation).
   For every $q_1\in Q_1\setminus \Omega_1$ and every $q_2\in Q_2$ with $\pi(q_2)=q_1$, if $f_1(q_1)$ is defined and equals $q_1'$, then there exists a finite $t\ge 1$ and a path in $C_2$

   $$
   q_2 = r_0 \xrightarrow{f_2} r_1 \xrightarrow{f_2} \cdots \xrightarrow{f_2} r_t
   $$

   such that

   $$
   \pi(r_j)=q_1\quad(0\le j<t),\qquad \pi(r_t)=q_1'.
   $$

   (So $C_2$ may “stutter”—take several internal steps that do not change the abstract state—before realizing one abstract $f_1$-step.)

4. Termination preservation (refinement of whole runs).
   For every input $x\in I_1$, if the $C_1$ run from $x$ halts in $k$ steps with output $\omega\in\Omega_1$, then the $C_2$ run from $e(x)$ halts after some $T\ge k$ steps in a state $\hat \omega\in\Omega_2$ with $\pi(\hat \omega)=\omega$.

This is the standard “forward simulation with stuttering” characterization used in refinement: $C_2$ can take more (finer-grained) steps and store more information, but when you *abstract* its state with $\pi$ you see exactly the abstract behavior of $C_1$.

Why this captures the intent

- “More steps”: Condition 3 allows multiple $f_2$ steps for a single $f_1$ step while keeping $\pi$ unchanged until the abstract change happens.
- “More information”: The abstraction $\pi$ can forget registers, temporary variables, program counters, etc.; $\pi\!\circ e=\mathrm{id}$ ensures no abstract information is lost when we embed.
- Correct inputs/outputs: Conditions 1–2 align the two machines at the beginning and end.
- Whole computations: Condition 4 ensures that every terminating abstract computation is realized concretely with the same abstract result.

How it looks on the Euclid example

In §1.1, Algorithm E is formalized as a $(Q,I,\Omega,f)$ system with states like $(m,n,0,1)$, $(m,n,r,2)$, $(n)$, etc., and a next-state function that embodies steps E1–E3. A machine program $C_2$ has states including memory, registers, and a program counter. Define:

- $e$: put the abstract $m,n,r,\ldots$ into designated registers/memory and set the PC to the code for the corresponding E-step.
- $\pi$: read back those designated registers and map *any* machine state of that code block to the corresponding abstract state (ignoring temps, flags, scratch space, etc.).

A long division routine in $C_2$ will execute many micro-steps while $\pi$ stays at the same abstract E1 state; upon finishing the division, $\pi$ jumps to the E2/E3 abstract state—exactly the stuttering behavior required by (3). When $n=0$, the program branches to a return sequence; $\pi$ maps those return-sequence states into $C_1$’s halting set, establishing (2)–(4).
