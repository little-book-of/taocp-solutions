## 1 [14]
What FCHECK does, and where itвҖҷs used

* Purpose. `FCHECK` decodes the FвҖ“field (in `rI3`) as a *partial field specification* of the form $8L + R$, ensures $L \le R \le 5$, then leaves the results in convenient places: it stores $L$ and $R$ in memory cells `L` and `R`, and sets `rI1 вҶҗ L`. (If invalid, it jumps to `FERROR`.)

* In `GETV`. The value-extraction subroutine begins by calling `FCHECK`, then fetches memory (`MEMORY`), and finally uses `L` and `R` to select the requested field and normalize the sign. (If the sign isnвҖҷt included, it forces the sign to `+`.)

* In `STORE`. The store routine also begins with `FCHECK`, then uses `L`/`R` to splice the source register into the destination wordвҖ”handling the special case when the sign-byte (i.e., `L=0`) is included.

* In `COMPARE`. The program *intentionally* avoids calling `FCHECK` twice: it calls `GETV` (which runs `FCHECK` once), then enters `GETV` through the special entrance `GETAV` (which jumps past the `FCHECK` code) to reuse the already-decoded `L`. This is the one place where the code is already organized to keep the `FCHECK` overhead to a single call.

* The exercise note. The exercise explicitly points you to вҖңstep 3 (Reexamination)вҖқ at the end of В§1.4.1, which suggests *enlarging or merging subroutines when the caller always does the same standard setup/cleanup around them*.
  The exercise statement itself appears at the bottom of the simulator section.

Diagnosis

Across the simulator:

* `FCHECK` is always followed immediately by *very similar* boilerplate in its callers (e.g., loading `-R` into `rI2`, checking whether the sign is included via `L==0`, then doing standard shifts).
* Only `COMPARE` already amortizes `FCHECK` over two uses of the same field (memory and register) by using the `GETAV` entrance.

ThatвҖҷs a textbook case for Step-3 refactoring: push the common pre/post work into a single place and remove redundant calls.

A better organization (two practical refactorings)

You can do either (A) вҖңpre-decode F once per instruction that needs (L\:R)вҖқ or (B) вҖңabsorb/inline the decode into the two places that actually need it.вҖқ Both obey Step-3.

A) Pre-decode once (preferred if youвҖҷre willing to touch the control path)

1. Mark which opcodes use (L\:R). Add a small parallel table to `OPTABLE` that classifies the FвҖ“field usage of each opcode (e.g., 0 = not a partial field, 1 = partial field (L\:R), 2 = shift count, вҖҰ). The control routine already fetches `F` into `rI3` (line 027). Use your new table to decide whether to decode now.

2. Decode centrally. If the current opcodeвҖҷs FвҖ“usage is вҖңpartial field,вҖқ call a beefed-up decoder once and for all. That decoder should:

   * Validate and compute `L` and `R` (as `FCHECK` does).
   * Also set `rI2 вҶҗ вҲ’R` since every client immediately needs it.
   * Optionally set a one-byte flag `SFLAG вҶҗ (L==0)` so clients donвҖҷt have to re-test.

3. Use вҖңpreparedвҖқ entries.

   * Change callers that currently do `JMP GETV` to `JMP GETAV` (the *already-prepared* entrance) because `rI1`/`R` are now set; `GETAV` already jumps past the `FCHECK` block.
   * Split `STORE` into two entries: keep `STORE` (old behavior) and add `STORE1` right after the `FCHECK` call site; have the opcode table jump to `STORE1` when the control routine has pre-decoded `F`. This removes the `FCHECK` call in the common path.

This removes *all duplicate decoding* in instructions that use the same (L\:R) twice (e.g., `COMPARE`) and also avoids decoding in instructions whose FвҖ“field isnвҖҷt a partial field (e.g., `JUMP`, `ADDROP`). It follows KnuthвҖҷs Step-3 guidance exactly: enlarge the central вҖңcontrolвҖқ routine slightly to relieve its callees of repeated setup.

B) Inline/merge (`FCHECK` вҶ’ callers) with a macro (minimal edits)

If you prefer not to touch the control path or tables:

* Delete `FCHECK` as a runtime subroutine and replace it by an *assembly macro* (say, `DECODEF`) that expands to the 12 lines of its body (compute `L`, `R`, checks, set `rI1`). Expand `DECODEF` at the start of `GETV` (before label `1F`) and at the start of `STORE`. Also have the macro optionally do `LD2N R` so all callers get `rI2` вҖңfor free.вҖқ
* Keep `GETAV` unchanged so `COMPARE` still jumps past the macro expansion in `GETV` and benefits from a single decode.

This keeps code simple, keeps `COMPARE` optimal, and respects Step-3 (вҖңperhaps several subroutines should be merged into one; or perhaps a subroutine is called only once and should not be a subroutine at allвҖқ).

Why this is better

* Fewer calls / simpler invariants. Operators that need fields no longer pay to rediscover `L`/`R` multiple times, and they get the always-needed `rI2 вҶҗ вҲ’R` (and optionally `SFLAG`) precomputed. See how `GETV` and `STORE` currently repeat the same checks right after `FCHECK`.
* Matches the one existing optimization. `COMPARE` already factors out the second decode via `GETAV`; the refactor generalizes that cleanliness instead of being a one-off.
* Faithful to Step-3. ItвҖҷs precisely the вҖңenlarge/merge subroutinesвҖқ advice Knuth gives after completing a first cut: push common pre/post code down so the outer structure is cleaner.

Concrete changes youвҖҷd make

* Control routine (option A): after `LD3 INST(4:4)` (line 027), if the opcodeвҖҷs FвҖ“class is вҖң(L\:R)вҖқ, call `FDECODE` (your enhanced `FCHECK`) which sets `L`, `R`, `rI1=L`, `rI2=вҲ’R`, and `SFLAG`. Then have the opcode table point to `GETAV` (instead of `GETV`) and to `STORE1` (the post-decode entry).
* GETV: remove the prologue call to `FCHECK`; enter at the existing label after that call (`GETAV` already does exactly this).
* STORE: add label `STORE1` at the point right after the original `JMP FCHECK` and retarget the table to `STORE1` when pre-decoding is in effect; otherwise fall back to `STORE` for compatibility.

Either approach solves Exercise 1: youвҖҷve studied all uses of `FCHECK`, noticed the duplicated patterns right after it, and reorganized so the decode happens once (or is inlined where it belongs), exactly the kind of Step-3 improvement Knuth advocates.

## 2 [20]

HereвҖҷs a clean, self-contained SHIFT routine for the MIX simulator. It plugs into the program exactly where opcode 6 (вҖңSHIFTвҖқ) is dispatched in the operator table (see the listing that maps `F=0..5` to `SLA, SRA, SLAX, SRAX, SLC, SRC`). I keep A/X signs in `SIGNA/SIGNX` and only rearrange magnitudes in `AREG/XREG`, matching the simulatorвҖҷs data layout.

What the routine must do (recap)

* F=0: `SLA m` вҖ” shift A left by `m` bytes, zero-fill on the right; sign unchanged.
* F=1: `SRA m` вҖ” shift A right by `m` bytes, zero-fill on the left; sign unchanged.
* F=2: `SLAX m` вҖ” shift the (A,X) 10-byte pair left by `m`, zero-fill; signs unchanged.
* F=3: `SRAX m` вҖ” shift the (A,X) pair right by `m`, zero-fill; signs unchanged.
* F=4: `SLC m` вҖ” rotate the (A,X) pair left by `m` (circular); signs unchanged.
* F=5: `SRC m` вҖ” rotate the (A,X) pair right by `m` (circular); signs unchanged.
  KnuthвҖҷs exercise simply asks us to вҖңwrite the SHIFT routineвҖҰ (operation code 6).вҖқ

MIX code 

```mixal
* ------------------------------------------------------------
* SHIFT вҖ” opcode 6
* Uses: rI3 = F (variant 0..5), rI5 = M (shift count, possibly signed)
* Touches: AREG/XREG magnitudes only; SIGNA/SIGNX unchanged
* Notes: If tracking timing exactly, add |M| to TIME in this routine,
*        since OPTABLE contributes the base 2u already for SHIFT.
* ------------------------------------------------------------
SHIFT   ENT2    0,5         rI2 вҶҗ M (shift count)
        J2NN    *+2         make rI2 вҶҗ |M|
        ENN2    0,2
        CMP3    =5=         F must be 0..5
        JG      CYCLE       (defensive: treat out-of-range F as NOP)

        JMP     SHTAB,3     dispatch by F

* --- table of tiny trampolines (same pattern used elsewhere) ---
SHTAB   JMP     S_SLA
        JMP     S_SRA
        JMP     S_SLAX
        JMP     S_SRAX
        JMP     S_SLC
        JMP     S_SRC

* --- F=0: SLA m (A only) ---
S_SLA   LDA     AREG
        SLA     0,2         shift A left by rI2 bytes
        STA     AREG
        JMP     S_DONE

* --- F=1: SRA m (A only) ---
S_SRA   LDA     AREG
        SRA     0,2         shift A right by rI2 bytes
        STA     AREG
        JMP     S_DONE

* --- F=2: SLAX m (A,X together) ---
S_SLAX  LDA     AREG
        LDX     XREG
        SLAX    0,2         shift (A,X) left by rI2 bytes
        STA     AREG
        STX     XREG
        JMP     S_DONE

* --- F=3: SRAX m (A,X together) ---
S_SRAX  LDA     AREG
        LDX     XREG
        SRAX    0,2         shift (A,X) right by rI2 bytes
        STA     AREG
        STX     XREG
        JMP     S_DONE

* --- F=4: SLC m (A,X circular left) ---
S_SLC   LDA     AREG
        LDX     XREG
        SLC     0,2         rotate (A,X) left by rI2 bytes
        STA     AREG
        STX     XREG
        JMP     S_DONE

* --- F=5: SRC m (A,X circular right) ---
S_SRC   LDA     AREG
        LDX     XREG
        SRC     0,2         rotate (A,X) right by rI2 bytes
        STA     AREG
        STX     XREG
        JMP     S_DONE

* --- wrap up; optional timing bump here ---
S_DONE  * If desired:  TIME вҶҗ TIME + rI2  (|M|)
        JMP     CYCLE
```

Why this is correct (brief reasoning)

1. Signs preserved. The simulator stores signs separately (`SIGNA`, `SIGNX`). We only load/store magnitudes from `AREG`/`XREG`, so shifting cannot disturb signs.
2. Zero-fill vs rotate. Host MIXвҖҷs `SLA/SRA/SLAX/SRAX` zero-fill the vacated bytes; `SLC/SRC` rotate within the 10-byte `rAX` pair. That exactly matches MIX semantics for opcode 6 variants.
3. Large or negative counts. We take `|M|`. For `SLA/SRA` beyond the register width, the host shift yields zeros (desired). For rotations, host `SLC/SRC` behave modulo 10 bytes, so any `M` works sensibly.
4. Timing (if modeled). The OPTABLE entry already accounts for the base вҖң2uвҖқ per SHIFT; if youвҖҷre simulating time precisely, add `|M|` here to reflect per-byte steps. (KnuthвҖҷs listing shows SHIFTвҖҷs table entry and the pattern for per-op timing updates.)

```mixal
*  SHIFT operators (opcode = 6)
*  Uses rI3 = F, rI5 = M (already set by the control routine).
*  Count = |M|; signs are preserved by MIX shift ops automatically.

SHIFT   DEC3    5           Is F > 5?
        J3P     FERROR      If so, invalid F for SHIFT.

        LD1     M(1:2)      rI1 вҶҗ |M| (shift count, magnitude of M).

        JMP     SFTABL,3    Dispatch to SLA/SRA/SLAX/SRAX/SLC/SRC.

* --- F = 0 : SLA (shift left rA) ---
SLAOP   LDA     AREG
        SLA     0,1         Shift rA left by rI1 bytes, zero-fill right.
        STA     AREG
        JMP     CYCLE

* --- F = 1 : SRA (shift right rA) ---
SRAOP   LDA     AREG
        SRA     0,1         Shift rA right by rI1 bytes, zero-fill left.
        STA     AREG
        JMP     CYCLE

* --- F = 2 : SLAX (shift left rAX as 10-byte word) ---
SLAXOP  LDA     AREG
        LDX     XREG
        SLAX    0,1         Shift rAX left by rI1 bytes, zero-fill right.
        STA     AREG
        STX     XREG
        JMP     CYCLE

* --- F = 3 : SRAX (shift right rAX) ---
SRAXOP  LDA     AREG
        LDX     XREG
        SRAX    0,1         Shift rAX right by rI1 bytes, zero-fill left.
        STA     AREG
        STX     XREG
        JMP     CYCLE

* --- F = 4 : SLC (circular left rAX) ---
SLCOP   LDA     AREG
        LDX     XREG
        SLC     0,1         Circular left shift of rAX by rI1 bytes.
        STA     AREG
        STX     XREG
        JMP     CYCLE

* --- F = 5 : SRC (circular right rAX) ---
SRCOP   LDA     AREG
        LDX     XREG
        SRC     0,1         Circular right shift of rAX by rI1 bytes.
        STA     AREG
        STX     XREG
        JMP     CYCLE

* Jump table for F = 0..5, addressed via "JMP SFTABL,3" after "DEC3 5".
* Place the label on the last entry (same pattern as the bookвҖҷs JUMP table).
        JMP     SLAOP
        JMP     SRAOP
        JMP     SLAXOP
        JMP     SRAXOP
        JMP     SLCOP
SFTABL  JMP     SRCOP
```

Why this matches the bookвҖҷs simulator

1. Dispatch pattern. The bookвҖҷs operator routines use `DEC3 k` followed by `JMP TABLE,3`, with the table label on the *last* entry so that negative offsets select earlier entries (see the `JUMP` operatorвҖҷs table). We do the same with `DEC3 5` and `SFTABL`.

2. Where the count comes from. The control routine has already computed the effective address and stored it in the variable `M`; it then sets `rI5 вҶҗ M`. We take the magnitude of `M` via `LD1 M(1:2)` to obtain a nonnegative byte count for shifting. (Signs of `M` are irrelevant to SHIFT.)

3. Which registers to touch. `SLA`/`SRA` affect only the A register; `SLAX`/`SRAX`/`SLC`/`SRC` treat `rAвҖ–rX` as a 10-byte word. In the simulator, magnitudes live in `AREG`/`XREG`, and signs in `SIGNA`/`SIGNX`. MIX shift operators do not change signs, so we update only `AREG`/`XREG`.

4. Opcode and timing. The switch table ties opcode 6 to `SHIFT(2)`, so no extra timing logic is needed in the routine; the control code already accounts for the two time units when it indexes the table.

Notes and edge cases

* If the count exceeds the register width (e.g., SLA/SRA by вүҘ5, SLAX/SRAX by вүҘ10), MIX semantics produce all zeros in the shifted region; `SLC`/`SRC` are naturally modulo-10 circular shifts. The host MIX instructions used above enforce these behaviors directly.

* The routine intentionally ignores the sign of `M` by using `|M|` as the count, which matches standard descriptions of MIX shift instructions (вҖңM specifies the number of bytes to be shiftedвҖқ).

## 3 [22]
What MOVE must do

* Semantics: move F words, one word at a time, from source block starting at M to destination block starting at the (simulated) rI1 address; when finished, rI1 is increased by F. If F=0, nothing happens. Execution time is `1 + 2F` units (the base вҖң1вҖқ comes from the opcode table; we add `2` per word moved in the routine).&#x20;
* Overlap behavior (important!): copying proceeds вҖңforwardвҖқ (from `M` to `M+1` вҖҰ while storing to `rI1`, `rI1+1`, вҖҰ). If destination starts inside the source and ahead of it (e.g., `rI1 = M+1`), the same word may be replicated, exactly as described in the manual. Your simulator should therefore not try to be вҖңmemmove-safeвҖқ; it must mimic MIXвҖҷs specified effect.&#x20;
* Control state the simulator has already set up for you: `rI5 вҶҗ M`, `rI3 вҶҗ F`, `rI4 вҶҗ C`, and the opcode table has placed the base time in `TIME` before jumping to the per-opcode routine. The magnitudes/signs of simulated registers live in memory cells like `I1REG`, `SIGN1`, etc.&#x20;

 MOVE routine (MIXAL)

Paste this under the label `MOVE` thatвҖҷs wired in the opcode table.

```mixal
* MOVE вҖ” opcode 7
MOVE    J3Z   CYCLE          * If F=0, nothing to do.

        JMP   MEMORY         * Fetch SOURCE word at address rI5:
                             * on return: rX = sign, rA = magnitude (of memory[rI5])

        SRAX  5              * Pack sign+magnitude so the full word sits in rX.

* Validate DEST = simulated rI1 (must be in [0, BEGIN))
        LD1   I1REG          * rI1 вҶҗ magnitude of simulated index register 1
        LDA   SIGN1          * rA  вҶҗ sign of simulated index register 1
        JAP   OKDEST         * if +, OK
        J1NZ  MEMERROR       * if negative and magnitude вү  0 вҶ’ invalid address
        STZ   SIGN1(0:0)     * normalize -0 as +0 (permit address 0)

OKDEST  CMP1  =BEGIN=        * rI1 вүҘ BEGIN ?
        JGE   MEMERROR       * out of simulated memory вҶ’ error

* Store the fetched word into DEST and account for time
        STX   0,1            * memory[rI1] вҶҗ fetched full word
        LDA   CLOCK          * add 2 units per moved word (base 1 was set already)
        INCA  2
        STA   CLOCK

* Advance pointers/count and loop
        INC1  1              * rI1++
        ST1   I1REG          * write back simulated I1
        INC5  1              * rI5++ (SOURCE++)
        DEC3  1              * F--
        JMP   MOVE           * keep moving until J3Z fires at top
```

This routine follows MIXвҖҷs defined behavior exactly:

* It moves one word at a time, always reading the source first (via `MEMORY`) and then writing to the current destination (`STX 0,1`).
* Because we progress `SOURCE` (`rI5`) and `DEST` (`rI1`) forward in lock-step, the overlap case where `DEST > SOURCE` produces the same вҖңreplicationвҖқ effect shown in the book (thatвҖҷs the intended semantics).&#x20;
* We charge time by adding `2` to `CLOCK` per moved word; together with the opcode tableвҖҷs base time, this yields `1 + 2F`.&#x20;
* We update the simulated `I1` immediately each step (by writing back `I1REG`), which is fine because programs canвҖҷt observe intermediate state during an instruction; at completion, `rI1` has increased by F, as required.&#x20;
* Address checks ensure both source (handled by `MEMORY`) and destination are inside simulated memory `[0, BEGIN)`, with `-0` treated as `+0`.&#x20;

Quick sanity check on overlap

* Example from the text: `F=3`, `M=1000`, `rI1=999` вҶ’ youвҖҷll copy `1000вҶ’999`, `1001вҶ’1000`, `1002вҶ’1001` (no surprise).
* If instead `rI1=1001`, youвҖҷll copy `1000вҶ’1001`, then (since `1001` now holds the old 1000) `1001вҶ’1002`, then `1002вҶ’1003`, so the original `CONTENTS(1000)` appears three timesвҖ”exactly as MIX specifies.&#x20;

Where these hooks live in the simulator

* The control routine unpacks `INST`, computes `M` with indexing, stores `F` in `rI3` and `M` in `rI5`, looks up the routine in `OPTABLE`, stores the base time in `TIME`, and jumps to `MOVE`. Your routine returns to `CYCLE` when `F` becomes zero.&#x20;

## 4 [14]
HereвҖҷs the smallest correct change to KnuthвҖҷs MIX-simulator so it вҖңstarts as if the GO button had been pushed.вҖқ

What the GO button does (per В§1.3.1, Ex. 26)

When GO is pressed (with a card reader present), MIX (1) reads one card into locations 0000вҖ“0015, (2) clears the overflow toggle and sets rJ вҶҗ 0, then (3) jumps to location 0000 to begin execution.&#x20;

What the simulator already does

In Program M (the simulator), the main loop already:

* Clears the overflow toggle (`STZ OVTog`) and
* Begins execution at location 0 by setting the вҖңnext instructionвҖқ register to 0 (`ENT6 0`, where rI6 is the location of the next instruction).&#x20;

So the only GO-state it doesnвҖҷt establish is rJ = 0.

Minimal patch

Add one instruction at program start to zero the simulated J register:

```
001 * MIX SIMULATOR
002 ORIG 3500
003 BEGIN   STZ TIME(0:2)   ; clear time
004          STZ OVTog      ; clear overflow toggle  (GO requirement)
004a         STZ JREG       ; * new: rJ вҶҗ 0       (GO requirement)
005          STZ COMPI
006          ENT6 0         ; start at location 0   (GO requirement)
007 CYCLE    LDA CLOCK
...
```

* `STZ JREG` makes the simulatorвҖҷs rJ match the GO buttonвҖҷs post-reset state.
* `STZ OVTog` and `ENT6 0` already matched the GO semantics (overflow cleared; вҖңPCвҖқ set to 0000, since rI6 is the location of the next instruction). The control routineвҖҷs use of rI6 for the fetch is described just above Program M.&#x20;

Notes

* We arenвҖҷt simulating the initial card read (GO step 1) here; I/O operators are intentionally left for later exercises (В§1.4.3.1, Ex. 6). This exercise only asks the simulator to *begin* as if GO had been used.
* After this one-line change, the simulatorвҖҷs initial state matches items (2)вҖ“(3) of GO: rJ=0, overflow cleared, and execution begins at 0000.

ThatвҖҷs itвҖ”one added `STZ JREG` at `BEGIN`.

## 5 [24]
Timing rules weвҖҷll use (real MIX)

MIX uses time unit u. From the timing summary:

* All LOAD/STORE, ADD/SUB, comparisons, shifts: 2u
* MOVE: 1u + 2u per word
* MUL/NUM/CHAR: 10u; DIV: 12u
* All remaining ops (including jumps, INC/DEC/ENT): 1u
  In particular, LDA = 2u, ENTA = 1u on real MIX.

Assumptions (to keep the path concrete)

* Instruction uses no indexing (I=0), so the indexing block in the control routine is skipped.
* No errors (no range, field, or memory errors).
* For LDA, the usual field (0:5) (sign included).
* For ENTA, the address M вү  0 (the вҖңM=0вҖқ special sign case is noted later).
  These are the straight-line, normal paths through the simulator.

The simulatorвҖҷs fixed control overhead per simulated instruction

Every instruction first goes through the control routine at label CYCLE (load CLOCK, fetch INST, unpack M/I/F/C, consult OPTABLE, and jump). Counting times by instruction class yields 34u before the jump to the operator routine when I=0 (lines 007вҖ“035: loads/stores/shift = 2u each; INC/DEC/J? = 1u).

Simulating LDA

Path: CYCLE вҶ’ OPTABLE вҶ’ LOAD вҶ’ GETV вҶ’ (FCHECK, MEMORY) вҶ’ LOAD1 вҶ’ SIZECHK вҶ’ CYCLE.
Key code: LOAD/LOADN (lines 196вҖ“205), GETV (149вҖ“158), FCHECK (135вҖ“146), MEMORY (125вҖ“134), SIZECHK/OVCHECK (289вҖ“291, 191).

* Control overhead to reach operator: 34u.

* Operator + subroutines:

  * `JMP GETV`: 1u
  * FCHECK (valid field): STJ, ENTA, ENTX, DIV, CMPX, JG, STX, STA, LD1, CMPA, JLE вҶ’ 27u
  * MEMORY: STJ, JS? (not taken), CMP, JGE (not taken), LDX, ENTA, SRAX, LDA, JMP вҶ’ 14u
  * Return in GETV (sign included, so skip setting + sign): J1Z, LD2N, SRA, JMP вҶ’ 6u
  * LOAD1 store magnitude/sign: ENT1, STA, STX, JMP вҶ’ 6u
  * SIZECHK/OVCHECK (no index overflow, no overflow): LD1, J1Z, JNOV вҶ’ 4u

  Sum operator-side = 1 + 27 + 14 + 6 + 6 + 4 = 58u (plus 5u for the two unconditional returns to GETVвҖҷs and MEMORYвҖҷs `JMP *` sites) вҶ’ 63u total operator segment.

* Total simulated time for LDA = 34u + 63u = 97u.

Real MIX time for LDA = 2u.
Slowdown вүҲ 97/2 вүҲ 48.5Г—.

Simulating ENTA

Path: CYCLE вҶ’ OPTABLE вҶ’ ADDROP вҶ’ (LOAD1) вҶ’ SIZECHK вҶ’ CYCLE.
Key code: ADDROP block (265вҖ“278), then LOAD1 and SIZECHK as above.

* Control overhead to reach operator: 34u.

* Operator path (typical M вү  0 branch):

  * ADDROP prologue & sign/magnitude setup: DEC3, J3P (not taken), ENTX, JXNZ (taken), SRAX, LDA, ENT1, `JMP 1F,3`, `JMP LOAD1` вҶ’ 11u
  * LOAD1 store magnitude/sign: STA, STX, `JMP SIZECHK` вҶ’ 5u
  * SIZECHK/OVCHECK: LD1, J1Z, JNOV вҶ’ 4u

  Sum operator-side = 20u.

* Total simulated time for ENTA = 34u + 20u = 54u.

Real MIX time for ENTA = 1u.
Slowdown вүҲ 54/1 = 54Г—.

Notes on variants

* Indexing present (Iвү 0): the control routine executes the indexing block (lines 018вҖ“026), adding \~15u more (several loads/stores/shifts/add/compare/jump).
* LDA with field not including sign: GETV executes three extra ops (ENTX; two shifts), adding \~5u.

Design (what to simulate)

MIX overlaps I/O with computation. Issuing IN/OUT/IOC takes 1 unit of CPU time and makes the device busy for T units; the CPU continues. The queries JBUS/JRED test that busy/ready state. We therefore maintain per-device вҖңready timesвҖқ and a CPU clock:

* `CLOCK` вҖ” current CPU time (u).
* `RDRFREE` вҖ” earliest time at which unit 16 (reader) is ready.
* `PRTFREE` вҖ” earliest time at which unit 18 (printer) is ready.
* Constants: `TREAD=10000`, `TSKIP=10000`, `TPRINT=7500`.
* Overlap rule: when (re)starting an I/O, let `start = max(CLOCK, DEVFREE)` then set `DEVFREE = start + T`. CPU time still advances by 1 for the instruction itself (the вҖң1+TвҖқ shown in the table), while the T portion accrues on the device timeline.&#x20;

## 7 [32]
HereвҖҷs a clean, self-contained way to solve this exercies. The point is to keep your MIX I/O simulator realistic: executing `IN`/`OUT` must not move data immediately; the actual transfer happens roughly halfway through the deviceвҖҷs service time, while the device remains busy for the full time. This prevents programs from (incorrectly) assuming that `IN`/`OUT` are instantaneous. The exercise builds directly on the I/O instructions and device behavior defined earlier (units 16 = card reader, 18 = line printer; `IN/OUT/IOC/JBUS/JRED` semantics), and on the timing constants given in Ex. 6 (read-card or skip-page: `T=10000u`; print-line: `T=7500u`).   The exerciseвҖҷs specific requirement (вҖңmake the transmission occur after about half the timeвҖқ) appears at the end of the sectionвҖҷs exercise list.&#x20;

Model the timing

Maintain a global simulator clock `now` (in MIX time units `u`) and, for each device `d вҲҲ {16,18}`, these fields:

* `ready_at[d]` вҖ” time when the device will become ready again.
* `xfer_at[d]` вҖ” time when the *data movement* (memory вҶ” device buffer) actually occurs; this is `start + ceil(T[d]/2)`.
* `pending[d]` вҖ” `None | ("IN", M) | ("OUT", M) | ("IOC", op)` describing the scheduled action.
* Optional device buffers (e.g., next card image for 16; print line buffer for 18).

Timing constants (from Ex. 6):
`T[16].read_card = 10000`, `T[18].print_line = 7500`, and `T[18].skip_to_new_page (IOC) = 10000`.&#x20;

Event processing hook

Before fetching each instruction in your main `CYCLE` loop, process any I/O events whose time has arrived:

```
def service_io_events():
    for d in (16, 18):
        if pending[d] and now >= xfer_at[d]:
            # Perform the scheduled data movement exactly once.
            if pending[d].op == "IN":
                mem[M ... M+block_size(d)-1] = device_in_buffer[d]
            elif pending[d].op == "OUT":
                device_out_buffer[d] = mem[M ... M+block_size(d)-1]
            # IOC with no data movement: nothing to do here.
            pending[d] = None
            xfer_at[d] = +вҲһ
```

This вҖңhalf-timeвҖқ transfer ensures that a program that uses the data too early (after `IN`) or changes it too late (after `OUT`) will behave incorrectlyвҖ”exactly the mistake this exercise wants to catch.

Instruction semantics (with half-time transfers)

Use the architectural rules from В§1.3.1: `IN/OUT` wait until the unit is ready *to start*; the transfer completes later; `IOC` initiates a device-specific operation; `JRED`/`JBUS` test readiness.&#x20;

`IN M(F=unit)`

```
d = unit
wait_until_ready(d)               # if now < ready_at[d], advance now to ready_at[d]
start = now
pending[d] = ("IN", M)            # schedule memory вҶҗ device
xfer_at[d] = start + ceil(T[d]/2) # "approximately half" the device time
ready_at[d] = start + T[d]        # unit remains busy for full time
```

Block size: 16 words for unit 16 (card reader), 24 for unit 18 (printer) if you ever support input there; here we limit to 16 for the card reader.&#x20;

`OUT M(F=unit)`

```
d = unit
wait_until_ready(d)
start = now
pending[d] = ("OUT", M)           # schedule device вҶҗ memory
xfer_at[d] = start + ceil(T[d]/2)
ready_at[d] = start + T[d]
```

For unit 18, the printed lineвҖҷs *content* is captured at `xfer_at[18]`; the physical printing completes at `ready_at[18]`.

`IOC M(F=unit)`

Wait until not busy, then perform the control; e.g., for the printer вҖңskip-to-new-pageвҖқ takes 10000 u but does no data transfer:

```
wait_until_ready(18)
start = now
pending[18] = ("IOC", "SKIP")
xfer_at[18] = +вҲһ                  # no data movement
ready_at[18] = start + 10000
```

(If you later support `IOC` on other units, replicate this pattern with their `T`.)

`JRED M(F=unit)` and `JBUS M(F=unit)`

Use the В§1.3.1 definitions: `JRED` jumps iff the unit is ready (`now вүҘ ready_at[d]`), `JBUS` jumps iff not ready.&#x20;

Important вҖңJBUS \*вҖқ special case

As hinted in Ex. 6, if the target of `JBUS` is the instruction itself (busy-wait spin), you must advance `now` to the next interesting time to avoid a frozen simulator:

```
if op == JBUS and target == PC and now < ready_at[d]:
    now = min(ready_at[d], min(xfer_at[k] for k with pending[k]))
```

This is the pragmatic fix that keeps the loop progressing.&#x20;

Why this satisfies Exercise 7

* `IN`/`OUT` no longer вҖңcompleteвҖқ at issue time: the simulator defers the data movement to `start + T/2`, while the device remains busy until `start + T`.
* Programs that incorrectly assume immediate availability (after `IN`) or immediate consumption (after `OUT`) will misbehave, reproducing the real-machine hazard the exercise is designed to prevent.&#x20;
